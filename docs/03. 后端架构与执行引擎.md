## 一、架构总览

### 1.1 核心设计原则

| 原则 | 描述 |
|:---|:---|
| **实时优先** | K线更新延迟 ≤ 50ms（Futu Tick → Redis → WebSocket → 前端） |
| **缓存分层** | 热数据 Redis（ms 级）→ 温数据 DuckDB（百ms 级）→ 冷数据 Parquet（秒级） |
| **接口隔离** | 外部 API（用户鉴权）/ 内部 API（Hermes Tool 专用）/ WebSocket（行情推送）三通道物理分离 |
| **安全纵深** | 凭证环境变量存储、JWT 会话、操作审计日志、敏感数据落盘加密 |
| **单一数据源** | 所有外部 API（Futu / YFinance / Finnhub）只在 `workers/` 中接触，其他模块只读内部接口 |

## 二、API 网关层与接口隔离

### 2.1 三通道隔离设计

系统将 API 通道物理分离为三类，各有独立的鉴权机制和访问控制：

| 通道 | 路径前缀 | 鉴权方式 | 调用方 | 用途 |
|:---|:---|:---|:---|:---|
| **公开 REST API** | `/api/v1/` | JWT Bearer Token | Web/App 客户端 | 查询、配置、回测提交 |
| **WebSocket 行情** | `/ws/v1/` | JWT（握手阶段） | Web/App 客户端 | 实时行情推送 |
| **SSE Agent 流** | `/sse/v1/` | JWT Bearer Token | Web/App 客户端 | Hermes 流式输出 |
| **内部 Tool API** | `/internal/v1/` | HMAC-SHA256 签名 | Hermes Agent Tools | 供工具调用后端数据 |

```python
# backend/core/middleware.py — 路由级鉴权策略分发

PUBLIC_PATHS = {"/api/v1/auth/login", "/api/v1/auth/refresh", "/health", "/metrics"}
INTERNAL_PREFIX = "/internal/v1/"

async def auth_middleware(request: Request, call_next):
    path = request.url.path

    if path in PUBLIC_PATHS:
        return await call_next(request)                    # 无需鉴权

    if path.startswith(INTERNAL_PREFIX):
        verify_hmac_signature(request)                     # 内部 HMAC 鉴权
        return await call_next(request)

    verify_jwt_token(request)                              # 外部 JWT 鉴权
    return await call_next(request)
```

## 三、用户认证与授权体系

### 3.1 设计原则

> 个人量化系统并非公开服务，但云端 VPS 暴露在公网，**必须实现访问控制防止 API 被扫描滥用**。设计务必轻量，不引入 OAuth2 等复杂协议。

### 3.2 JWT 双 Token 机制

```
登录成功
  ↓
颁发 access_token（有效期 1 小时）
  + refresh_token（有效期 7 天，存 HttpOnly Cookie）
  ↓
客户端每次请求携带 Authorization: Bearer <access_token>
  ↓
access_token 过期 → 用 refresh_token 换新 access_token（无感刷新）
  ↓
refresh_token 过期 → 重新登录
```

```python
# backend/core/auth.py

from datetime import datetime, timedelta
from jose import jwt, JWTError
from backend.core.config import settings

ACCESS_TOKEN_EXPIRE = timedelta(hours=1)
REFRESH_TOKEN_EXPIRE = timedelta(days=7)

def create_access_token(user_id: str) -> str:
    payload = {
        "sub": user_id,
        "type": "access",
        "exp": datetime.utcnow() + ACCESS_TOKEN_EXPIRE,
        "iat": datetime.utcnow(),
    }
    return jwt.encode(payload, settings.SECRET_KEY, algorithm="HS256")

def verify_access_token(token: str) -> str:
    """返回 user_id，失败抛 HTTPException 401"""
    try:
        payload = jwt.decode(token, settings.SECRET_KEY, algorithms=["HS256"])
        if payload.get("type") != "access":
            raise JWTError("wrong token type")
        return payload["sub"]
    except JWTError:
        raise HTTPException(status_code=401, detail="Token invalid or expired")
```

### 3.3 内部 Tool API — HMAC 签名鉴权

Hermes Agent 的 Tool 调用走内部通道，使用独立的 HMAC-SHA256 请求签名，与 JWT 完全隔离。

```python
# hermes_agent/tools/base.py — Tool 内网请求签名

import hmac, hashlib, time

def sign_internal_request(path: str, body: str, secret: str) -> dict:
    """生成内部 API 请求头"""
    timestamp = str(int(time.time()))
    message = f"{timestamp}:{path}:{body}"
    signature = hmac.new(secret.encode(), message.encode(), hashlib.sha256).hexdigest()
    return {
        "X-Internal-Timestamp": timestamp,
        "X-Internal-Signature": signature,
    }

# backend/core/middleware.py — 服务端验证
def verify_hmac_signature(request: Request):
    timestamp = int(request.headers.get("X-Internal-Timestamp", "0"))
    if abs(time.time() - timestamp) > 30:          # 时间窗口 30 秒防重放
        raise HTTPException(status_code=401, detail="Request timestamp expired")
    # 重新计算签名并比较（省略）
```

### 3.4 Token 存储与轮换

| Token 类型 | 存储位置 | 特点 |
|:---|:---|:---|
| `access_token` | 前端内存（`useRef`） | 短期，不写 localStorage（防 XSS） |
| `refresh_token` | HttpOnly Cookie | 不可被 JS 读取，防 XSS |
| 内部 Tool Secret | 环境变量 `INTERNAL_API_SECRET` | 容器内部传递，不出 VPS |

## 四、K 线实时数据流水线

### 4.1 从 Tick 到 K 线的完整链路

K线实时性的核心是：**每根 K 线被分为"历史完成线"和"当前活跃线"两个状态**，分别用不同路径处理。

```
Futu OpenD 推送 Tick（逐笔成交）
  ↓
workers/quote_publisher.py
  ├── [实时 K 线聚合]
  │    按 period（1m/5m/15m/1h/1d）将 tick 聚合
  │    更新当前活跃 K 线的 O/H/L/C/V
  │    写入 Redis Hash：HSET kline:live:{symbol}:{period} ...
  │
  ├── [K 线闭合检测]
  │    当 tick 时间戳跨越 period 边界时：
  │    → 当前活跃 K 线"闭合"
  │    → RPUSH kline:hist:{symbol}:{period} {json}（写入历史队列）
  │    → 持久化写入 DuckDB/Parquet（异步）
  │    → PUBLISH kline.closed:{symbol}:{period} {json}（通知前端）
  │
  └── [实时推送]
       PUBLISH tick:{symbol} {tick_json}     ← 原始 Tick
       PUBLISH kline.update:{symbol}:{period} {kline_json}  ← K线更新
```

### 4.2 Redis 数据结构设计

```
# 1. 当前活跃 K 线（最新未闭合的那根）
HASH  kline:live:{symbol}:{period}
  open:    "182.50"
  high:    "183.20"
  low:     "181.90"
  close:   "183.10"
  volume:  "12500"
  time:    "1719475200000"    ← 该K线起始时间戳(ms)
  expires: 3600               ← TTL = 1小时

# 2. 历史 K 线缓冲（最近 300 根，用于 WebSocket 首屏加载）
LIST  kline:hist:{symbol}:{period}   ← RPUSH 追加，LTRIM 控制长度
  MAX_LEN = 300
  expires = 86400             ← TTL = 1天

# 3. 行情快照（最新 Tick 数据）
HASH  quote:{symbol}
  price:      "183.10"
  change_pct: "+0.87"
  volume:     "1234567"
  last_ts:    "1719475260000"
  expires:    300             ← TTL = 5分钟

# 4. 订阅注册表（防止重复订阅 Futu 同一标的）
SET   subscribed_symbols        ← 当前已向 Futu 订阅的标的集合
  "AAPL" "0700.HK" "00005.HK"
```

### 4.4 订阅管理与资源控制

```python
# backend/workers/quote_publisher.py — 订阅配额管理

class FutuSubscriptionManager:
    MAX_REALTIME_QUOTA = 100        # 富途免费版实时订阅上限
    _subscribed: set[str] = set()

    async def subscribe(self, symbol: str):
        if symbol in self._subscribed:
            return  # 已订阅，不重复

        if len(self._subscribed) >= self.MAX_REALTIME_QUOTA:
            logger.warning("subscription_quota_exceeded",
                          current=len(self._subscribed), max=self.MAX_REALTIME_QUOTA)
            # 降级：切换为 60 秒轮询快照，而非实时推送
            await self._start_polling_fallback(symbol)
            return

        await self._futu_client.subscribe(symbol, SubType.QUOTE)
        await self._futu_client.subscribe(symbol, SubType.K_1M)
        self._subscribed.add(symbol)
        logger.info("symbol_subscribed", symbol=symbol, total=len(self._subscribed))
```

## 五、历史 K 线多级缓存架构

### 5.1 三级缓存模型

```
请求: GET /api/v1/market/kline/AAPL?period=1d&count=250

Level 1: Redis（热缓存）
  查询 kline:hist:AAPL:1d
  命中率：近 300 根 K 线 ~100%
  延迟：< 5ms

Level 2: DuckDB（温缓存）
  SELECT * FROM 'data/klines/AAPL_1d.parquet'
  WHERE time >= ? ORDER BY time DESC LIMIT ?
  延迟：< 100ms（SSD 列式读取）

Level 3: Futu API（冷数据源）
  futu_client.request_history_kline(symbol, period, start, end)
  延迟：200ms ~ 2000ms（网络 + 限速）
  触发条件：请求超出本地缓存范围
  写回：同步写入 DuckDB Parquet，异步写入 Redis
```

```python
# backend/services/kline_warehouse.py — 分级缓存读取

class KlineWarehouse:
    async def get_klines(self, symbol: str, period: str, count: int) -> list[KlineBar]:
        # Level 1: Redis 热缓存
        cached = await self._read_redis_cache(symbol, period, count)
        if len(cached) >= count:
            logger.debug("kline_cache_hit", level="redis", symbol=symbol, count=len(cached))
            return cached[:count]

        # Level 2: DuckDB 本地数据湖
        local = await asyncio.to_thread(self._read_duckdb, symbol, period, count)
        if len(local) >= count:
            logger.debug("kline_cache_hit", level="duckdb", symbol=symbol, count=len(local))
            # 回填 Redis（异步，不阻塞响应）
            asyncio.create_task(self._backfill_redis(symbol, period, local))
            return local[:count]

        # Level 3: Futu API
        logger.info("kline_cache_miss", level="all", symbol=symbol,
                    count_needed=count, count_local=len(local))
        remote = await self._fetch_from_futu(symbol, period, count)
        # 写入 DuckDB + Redis（后台）
        asyncio.create_task(self._persist_klines(symbol, period, remote))
        return remote

    async def _read_redis_cache(self, symbol: str, period: str, count: int) -> list[KlineBar]:
        """读取 Redis List 中最近 N 根 K 线"""
        raw = await redis.lrange(f"kline:hist:{symbol}:{period}", -count, -1)
        return [KlineBar.model_validate_json(r) for r in raw]

    def _read_duckdb(self, symbol: str, period: str, count: int) -> list[KlineBar]:
        """DuckDB 直读 Parquet，在 thread pool 中执行（避免阻塞 asyncio）"""
        parquet_path = f"data/klines/{symbol}_{period}.parquet"
        if not Path(parquet_path).exists():
            return []
        conn = duckdb.connect(read_only=True)
        rows = conn.execute(
            f"SELECT * FROM '{parquet_path}' ORDER BY time DESC LIMIT {count}"
        ).fetchall()
        return [KlineBar(*r) for r in reversed(rows)]
```

### 5.2 缓存失效策略

| 缓存层 | TTL | 主动失效触发 |
|:---|:---|:---|
| Redis 活跃 K 线（`kline:live`） | 1 小时 | 每次 Tick 更新重置 TTL |
| Redis 历史 K 线（`kline:hist`） | 24 小时 | 每根 K 线闭合时 RPUSH + LTRIM |
| Redis 行情快照（`quote`） | 5 分钟 | 每次 Tick 更新重置 |
| DuckDB/Parquet | 永久存储 | 手动清理过期文件（定期维护脚本） |

### 5.3 历史 K 线数据同步脚本

```bash
# scripts/sync_klines.py — 定期（每天凌晨）同步增量 K 线数据
```

## 六、Hermes Agent 集成协议

### 6.1 Tool 调用架构

```
Hermes Agent（ReAct 推理）
  ↓ 决策调用某 Tool（如 get_broker_market_data）
hermes_agent/tools/broker_market_tool.py
  ↓ 构造内部 HTTP 请求 + HMAC 签名
  ↓ POST /internal/v1/market/quote/AAPL
backend 内部 Tool API（/internal/v1/）
  ↓ 验证 HMAC 签名
  ↓ 读取 Redis 行情快照（0ms 延迟，100% 缓存命中）
  ↓ 返回标准化 JSON
Tool 接收结果，返回给 Agent
  ↓ Verify 步骤：检查数据非空、时间戳新鲜
  ↓ 写入 Tool 执行上下文
Agent 继续 ReAct 推理
```

### 6.2 Tool API 路由约定

内部 Tool API 与公开 API 共享 Service 层，但走独立的路由前缀与鉴权：

```python
# backend/routers/internal_market.py

from fastapi import APIRouter, Request
from backend.core.middleware import require_internal_auth

router = APIRouter(prefix="/internal/v1", tags=["internal"])

@router.get("/market/quote/{symbol}", dependencies=[Depends(require_internal_auth)])
async def get_quote_for_tool(symbol: str):
    """供 Hermes Tool 调用，直接读 Redis 快照，不走外部 API"""
    quote = await redis.hgetall(f"quote:{symbol}")
    if not quote:
        raise HTTPException(404, detail=f"Quote not found for {symbol}")
    return {"status": "success", "data": quote, "source": "redis_cache"}
```

### 6.3 Tool 与 Service 层的边界

```
✅ Tool 允许调用的：
  - /internal/v1/market/*（只读行情缓存）
  - /internal/v1/screener（触发选股）
  - /internal/v1/knowledge/*（RAG 检索/写入）
  - /internal/v1/oms/order（沙箱下单，受 REAL_TRADE_EXECUTE 开关控制）

❌ Tool 绝对不允许：
  - 直接访问 Futu OpenD SDK（只有 workers/ 可以）
  - 直接读写 PostgreSQL（通过 Service 层中转）
  - 调用 /api/v1/* 公开接口（用户鉴权通道不适用于 Agent）
```

## 七、敏感数据安全隔离

### 7.1 凭证安全模型

```
等级 1 — 绝不落盘：Futu 账号密码、OpenAI API Key
  存储：OS 环境变量 / Docker Secret
  禁止：硬编码、写入 .env 文件提交 Git、写入数据库

等级 2 — 加密落盘：用户 JWT Secret Key
  存储：环境变量（生产环境），定期轮换（每 30 天）
  禁止：明文存入数据库或日志

等级 3 — 脱敏展示：交易账户资金数据
  规则：API 响应中金额字段超过 ¥10万的，默认脱敏（返回量级而非精确值）
  前端规则：非实盘聚焦模式下，资金字段显示为 ***
```

### 7.2 数据分级访问控制

```python
# backend/core/data_sensitivity.py — 响应字段过滤

class DataSensitivity:
    TRADING_SENSITIVE_FIELDS = {"account_balance", "available_cash", "total_pnl",
                                 "position_cost", "margin_ratio"}

    @staticmethod
    def mask_trading_data(data: dict, reveal_mode: bool = False) -> dict:
        """非实盘模式下，对敏感交易字段脱敏"""
        if reveal_mode:
            return data
        return {
            k: "***" if k in DataSensitivity.TRADING_SENSITIVE_FIELDS else v
            for k, v in data.items()
        }
```

## 八、高并发设计

### 8.1 异步任务分级

```python
# FastAPI 路由中的任务分级原则

# 轻量 I/O（Redis 读、PG 简单查询）→ 直接 await
@router.get("/market/quote/{symbol}")
async def get_quote(symbol: str):
    return await redis.hgetall(f"quote:{symbol}")  # < 5ms，直接 await

# CPU 密集（Pandas 运算、因子计算）→ asyncio.to_thread
@router.post("/screener")
async def run_screener(req: ScreenerRequest):
    result = await asyncio.to_thread(screener_service.filter, req)  # 放入线程池
    return result

# 超重计算（回测引擎、LLM 同步调用）→ ProcessPoolExecutor
@router.post("/backtest")
async def submit_backtest(req: BacktestRequest):
    loop = asyncio.get_event_loop()
    result = await loop.run_in_executor(process_pool, run_backtest, req)
    return result
```

### 8.2 背压控制

```python
# backend/core/middleware.py — WebSocket 背压保护

class BackpressureManager:
    SEND_QUEUE_MAX = 50     # 每个 WS 连接最大排队消息数

    async def send_with_backpressure(self, ws: WebSocket, message: dict):
        if self._queue_depth > self.SEND_QUEUE_MAX:
            # 丢弃旧消息（只保留最新 Tick），防止内存堆积
            await self._clear_tick_queue()
            logger.warning("ws_backpressure_triggered", client=ws.client.host)
        await ws.send_json(message)
```

## 九、存储架构详解

### 9.1 PostgreSQL 表设计原则

```sql
-- 表清单（核心表）
users              -- 用户账户（单用户系统只有1行，但保留扩展性）
orders             -- 订单记录（ACID，append-only）
positions          -- 持仓快照（每次变动 INSERT 新行，不 UPDATE）
strategies         -- 策略代码与参数（JSONB 存储参数）
backtest_results   -- 回测结果（Tear Sheet JSON）
alert_rules        -- 告警规则配置
audit_logs         -- 操作审计日志（只写不改）
knowledge_vectors  -- pgvector RAG 向量存储
```

### 9.2 Redis Key 命名规范

```
格式：{业务域}:{资源}:{标识符}:{可选维度}

行情：
  quote:{symbol}                    → Hash，行情快照
  kline:live:{symbol}:{period}      → Hash，当前活跃 K 线
  kline:hist:{symbol}:{period}      → List，历史 K 线队列（最近300根）

OMS：
  order:status:{order_id}           → String，订单状态
  position:snapshot:{user_id}       → Hash，持仓快照

Agent：
  session:chat:{session_id}         → List，对话历史
  tool:cache:{tool_name}:{hash}     → String，Tool 结果缓存（短 TTL）

系统：
  subscribed_symbols                → Set，已订阅标的集合
  rate_limit:{ip}:{endpoint}        → String，速率计数器
```

### 9.3 DuckDB 数据湖文件组织

```
data/
└── klines/
    ├── {SYMBOL}_{PERIOD}.parquet   ← 按标的+周期分文件
    │    e.g.: AAPL_1d.parquet
    │          0700_HK_1m.parquet
    ├── schema.json                 ← 字段类型定义（防止版本漂移）
    └── .gitignore                  ← 数据文件不入 Git
```
