## 一、现状盘点与差距结论（2026-07-12 代码摸底）

> 摸底修正了 TODO.md 的过时描述：前端**不是**臃肿单文件，三栏骨架已拆分（`frontend/src/features/strategy/`，约 2419 行、13 个文件）。真实差距如下：

| 任务 | 设计目标（docs/01 §四） | 代码现状 | 差距定级 |
|:---|:---|:---|:---:|
| STRAT-01 骨架 | 三栏 IDE + 4 个 Zustand Slice + Topbar 全局动作 | 骨架完成（`strategy-ide.tsx` ResizablePanelGroup）；但 Store 是单文件 `useStrategyStore.ts`（164 行）未拆 Slice；`topbar.tsx` 三个按钮**无 onClick**，动作逻辑散落在 `monaco-editor.tsx` / `right-sidebar.tsx` | 🟠 半成品 |
| STRAT-02 AI Diff | AI 生成不直接覆盖，经 `[Apply]` → Monaco Diff Editor 确认合入 | **零实现**。三条路径全部 `setCode()` 直接覆盖：`ai-chat.tsx:98`、`bottom-terminal.tsx`（Auto-Fix）、`monaco-editor.tsx`（AST 修复）；全局 Hermes 经 CustomEvent 投递同样直接覆盖 | 🔴 缺失 |
| STRAT-03 版本存储 | 保存自动生成 PG 版本快照 + 左侧时间线 + 一键恢复 + code_hash | `POST /strategy/save` 直接覆盖 `strategies/drafts/{name}.py`，无历史、无 hash；`core/models.py` 无任何策略表；`list` 返回 `version:"Draft"` 硬编码 | 🔴 缺失 |
| STRAT-04 Auto-Debug | 堆栈捕获 → `[AI 分析与修复]` → 投喂 AI → 修复 Diff | 按钮与 prompt 拼接已有（`bottom-terminal.tsx handleAutoFix`），但后端错误是**非结构化字符串**（traceback 截断 1500 字符塞进 `message`），且修复结果直接覆盖 | 🟠 半成品 |
| STRAT-05 参数面板 | AST 解析参数 → 滑块 → 拖动实时沙箱重跑 | 基本可用（`dynamic-strategy-form.tsx` `onPointerUp` 静默重跑），缺请求竞态治理与与新契约对齐 | 🟢 收尾 |

**架构级问题（本设计的收敛目标）**：

1. **两套 AI 链路并行**：策略页专用 `/strategy/generate`（NDJSON `{status,data}`）与全局 Hermes `/chat`（NDJSON `{type,...}` + CustomEvent 投递）各走各的，代码合入点不统一——Diff 工作流若不收口合入点，会出现"有的路径走 Diff、有的路径直接覆盖"的安全漏洞。
2. **错误契约缺失**：沙箱崩溃返回 `{status:"error", message:"<截断traceback>"}`，无 `error_code`、无结构化堆栈、无行号——Auto-Debug 的投喂质量与前端定位能力都受限。
3. **文件行数红线违规**：`backtest-report.tsx`（684 行）、`right-sidebar.tsx`（416 行）超 300 行红线，重构须一并拆分。
4. **版本与部署耦合**：`deploy-to-oms` 直接把编辑器当前内容写 `strategies/live/` 并启动 Bot——没有"部署自某个已验证版本"的概念，实盘代码不可溯源。

## 二、总体架构

```
┌────────────────────────── 前端 features/strategy/ ──────────────────────────┐
│  Topbar（全局动作：保存→版本快照 / 运行沙箱 / 部署）                            │
│ ┌──────────┬─────────────────────────────────┬───────────────────────────┐ │
│ │ Left     │ Main Workspace (Tabs)           │ Right Sidebar (Tabs)      │ │
│ │ 草稿库    │  [源码 Monaco] [回测报告]        │  [AI Copilot] [参数滑块]   │ │
│ │ 版本时间线│  ── Diff 覆盖层 ──               │                           │ │
│ │ (STRAT-03│  Monaco DiffEditor              │  AI 流式生成               │ │
│ │  时间线)  │  [Apply] [Reject]（STRAT-02）    │  → pendingDiff（不覆盖）   │ │
│ ├──────────┴─────────────────────────────────┴───────────────────────────┤ │
│ │ Bottom Terminal：AST 日志 / 运行报错(结构化) / [✨AI 分析与修复]（STRAT-04）│ │
│ └─────────────────────────────────────────────────────────────────────────┘ │
│  Zustand 4 Slices: editorSlice / aiSlice(diff状态机) / backtestSlice / layoutSlice │
└──────────────┬───────────────────────────────────────────────┬─────────────┘
               │ REST + NDJSON 流                               │ CustomEvent（Hermes 投递收口）
┌──────────────▼───────────────────────────────────────────────▼─────────────┐
│                        后端 routers/strategy.py                             │
│  /generate（AI 流）  /run-sandbox  /parse-config  /format                    │
│  /versions CRUD（新增，STRAT-03）   /deploy-to-oms（改造：从版本部署）          │
├──────────────────────────────────────────────────────────────────────────────┤
│  services/strategy_version_service.py（新增）── PostgreSQL: strategies +     │
│                                                strategy_versions（不可变快照）│
│  backend/backtest/sandbox.py ── AST 审查 + settrace 超时/内存熔断（沿用加固）  │
│  引擎：run_dynamic_sandbox_backtest（现网）⇄ engine v2 BacktestDriver（BT-01）│
└──────────────────────────────────────────────────────────────────────────────┘
```

**三条核心原则**：

1. **编辑器合入单点化**：任何来源的代码（AI 生成 / Auto-Fix / Hermes 投递 / 版本恢复）一律进入 `aiSlice.pendingDiff`，经 Diff 确认后才写 `editorSlice.code`。`setCode` 从公共 API 降级为 Diff 状态机的内部实现。
2. **版本是不可变事实**：`strategy_versions` 只增不改，`code_hash = sha256(源码)` 与 `docs/15 RunManifest.code_hash` 同算法——回测报告（BT-02）与实盘部署都引用版本，不引用"编辑器当前内容"。
3. **错误是结构化契约**：沙箱错误升级为 `error_code + data.error_detail{exc_type, exc_message, lineno, traceback, debug_tail}`，同时服务于前端展示（行号高亮）与 Auto-Debug 投喂。

## 三、前端架构设计

### 3.1 Zustand Store 拆分（STRAT-01 收口）

单 `useStrategyStore.ts` 拆为 4 个 Slice（组合成单 Store，切片文件独立）：

```
frontend/src/features/strategy/stores/
├── index.ts             # create() 组合入口 + 类型导出
├── editor.slice.ts      # code / isDirty / astParams / formatState
├── ai.slice.ts          # 对话历史 / 流式状态 / diff 状态机（见 3.2）
├── backtest.slice.ts    # 回测表单 / running / report / structuredError
└── layout.slice.ts      # 侧栏折叠 / 激活 Tab / 终端高度
```

约束：

- 每个 Slice 文件 ≤150 行；跨 Slice 只经 action 交互，禁止组件直接 set 别的 Slice 状态。
- 回测报告数据（equity_curve 数千点）**不进** Zustand 高频路径——报告为一次性快照，可存 `backtest.slice`，但滑块实时重跑的中间帧走组件局部 state，防全树重渲。
- `topbar.tsx` 按钮全部接线为 Slice action（`saveVersion()` / `runSandbox()` / `openDeployDialog()`），消除逻辑散落。

### 3.2 AI Diff 工作流（STRAT-02，核心设计）

**Diff 状态机（`ai.slice.ts`）**：

```
        AI流式完成 / Hermes投递 / AutoFix完成 / 版本恢复
idle ────────────────────────────────────────────► pendingDiff
  ▲                                                    │
  │   [Reject] / 开始新的生成                            │ [Apply]
  │◄───────────────────────────────────────────────────┤
  │                                                    ▼
  └──────────────────────────────────────────── applied（写 editorSlice.code，
                                                        标记 isDirty，回 idle）
```

状态字段：

```typescript
interface DiffState {
  status: 'idle' | 'streaming' | 'pendingDiff'
  original: string        // 进入 Diff 时的编辑器快照
  incoming: string        // AI/版本 产出的完整代码
  source: 'ai-chat' | 'auto-fix' | 'ast-fix' | 'hermes' | 'version-restore'
  meta?: { versionId?: string; errorRef?: string }   // 溯源
}
```

**UI 形态**：Main Workspace 增加 Diff 覆盖层——`pendingDiff` 时主编辑器切换为 Monaco `DiffEditor`（original vs incoming，side-by-side，窄屏降级 inline），顶部悬浮 `[Apply] [Reject] [在 Diff 中编辑 incoming]` 操作条。流式生成期间（`streaming`）右侧 AI Chat 正常打字机展示，**不动主编辑器**。

**四条来源路径全部收口**：

| 来源 | 现状 | 改造 |
|:---|:---|:---|
| AI Chat 生成（`/strategy/generate`） | `setCode` 直接覆盖 | 完成帧 → `enterDiff(source:'ai-chat')` |
| Auto-Fix（终端报错修复） | 直接覆盖 | → `enterDiff(source:'auto-fix', meta.errorRef)` |
| AST 一键修复（Monaco 内） | 直接覆盖 | → `enterDiff(source:'ast-fix')` |
| Hermes Copilot（CustomEvent `quant_strategy_code_invoke`） | `strategy-ide.tsx` 监听后直接 `setCode` | 监听器改为 `enterDiff(source:'hermes')` ——全局 AI 与本地 AI 走同一确认闸门 |

例外：编辑器为空（新建策略）时跳过 Diff 直接落码（Diff 对空文档无意义）。

**协议决策**：策略实验室 AI 保持走 `/strategy/generate`（NDJSON `{status: reasoning|success|error}`），**不**并入 Hermes `/chat`——理由：策略生成是单工具、无 ReAct 需求、prompt 高度专用，并入 Hermes 只增加 tool 编排开销与解析复杂度。两链路仅在**合入点**（Diff 状态机）收口，传输层保持独立。

### 3.3 文件行数红线治理（随 STRAT-01 重构一并执行）

| 违规文件 | 拆分方案 |
|:---|:---|
| `backtest-report.tsx`（684 行） | 拆为 `report/`目录：`report-tabs.tsx`（容器）+ `equity-panel.tsx`（Lightweight-Charts）+ `drawdown-panel.tsx` / `distribution-panel.tsx`（ECharts）+ `trades-table.tsx` + `debug-logs-panel.tsx`（**补齐缺失的 debug_logs Tab**，对齐 `features/trading/backtest.tsx` 已有能力） |

图表引擎遵守既有铁律：净值/K线 = Lightweight-Charts；热力图/分布 = ECharts。

### 3.4 参数面板与实时重跑（STRAT-05 收尾）

现有 `onPointerUp` 静默重跑保留，补三项治理：

1. **竞态取消**：`use-sandbox-run.ts` 持有 `AbortController`，新请求发起时 abort 在途请求；响应带请求序号，过期响应丢弃。
2. **节流**：拖动结束后 300ms debounce 才发请求；重跑期间滑块不禁用（乐观 UI），报告区显示轻量 loading 蒙层而非整页刷新。
3. **参数来源统一**：滑块定义来自 `POST /strategy/parse-config`（后端 AST 解析 `self.params` / 构造参数），BT-01 新契约落地后解析目标切换为 `Strategy.params` 类属性（见 §八 双轨）。

## 四、版本存储设计（STRAT-03）

### 4.1 数据模型（PostgreSQL，Alembic 迁移）

```
strategies                          # 策略主体（草稿的 SSOT 从文件系统迁到 PG）
├── id            UUID PK
├── name          VARCHAR(64) UNIQUE   # 类名，如 MaCrossAtrStrategy
├── display_name  VARCHAR(128)
├── head_version  UUID FK → strategy_versions.id   # 当前指针
├── deployed_version UUID FK NULL      # 当前实盘部署的版本（可空）
├── created_at / updated_at
└── is_archived   BOOL

strategy_versions                   # 不可变快照，只增不删
├── id            UUID PK
├── strategy_id   UUID FK
├── seq           INT                  # 该策略内递增序号（时间线展示用）
├── code          TEXT                 # 完整源码
├── code_hash     CHAR(64)             # sha256(code)，与 docs/15 RunManifest.code_hash 同算法
├── params_schema JSONB                # 保存时的 AST 解析结果（参数面板快照）
├── source        VARCHAR(16)          # manual | ai-apply | auto-fix | restore
├── message       VARCHAR(256) NULL    # 用户可选的版本说明
├── parent_id     UUID FK NULL         # 恢复/分叉溯源
└── created_at
UNIQUE(strategy_id, code_hash)      # 同内容不重复建版本（保存无变更时幂等返回）
```

**关键决策**：

- **PG 是草稿与版本的 SSOT**；`strategies/drafts/*.py` 文件系统降级为兼容读取层（迁移期一次性导入为各策略的 v1），迁移完成后 `save`/`list`/`draft/{name}` 全部改走 PG。
- **保存即版本**：`POST /strategy/save` 语义升级为"创建版本 + 移动 head 指针"。同 hash 幂等（不产生空版本）。无需"暂存/提交"两段式——个人系统，YAGNI。
- **恢复即新版本**：一键恢复不回退指针，而是以旧版本内容创建 `source=restore, parent_id=旧版本` 的新版本——时间线永远线性向前，避免分支管理复杂度。
- **不做版本内容 diff 存储**（增量压缩）：策略源码 <50KB，全量存储简单可靠。

### 4.2 API 设计（挂 `routers/strategy.py`，Pydantic 契约先行）

| 方法 | 路径 | 说明 |
|:---|:---|:---|
| POST | `/strategy/save` | 改造：入参增加 `message?`；落 PG 版本；响应返回 `{version_id, seq, code_hash}` |
| GET | `/strategy/{name}/versions?limit=50` | 版本时间线（seq 倒序，含 source/message/hash 摘要，**不含 code**） |
| GET | `/strategy/versions/{version_id}` | 单版本全文（供 Diff 预览与恢复） |
| POST | `/strategy/{name}/restore` | `{version_id}` → 创建 restore 新版本并返回代码（前端进 Diff 确认后落编辑器） |
| POST | `/strategy/deploy-to-oms` | 改造：入参从裸 `source_code` 改为 `{version_id}`——**只允许部署已存在的版本**；写 `strategies/live/` 文件时在头部注入 `# deployed from version {id} hash {hash}` 溯源注释；回写 `strategies.deployed_version` |

响应格式遵守全局契约 `{status, message, data}`。

### 4.3 与 BT-02 / docs/15 的对齐

- 回测请求（`run-sandbox`）增加可选 `version_id`：传入时后端按版本代码回测（而非编辑器裸代码），`RunManifest.code_hash` 直接取版本 hash——「已保存版本的回测」自动可复现；编辑器未保存代码的回测则实时计算 hash 并在报告标注 `unversioned`。
- 左侧时间线 UI：每个版本节点旁展示关联回测结果摘要（BT-02 落地后由 `backtest_reports.code_hash` 反查，本期仅预留 UI 插槽）。

## 五、沙箱隔离设计（现状确认 + 加固边界）

> 结论先行：现有 AST 沙箱（`sandbox.py` `SandboxSecurityVisitor`）+ `settrace` 熔断（5s/10s 超时、500MB 内存增量、逐行追踪）已达个人系统安全水位，**本期不引入进程级隔离**（与 docs/15 §八风险 5 的 V1 决策一致）。STRAT 系列对沙箱只做三处适配性改造：

1. **错误契约结构化**（服务 STRAT-04，见 §六）——sandbox/engine 抛错时保留 `exc_type / lineno / 完整 traceback`，不再截断进 message。
2. **AST 审查规则双轨**：BT-01 新契约（`Strategy` 基类 + `signals()` 类方法）落地后，`SandboxSecurityVisitor` 的基类识别与白名单需同步支持新旧两套契约（`ENGINE_V2_ENABLED` 开关联动，见 §八）。
3. **debug_mode 的产品化说明**：`debug_mode=true` 强制降级事件驱动引擎（放弃 VectorBT），前端在开关旁明示「调试日志 = 回测速度下降」，且 Auto-Debug 自动重跑时默认带 `debug_mode=true` 以获取逐 bar 日志。

**明确不做**（防过度设计）：Docker-in-Docker 每次回测起容器、gVisor/Firecracker、WASM 运行时——单用户系统，AST 白名单 + 资源熔断的威胁模型已覆盖"AI 生成了危险/低效代码"这一真实风险；进程级隔离列为 V2 观察项，触发条件是开放多用户。

## 六、Auto-Debug 闭环设计（STRAT-04）

### 6.1 后端：结构化错误契约

沙箱/引擎失败响应从裸字符串升级为（保持全局格式外壳）：

```json
{
  "status": "error",
  "message": "沙箱运行崩溃: ZeroDivisionError",
  "error_code": "SANDBOX_RUNTIME_ERROR",
  "data": {
    "error_detail": {
      "exc_type": "ZeroDivisionError",
      "exc_message": "float division by zero",
      "lineno": 42,
      "user_code_frame": "...出错行上下 3 行片段...",
      "traceback": "<完整 traceback，用户代码帧优先>",
      "debug_tail": ["最后 20 条 debug_logs（若有）"]
    }
  }
}
```

`error_code` 枚举：`SANDBOX_AST_REJECTED`（静态审查拒绝，附违规节点与行号）/ `SANDBOX_TIMEOUT` / `SANDBOX_OOM` / `SANDBOX_RUNTIME_ERROR` / `DATA_MISSING`（保留现有 `LOCAL_DATA_MISSING` 语义）。

### 6.2 前端：修复流水线

```
运行失败（structuredError 入 backtest.slice）
  → Bottom Terminal 结构化展示：exc_type 徽章 + 出错行号（点击跳转 Monaco 定位/波浪线标注）
  → [✨ AI 分析与修复]
      → 组装 FixContext：{当前代码, 参数, error_detail 全量, debug_tail}
      → POST /strategy/generate（专用 fix prompt 模板，要求返回完整修正代码）
      → 完成 → enterDiff(source:'auto-fix', meta.errorRef)   ← 不再直接覆盖
  → [Apply] 后提供「一键重跑验证」：自动以相同参数重跑沙箱，成功则在终端标记 ✅ 修复闭环
```

**熔断**：同一 errorRef 连续 3 次修复后仍失败 → 停止建议自动修复，提示人工介入（对齐工具调用熔断铁律）。

**修复 prompt 模板**（后端 `prompts/` 目录管理，要点）：注入沙箱约束清单（禁 while/iterrows/装饰器、import 白名单、必须矢量化）——防止 AI 修复代码再次撞 AST 审查，这是现状 Auto-Fix 成功率的主要损耗点。

## 八、契约双轨过渡（与 BT-01 的排期解耦）

STRAT 不等待 BT-01 完工，按以下开关矩阵过渡：

| 阶段 | 策略模板/AST 解析/沙箱审查 | 说明 |
|:---|:---|:---|
| BT-01 落地前 | 全部面向旧契约（`BaseStrategySandbox` + `_generate_signals`/`on_bar`） | STRAT 全系列可独立交付 |
| `ENGINE_V2_ENABLED` 开启后 | AI 生成 prompt 模板、新建策略脚手架、parse-config 解析目标、AST 审查基类识别切换为新 `Strategy` 契约；存量版本照常打开（旧契约经 `adapters/legacy.py` 仍可回测） | 切换点集中在后端 prompt 模板与 parser，前端无感 |

版本存储（STRAT-03）对契约**中立**——存的是源码文本与 hash，契约演进不产生迁移成本。

## 九、风险与开放问题

| # | 风险/开放问题 | 处置 |
|:---|:---|:---|
| 1 | drafts 文件 → PG 迁移期间双写不一致 | 一次性迁移脚本 + 迁移后文件系统只读兜底（读不到 PG 才回退文件）；两周后移除兜底 |
| 2 | Diff 强制确认增加操作摩擦，高频迭代场景可能烦人 | 空编辑器直落例外已设；再评审是否给 `ai-chat` 来源提供"本次会话内自动 Apply"用户开关（默认关） |
| 3 | 结构化错误暴露完整 traceback 是否泄露服务端路径 | 单用户内网系统风险可控；traceback 过滤 site-packages 帧、仅保留用户代码帧 + 引擎入口帧 |
| 4 | `strategy_versions` 无限增长 | 源码体量小（<50KB/版），单策略千版本 <50MB，不设清理策略（YAGNI）；`is_archived` 策略整体归档即可 |
| 5 | fix prompt 注入沙箱约束后 token 变长、生成变慢 | 约束清单压缩为 <300 token 的要点式规则；实测后调优 |
| 6 | STRAT-03a 与 BT-01f 都改 `deploy-to-oms`，合并冲突风险 | 排期错开（§七依赖表），后动手的一方基于先合入的实现叠加 |
