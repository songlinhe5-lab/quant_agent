## 一、框架定位与设计原则

### 1.1 核心定位

构建一套**数据源无关**的抽象层，使主应用（FastAPI Main App）与具体数据源实现完全解耦。所有数据源通过统一接口注册到 `DataSourceRegistry`，主应用只通过 Registry 获取数据，不感知底层是进程内调用、HTTP 远程调用，还是 TCP 直连。

### 1.2 设计原则

| 原则 | 说明 |
|:---|:---|
| **统一抽象 (Uniform Interface)** | 所有数据源（Futu / YFinance / AKShare / Finnhub / 未来新增）共享同一套接口定义，上层业务代码不区分数据源类型 |
| **双模运行 (Dual Mode)** | 每个数据源可内嵌在主进程（internal），也可作为独立 HTTP 服务运行在远程 VPS（external），模式通过环境变量切换 |
| **热插拔 (Hot-Swap)** | 数据源实例通过 `DataSourceRegistry` 注册表管理，支持运行时动态注册、注销、替换，无需重启服务 |
| **配置驱动 (Config-Driven)** | 数据源的启用/禁用、运行模式、远程地址、限流策略全部通过环境变量控制，禁止硬编码 |
| **零侵入 (Zero Intrusion)** | 新增数据源只需实现接口 + 注册，主应用现有代码无需任何改动 |
| **健康隔离 (Health Isolation)** | `/api/v1/health` 只检查主应用自身基础设施（Redis/DB），数据源可用性不影响主应用健康状态 |

### 1.3 当前支持的数据源

| 数据源 | 标识名 | 连接方式 | 主要能力 |
|:---|:---|:---|:---|
| Futu OpenD | `futu` | TCP (长连接) | 实时行情、历史K线、资金流、期权链 |
| Yahoo Finance | `yfinance` | HTTP (REST) | 行情快照、历史K线、宏观指标 |
| AKShare | `akshare` | HTTP (爬虫) | 南向资金、北向资金、沪深港通 |
| Finnhub | `finnhub` | HTTP (REST, API Key) | 内幕交易、新闻、财报 |

## 二、统一接口定义 (DataSource Interface)

### 2.1 核心接口 (Protocol)

所有数据源**必须**实现以下接口。使用 Python `Protocol`（结构化类型）而非 ABC，避免强制继承。

```
DataSourceInterface (Protocol)
├── name: str                    # 数据源唯一标识 (futu / yfinance / akshare / finnhub)
├── version: str                 # 接口版本号 (如 "1.0.0")
├── capabilities: list[str]      # 支持的操作列表
├── mode: str                    # 当前运行模式 (internal / external)
├── is_available() -> bool       # 当前是否可用
├── health() -> HealthInfo       # 健康状态详情
├── fetch(action, params) -> Result  # 统一数据获取入口
├── subscribe(action, params, callback) -> SubscriptionId  # 长连接订阅（可选）
└── unsubscribe(subscription_id) -> None                    # 取消订阅（可选）
```

**接口约束**：
- `subscribe` / `unsubscribe` 为**可选**接口，仅长连接型数据源（如 Futu）需要实现
- HTTP 短连接型数据源（YFinance / AKShare / Finnhub）可不实现订阅接口，调用时返回 `NotImplementedError`
- `fetch()` 是唯一的**必需**数据获取入口，所有请求统一走此方法

### 2.2 Result 统一返回结构

```
Result
├── status: "success" | "error" | "degraded" | "rate_limited"
├── data: Any                    # 实际数据内容（由各数据源自行定义结构）
├── source: str                  # 实际提供数据的节点标识 (如 "futu-local", "yf-node-ca-01")
├── latency_ms: float            # 本次请求耗时（毫秒）
├── cached: bool                 # 是否命中缓存
└── error: Optional[ErrorInfo]   # 错误时的详细信息
```

```
ErrorInfo
├── code: str                    # 错误码 (如 "FUTU_DISCONNECTED", "YFINANCE_429")
├── message: str                 # 人类可读的错误描述
├── retryable: bool              # 是否可重试（如限流=可重试，参数错误=不可重试）
├── category: "normal" | "rate_limit" | "quota_exhausted" | "ip_blocked"
│   # 错误分类：普通错误 vs 限流/配额/IP封禁，用于触发不同的退避策略
└── rate_limit_info: Optional[RateLimitInfo]  # 仅 category != "normal" 时填充
```

```
RateLimitInfo（限流详情，仅在限流类错误时填充）
├── retry_after_seconds: Optional[float]     # 服务端建议的重试等待秒数（如 Retry-After header）
├── estimated_reset_seconds: Optional[float] # 预估限流窗口重置时间
├── current_rpm: Optional[int]              # 当前推测的每分钟请求数
├── limit_rpm: Optional[int]                # 推测的限流阈值（每分钟请求数）
└── source_header: Optional[str]            # 原始限流响应头（如 "X-RateLimit-Remaining: 0"）
```

### 2.3 HealthInfo 结构

```
HealthInfo
├── healthy: bool                # 数据源自身是否健康
├── mode: str                    # internal / external
├── connected: bool              # 连接状态
├── uptime_seconds: float        # 运行时长
├── last_error: Optional[str]    # 最近一次错误信息
├── stats: dict                  # 统计数据 (请求总数、错误总数、当前并发等)
└── rate_limit_status: RateLimitStatus  # 限流状态感知
```

```
RateLimitStatus（数据源限流状态，HealthInfo 的组成部分）
├── is_throttled: bool            # 当前是否处于退避冷却期
├── throttle_until: Optional[float]  # 退避结束时间戳
├── estimated_rpm: Optional[int]     # 动态推测的当前实际 RPM
├── estimated_limit_rpm: Optional[int]  # 动态推测的限流阈值 RPM
├── consecutive_rate_limits: int     # 连续触发限流次数
├── total_rate_limits_1h: int        # 过去 1 小时触发限流总次数
└── backoff_strategy: str            # 当前退避策略 ("none" | "linear" | "exponential" | "adaptive")
```

### 2.4 能力矩阵 (Capabilities)

每个数据源声明其支持的 `action` 列表，Registry 据此做请求路由：

| 数据源 | capabilities |
|:---|:---|
| `futu` | `quote`, `history`, `fund_flow`, `option_chain`, `subscribe_quote` |
| `yfinance` | `quote`, `history`, `macro`, `batch_quote` |
| `akshare` | `southbound_flow`, `northbound_flow`, `hsgt_top10`, `macro_radar` |
| `finnhub` | `insider_trading`, `company_news`, `earnings`, `sec_filings` |

## 三、运行模式

### 3.1 模式定义

| 模式 | 标识 | 说明 | 适用场景 |
|:---|:---|:---|:---|
| **内部模式** | `internal` | 数据源代码在主 app 进程内执行，直接调用底层库/API | 开发、单机部署、资源受限环境 |
| **外部模式** | `external` | 数据源作为独立 HTTP 服务运行在远程 VPS，主 app 通过 HTTP 调用 | 分布式部署、限流隔离、独立扩缩容 |
| **混合模式** | `hybrid` | 优先 internal，失败自动降级到 external 节点 | 高可用生产环境 |

### 3.2 模式切换

模式通过环境变量控制，也支持运行时通过 Admin API 动态切换：

```
PUT /api/v1/datasource/{source_name}/mode
Content-Type: application/json
{"mode": "external"}
```

切换时 Registry 会：
1. 注销当前数据源实例
2. 根据新模式创建新实例（internal → 进程内对象；external → HTTP 客户端）
3. 注册新实例到 Registry
4. 记录切换日志与指标

## 四、部署拓扑

### 4.1 单机部署（开发 / 小规模生产）

```
单 VPS
+--------------------------------------------------+
| FastAPI Main App                                  |
| ├── DataSourceRegistry                            |
| │   ├── futu   (internal) → Futu OpenD TCP       |
| │   ├── yf     (internal) → Yahoo HTTP API        |
| │   ├── akshare(internal) → AKShare HTTP          |
| │   └── finnhub(internal) → Finnhub REST API      |
| ├── Redis                                         |
| └── PostgreSQL                                    |
+--------------------------------------------------+
```

所有数据源以 internal 模式运行在主 app 进程内。简单可靠，适合开发和小规模生产。

### 4.2 分布式部署（生产 / 高可用）

```
主 VPS (API + 业务逻辑)          数据源 VPS-A           数据源 VPS-B
+----------------------+         +------------------+   +------------------+
| FastAPI Main App     |         | DataSource Node  |   | DataSource Node  |
|                      |  HTTP   | FUTU=true        |   | YFINANCE=true    |
| DataSourceRegistry   |-------->| YFINANCE=true    |   | AKSHARE=true     |
|  |- futu → external  |         | Futu OpenD local |   |                  |
|  |- yf → external    |         | port 8000        |   | port 8000        |
|  |- akshare → ext    |         +------------------+   +------------------+
|  |- finnhub → int    |
|                      |
| Redis + PostgreSQL   |
+----------------------+
```

- 主 app 不直接依赖任何数据源库，只通过 HTTP 访问外部数据源节点
- 每个数据源节点是独立的 FastAPI 进程，可独立部署、扩缩容、重启
- 同一数据源可部署多个节点（多 IP 分散限流风险）

### 4.3 数据源节点架构

每个 external 数据源节点是一个轻量 FastAPI 应用：

```
DataSource Node (独立进程)
├── FastAPI 入口
│   ├── POST /ds/{source}/{action}   # 统一数据获取
│   ├── GET  /ds/{source}/health     # 节点健康
│   └── GET  /ds/health              # 节点总览
├── 数据源实现层
│   ├── FutuDataSource (TCP → OpenD)
│   ├── YFinanceDataSource (HTTP → Yahoo)
│   ├── AKShareDataSource (HTTP → 东方财富)
│   └── FinnhubDataSource (HTTP → Finnhub API)
├── HMAC 签名验证中间件
├── 限流器 (per-source)
└── 健康探针 (定时上报)
```

## 五、数据源注册与热插拔

### 5.1 DataSourceRegistry

全局注册表，管理所有数据源实例的生命周期。

**核心职责**：
- 启动时根据配置文件自动注册所有启用的数据源
- 提供运行时 API 支持动态注册/注销
- 定时健康探针，自动标记不健康节点
- 请求路由：根据数据源名 + action 找到对应实例

**注册表数据结构**：

```
DataSourceRegistry
├── _sources: dict[str, list[DataSourceInterface]]
│   # key = 数据源名称 (futu/yfinance/...)
│   # value = 该数据源的所有实例列表（支持多节点）
├── _health_status: dict[str, dict[str, HealthInfo]]
│   # key = source_name, inner key = instance_id
└── _circuit_breakers: dict[str, CircuitBreaker]
    # key = instance_id，每个实例一个独立熔断器
```

### 5.2 注册方式

**方式一：配置文件声明（启动时自动注册）**

```python
# 启动时读取配置，自动实例化并注册
for source_name, config in datasource_configs.items():
    if config.enabled:
        instance = create_datasource(source_name, config)
        registry.register(source_name, instance)
```

**方式二：运行时 API 动态注册/注销**

```
# 注册新数据源实例
POST /api/v1/datasource/register
Content-Type: application/json
{
  "name": "yfinance",
  "mode": "external",
  "nodes": ["http://vps-b:8000"],
  "capabilities": ["quote", "history", "macro"]
}

# 注销数据源实例
DELETE /api/v1/datasource/{source_name}/{instance_id}
```

### 5.3 健康探针

Registry 后台定时任务（默认 10s 间隔）：

1. 对每个注册的数据源实例调用 `health()`
2. 更新 `_health_status`
3. 连续 3 次不健康 → 触发熔断器 OPEN
4. 连续 30s 无响应 → 标记实例为 `unreachable`
5. 恢复后 → 熔断器 HALF_OPEN → 探测成功 → CLOSED

### 5.4 熔断降级策略

```
请求流程:
  Registry.get(source_name, action)
    │
    ├── 找到支持该 action 的实例列表
    │
    ├── 过滤掉熔断器 OPEN 的实例
    │
    ├── 按权重轮询选出首选实例
    │   ├── 调用成功 → 返回 Result ✅
    │   ├── 调用失败 → 熔断器 failures+1 → failover 下一个实例
    │   └── 熔断器 OPEN (连续失败 ≥3 次) → 60s 后半开探测
    │
    └── 所有实例失败
        ├── 有 STALE 缓存 → 返回过期数据 + stale=true 标记
        └── 无缓存 → 返回 Result(status="error") + 告警
```

## 六、HTTP API 协议（External 模式）

### 6.1 请求格式

复用现有 `routers/data_source.py` 的 HMAC 签名机制：

```
POST /ds/{source}/{action}
Headers:
  X-DS-Sig: <HMAC-SHA256 签名>
  X-DS-Timestamp: <Unix 时间戳>
  X-DS-Node-Id: <调用方节点标识>
  Content-Type: application/json

Body:
{
  "params": {
    "ticker": "AAPL",
    "period": "1y",
    "interval": "1d"
  }
}
```

### 6.2 响应格式

```
HTTP 200:
{
  "status": "success",
  "data": { ... },
  "source": "yf-node-ca-01",
  "latency_ms": 45.2,
  "cached": false
}

HTTP 429 (限流):
{
  "status": "rate_limited",
  "data": null,
  "error": {
    "code": "RATE_LIMITED",
    "message": "节点限流，请稍后重试",
    "retryable": true
  }
}

HTTP 500 (内部错误):
{
  "status": "error",
  "data": null,
  "error": {
    "code": "SOURCE_INTERNAL_ERROR",
    "message": "Futu OpenD 连接断开",
    "retryable": true
  }
}
```

### 6.3 HMAC 签名验证

```
签名算法: HMAC-SHA256
签名内容: {timestamp}.{body_md5}
密钥: DATASOURCE_HMAC_SECRET (所有节点共享)
有效期: 请求时间戳 ±300s（防重放攻击）
```

### 6.4 IP 白名单（可选）

通过 `DATASOURCE_ALLOWED_IPS` 环境变量限制可访问数据源节点的 IP 列表。未配置则不限制。

## 七、配置规范

### 7.1 环境变量矩阵

**全局配置**：

| 变量名 | 默认值 | 说明 |
|:---|:---|:---|
| `DATASOURCE_HMAC_SECRET` | — | 节点间 HMAC 签名密钥 |
| `DATASOURCE_RATE_LIMIT` | `200/minute` | 数据源节点全局限流 |

| `DATASOURCE_STALE_CACHE_TTL` | `604800` | STALE 缓存 TTL（秒，默认 7 天） |

| 变量名 | 默认值 | 说明 |
|:---|:---|:---|
| `DS_NODE_ID` | — | 节点唯一标识 |
| `DS_NODE_PORT` | `8000` | 监听端口 |
| `DS_SOURCES_ENABLED` | — | 本节点启用的数据源（逗号分隔，如 `futu,yfinance`） |
| `DS_HMAC_SECRET` | — | HMAC 签名密钥（须与主 app 一致） |
| `DS_RATE_LIMIT` | `100/minute` | 本节点限流 |

## 九、扩展新数据源的标准流程

### 9.1 四步接入

```
步骤 1: 实现接口
  └── 创建 XxxDataSource 类，实现 DataSourceInterface Protocol
      ├── name = "xxx"
      ├── capabilities = ["action_a", "action_b"]
      ├── is_available() → 检查底层连接
      ├── health() → 返回健康详情
      ├── fetch(action, params) → 根据 action 路由到具体实现
      └── [可选] subscribe / unsubscribe

步骤 2: 注册
  └── 在 Registry 注册入口添加该数据源的工厂方法
      └── 配置环境变量: DATASOURCE_XXX_ENABLED / MODE / NODES

步骤 3: 配置部署
  └── 单机模式: 直接在主 app 进程内实例化
  └── 分布式模式: 在数据源节点 VPS 的 DS_SOURCES_ENABLED 中添加 "xxx"

步骤 4: 声明能力
  └── 在本文档 §八 能力矩阵中更新该数据源的 capabilities 列表
```

- [ ] `health()` 返回标准 `HealthInfo` 结构

- [ ] 添加了对应的测试用例
- [ ] 更新了本文档能力矩阵
- [ ] 如果支持 external 模式：数据源节点可独立启动运行

## 十二、限流感知与自适应退避 (Rate Limit Awareness & Adaptive Throttling)

> **核心原则**：限流错误（HTTP 429 / 配额耗尽 / IP 封禁）与普通错误（网络超时 / 参数错误）有本质区别。  
> 限流是**数据源主动告知“你太快了”**，而非“你错了”。数据源模块必须区分这两类错误，并采取不同的内部退避策略。

### 12.1 错误分类体系

所有数据源错误必须归类到以下四种 category 之一（通过 `ErrorInfo.category` 字段暴露）：

| category | 含义 | 典型场景 | 内部处理策略 |
|:---|:---|:---|:---|
| `normal` | 普通业务/网络错误 | 参数错误、连接超时、数据解析失败 | 标准熔断器逻辑（连续失败 N 次熔断） |
| `rate_limit` | 频率限流 | HTTP 429、Yahoo Too Many Requests | **立即启动退避冷却**，不计入熔断器失败计数 |
| `quota_exhausted` | 配额耗尽 | Finnhub 日调用上限、免费 tier 用完 | **长周期退避**（按配额重置时间推算），告警 |
| `ip_blocked` | IP 封禁 | Yahoo 封 IP 段、反爬触发 | **熔断该节点 + 告警**，failover 到其他 IP 节点 |

**关键区分规则**：
- 限流类错误 **不计入** 熔断器的连续失败计数（避免限流导致不必要的节点熔断）
- 限流类错误触发**独立的退避机制**，与熔断器并行运作
- 普通错误才走标准熔断器 CLOSED → OPEN 状态机

### 12.2 数据源内部自适应退避

每个数据源实例内部维护一个 `RateLimitThrottler`，负责在限流触发后主动降速：

#### 退避策略矩阵

| 策略 | 标识 | 算法 | 适用场景 |
|:---|:---|:---|:---|
| 无退避 | `none` | 正常请求，不降速 | 无限流记录 |
| 线性退避 | `linear` | `wait = base_delay + step * consecutive_count` | 轻度限流（偶发 429） |
| 指数退避 | `exponential` | `wait = base_delay * 2^consecutive_count` (上限 `max_delay`) | 密集限流（连续 429） |
| 自适应退避 | `adaptive` | 根据历史限流频率动态调整请求间隔 | 长期运行的稳定数据源 |

#### 自适应退避算法

```
初始化:
  request_interval = 0           # 当前请求间隔（秒）
  consecutive_limits = 0         # 连续限流计数
  success_streak = 0             # 连续成功计数
  estimated_limit_rpm = null     # 推测限流阈值

限流触发时:
  consecutive_limits += 1
  success_streak = 0

  if retry_after_seconds:
    # 服务端明确告知等待时间，直接采纳
    request_interval = retry_after_seconds
  else:
    # 指数退避 + 抖动
    base = 2.0
    cap = 300.0                  # 上限 5 分钟
    request_interval = min(base * 2^consecutive_limits + random(0, 1), cap)

  estimated_limit_rpm = 60 / request_interval   # 推测限流阈值

请求成功时:
  success_streak += 1
  if success_streak >= 10:
    # 连续 10 次成功，逐步降速间隔
    request_interval = max(request_interval * 0.8, 0)
    consecutive_limits = max(consecutive_limits - 1, 0)
  if request_interval < 0.1:
    request_interval = 0        # 回到无退避状态

退避期间:
  - fetch() 调用前检查: if now < throttle_until → 直接返回 STALE 缓存
  - 不发起真实请求，避免加剧限流
  - 日志记录: event=datasource.throttled, source, wait_seconds
```

### 12.3 限流频率动态分析

数据源模块内部维护一个**限流频率分析器** (`RateLimitAnalyzer`)，基于历史数据推测限流模式：

#### 分析维度

| 维度 | 说明 | 用途 |
|:---|:---|:---|
| 时间窗口分布 | 统计每小时/每天的限流触发次数 | 识别限流高峰时段 |
| 请求量-限流相关性 | 分析请求频率与限流触发的关系 | 推测安全 RPM 阈值 |
| 恢复时间分布 | 统计限流后到恢复正常的平均耗时 | 校准退避时间参数 |
| IP-限流关联 | 多节点场景下对比各 IP 的限流频率 | 识别被特殊对待的 IP |

#### 推测频率查询 API

外部可通过以下端点查询数据源的限流分析结果：

```
GET /api/v1/datasource/{source_name}/rate-limit-analysis

Response:
{
  "source": "yfinance",
  "analysis_window": "24h",
  "estimated_limit_rpm": 30,
  "current_effective_rpm": 25,
  "total_rate_limits_24h": 47,
  "peak_hours": ["09:00-11:00", "14:00-16:00"],
  "avg_recovery_seconds": 62.5,
  "recommended_interval_seconds": 2.5,
  "confidence": 0.85,
  "history": [
    {"hour": "09:00", "requests": 120, "rate_limits": 8, "limit_ratio": 0.067},
    {"hour": "10:00", "requests": 95, "rate_limits": 3, "limit_ratio": 0.032}
  ]
}
```

#### 推测算法

```
输入: 过去 24h 的限流事件时间序列
      [{timestamp, source, action, response_code}, ...]

输出:
  estimated_limit_rpm: 推测的限流阈值
    = 成功请求的 RPM P75（第 75 百分位，排除异常高峰）
  recommended_interval: 推荐的安全请求间隔
    = 60 / (estimated_limit_rpm * 0.8)    # 保留 20% 安全裕度
  peak_hours: 限流高峰时段
    = 按小时聚合，限流率 > 5% 的时段
  avg_recovery_seconds: 平均恢复时间
    = 限流事件到下一次成功请求的平均间隔
  confidence: 推测可信度
    = min(sample_count / 100, 1.0)       # 样本越多越可信
```

### 12.4 外部感知机制

限流状态必须对外可感知，上层业务和监控系统能实时获知数据源的限流状况：

#### 12.4.1 HealthInfo 中的限流字段

`HealthInfo.rate_limit_status` 始终填充，即使当前无限流（`is_throttled: false`）也返回分析数据，供上层判断数据源的“限流压力”。

#### 12.4.2 专用限流端点

| 端点 | 说明 |
|:---|:---|
| `GET /api/v1/datasource/{name}/rate-limit-status` | 实时限流状态（是否退避中、退避剩余时间、推测 RPM） |
| `GET /api/v1/datasource/{name}/rate-limit-analysis` | 限流频率分析（24h 趋势、高峰时段、推荐间隔） |
| `GET /api/v1/datasource/rate-limit-overview` | 所有数据源限流状态总览 |

#### 12.4.3 Prometheus 限流指标

| 指标名 | 类型 | 标签 | 说明 |
|:---|:---|:---|:---|
| `ds_rate_limit_total` | Counter | `source, action` | 限流触发总次数 |
| `ds_rate_limit_throttled_seconds` | Gauge | `source` | 当前退避剩余秒数（0=无限流） |
| `ds_rate_limit_estimated_rpm` | Gauge | `source` | 推测的限流阈值 RPM |
| `ds_rate_limit_effective_rpm` | Gauge | `source` | 当前实际有效 RPM |
| `ds_backoff_state` | Gauge | `source` | 退避策略状态 (0=none, 1=linear, 2=exponential, 3=adaptive) |

#### 12.4.4 告警规则

| 规则 | 条件 | 级别 | 通知 |
|:---|:---|:---|:---|
| 限流频率飙升 | `rate(ds_rate_limit_total[5m]) > 10` | Warning | 飞书 |
| 长时间退避 | `ds_rate_limit_throttled_seconds > 120` 持续 >2min | Critical | 飞书 |
| 配额耗尽 | `ds_rate_limit_total` 的 `category=quota_exhausted` | Critical | 飞书 + 电话 |
| IP 封禁 | `ds_rate_limit_total` 的 `category=ip_blocked` | Critical | 飞书 + 电话 |

### 12.5 配置规范

| 变量名 | 默认值 | 说明 |
|:---|:---|:---|
| `DATASOURCE_{NAME}_BACKOFF_STRATEGY` | `adaptive` | 退避策略: `none` / `linear` / `exponential` / `adaptive` |
| `DATASOURCE_{NAME}_BACKOFF_BASE_DELAY` | `2.0` | 基础退避延迟（秒） |
| `DATASOURCE_{NAME}_BACKOFF_MAX_DELAY` | `300.0` | 最大退避延迟（秒） |
| `DATASOURCE_{NAME}_BACKOFF_JITTER` | `true` | 是否添加随机抖动（防雷群效应） |
| `DATASOURCE_{NAME}_RATE_LIMIT_ANALYSIS_WINDOW` | `24h` | 限流分析时间窗口 |
| `DATASOURCE_{NAME}_RATE_LIMIT_SAMPLE_MIN` | `100` | 推测可信度达到 1.0 所需的最小样本数 |

### 12.6 各数据源限流特征

| 数据源 | 限流机制 | 典型阈值 | 建议策略 |
|:---|:---|:---|:---|
| Futu | 并发连接数限制、订阅数量上限 | 500 订阅/账号 | 监控订阅数，接近上限时预警 |
| YFinance | 按出口 IP 的 HTTP 429 | ~30 RPM/IP（推测） | 多 IP 分散 + adaptive 退避 |
| AKShare | 反爬频率限制、IP 临时封禁 | ~60 RPM（推测） | 请求间隔 ≥ 1s + 指数退避 |
| Finnhub | API Key 级别限流 | 60 calls/min (免费) | 严格 RPM 控制 + quota 监控 |
