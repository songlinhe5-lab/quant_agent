# 分布式数据源服务架构 (V1.0)

> **文档定位**：针对 YFinance 等免授权但限流严格的数据源，设计多 VPS / 多进程驻留服务架构，通过服务注册发现 + 智能路由 + 自动 failover，最大化数据源吞吐能力，规避单点限流瓶颈。  
> **最后更新**：2026-06-28 | **版本**：V1.0  
> **关联文档**：`docs/03.`（后端架构）| `docs/06.`（部署方案）| `docs/12.`（运维手册）| `docs/TODO.md`（DIST 任务）

---

## 一、背景与痛点

### 1.1 现状审计

当前 `backend/services/yfinance_service.py`（1338 行）存在以下结构性缺陷：

| 症状 | 根因 | 影响 |
|:---|:---|:---|
| 单节点漏桶限流 `1 req/2s`，高峰期请求排队 | 单一出口 IP，Yahoo 429 一来全线瘫痪 | 量化选股、批量行情拉取严重受限 |
| 简易熔断器（裸时间戳 `_circuit_breaker_until`） | 未复用 `core/circuit_breaker.py` 通用熔断器 | 两套熔断逻辑并存，维护混乱 |
| 429 后强制冷却 60s，期间所有请求短路 | 无备选节点可 failover，只能干等 | 单次限流 = 60s 服务中断 |
| 宏观守护进程用 Redis 分布式锁做 Leader 选举 | 已有跨节点协调意识，但未泛化为通用服务注册 | 无法横向扩展数据获取能力 |

### 1.2 设计目标

| 目标 | 指标 |
|:---|:---|
| **水平扩展** | N 个独立出口 IP = N 倍限流配额，线性叠加 |
| **透明 failover** | 单节点 429 / 宕机，上层业务零感知自动切换 |
| **优雅降级** | 全部节点不可用时返回 STALE 缓存，而非硬错误 |
| **零侵入** | 上层调用方（routers / tools）接口签名不变 |
| **动态发现** | 子服务启动即注册，主服务自动感知，无需手动配置节点列表 |

### 2.2 核心设计原则

| 原则 | 说明 |
|:---|:---|
| **出口 IP 多样性** | 每个 Worker 跑在独立 IP 的机器上（VPS / 本机端口映射），Yahoo 按独立 IP 配额限流，N 个 IP = N 倍配额 |
| **无状态 Worker** | Worker 节点本身不持有跨请求状态，缓存统一在主服务 Redis 层，Worker 随时可重启 |
| **熔断器矩阵** | 复用 `core/circuit_breaker.py`，每个节点一个 service key（`yf_node:{id}`），节点级熔断互不影响 |
| **本地节点列表缓存** | 路由器缓存节点列表 5s，避免每次请求查 Redis，同时后台定时刷新 |
| **STALE 降级兜底** | 成功响应写入 Redis STALE 缓存（7d），全节点失败时返回过期数据 + 标记 `stale=true` |

## 三、核心组件设计

### 3.1 YFinance Worker Node（子服务）

每个 VPS / 本机端口跑一个独立 FastAPI 进程，职责单一：**接收 HTTP 请求 → 调用 yfinance → 返回数据**。

#### 3.1.1 目录结构

```
data_subservice/                    # 独立 Python 包，可单独部署
├── main.py                         # FastAPI 入口，启动时注册 + 启动心跳
├── registry_client.py              # 注册客户端（心跳 + 自动注册 + 断线重注册）
├── auth.py                         # HMAC-SHA256 签名验证中间件
├── routes/
│   ├── yfinance.py                 # yfinance 数据接口
│   └── health.py                   # 健康检查 / 节点状态
├── services/
│   └── yfinance_handler.py         # 从主服务迁移过来的 yfinance 核心逻辑
├── tests/
│   └── test_worker.py              # 子服务单测
├── pyproject.toml                  # 独立依赖管理
├── Dockerfile
└── docker-compose.yml
```

#### 3.1.2 子服务接口契约

子服务暴露给主服务的 HTTP 接口（均需 HMAC 签名）：

| 方法 | 路径 | 入参 | 出参 | 说明 |
|:---|:---|:---|:---|:---|
| POST | `/v1/quote` | `{ticker}` | `{price, change, change_pct, volume, ...}` | 单标的实时报价 |
| POST | `/v1/history` | `{ticker, period, interval}` | `{ohlcv: [{date, open, high, low, close, volume}]}` | 历史 K 线 |
| POST | `/v1/batch` | `{tickers: [...], period}` | `{quotes: {ticker: {...}}}` | 批量报价（微批处理） |
| POST | `/v1/macro` | `{indicators: [...]}` | `{data: {indicator: value}}` | 宏观指标批量拉取 |
| GET | `/v1/health` | — | `{status, node_id, load, queue_depth, last_error}` | 健康检查 |

#### 3.1.3 子服务启动流程

```
1. 读取环境变量: NODE_ID, NODE_WEIGHT, NODE_PORT, REDIS_URL, HMAC_SECRET
2. 连接 Redis，调用 registry_client.register() 写入节点信息
3. 启动心跳协程，每 10s 调用 registry_client.heartbeat()
4. 启动 FastAPI，监听 0.0.0.0:{NODE_PORT}
5. 优雅停机: SIGTERM → 取消心跳 → registry_client.deregister() → 退出
6. 断线重连: 心跳失败时指数退避重试，恢复后自动重新注册
```

#### 3.1.4 子服务限流策略

子服务内部保留 `RateLimitedSession`（从主服务迁移），但参数可按节点调整：

```python
# 加州节点（直连 Yahoo，延迟低）：1 req/2s
RateLimitedSession(max_requests=1, per_seconds=2.0)

# 国内节点（经代理，延迟高）：1 req/3s
RateLimitedSession(max_requests=1, per_seconds=3.0)
```

子服务被 Yahoo 限流时，返回 HTTP 429 给主服务（而非自己吞掉重试），由主服务路由器决定是否 failover。

### 3.2 ServiceRegistry（服务注册表）

基于 Redis 实现，复用现有 `redis_client.py`。

#### 3.2.1 Redis 数据结构

```
# 1. 节点信息 (Hash) — 每个节点一个 key
yf:nodes:california-01  → {
    "node_id": "california-01",
    "url": "http://100.64.0.2:9100",        # Tailscale 内网 IP 或公网 IP
    "weight": 3,                              # 权重（出口 IP 质量越好权重越高）
    "region": "us-ca",                        # 地域标识
    "status": "active",                       # active / draining / down
    "capabilities": ["quote","history","batch","macro"],  # 支持的接口能力
    "registered_at": "1719600000",
    "last_heartbeat": "1719600050",
    "total_requests": "15234",
    "total_errors": "23",
    "last_error": "429 Too Many Requests",
    "current_load": "2"                       # 当前并发请求数（用于负载均衡）
}

# 2. 心跳排序表 (Sorted Set) — 用于快速发现过期节点
yf:heartbeat  → { "california-01": 1719600050, "local-01": 1719600048 }

# 3. 节点 ID 集合 (Set) — 用于快速列举所有节点
yf:node_ids   → { "california-01", "local-01", "backup-01" }
```

#### 3.2.2 核心 API

```python
# backend/core/service_registry.py

class NodeInfo(BaseModel):
    node_id: str
    url: str
    weight: int
    region: str
    status: str  # active / draining / down
    capabilities: list[str]
    last_heartbeat: float
    total_requests: int
    total_errors: int
    current_load: int

class ServiceRegistry:
    """基于 Redis 的服务注册表"""

    HEARTBEAT_TIMEOUT = 30   # 秒，超过则认为节点死亡
    CLEANUP_INTERVAL = 15    # 秒，后台清理任务执行间隔
    NODE_PREFIX = "yf:nodes:"

    async def register(
        self, node_id, url, weight, region, capabilities
    ) -> None:
        """节点启动时调用，写入节点信息 + 加入心跳表"""

    async def heartbeat(
        self, node_id, stats: dict
    ) -> None:
        """每 10s 调用，更新 last_heartbeat + 运行统计"""

    async def deregister(self, node_id) -> None:
        """节点停机时调用，删除所有相关 key"""

    async def discover(
        self, capability: str | None = None
    ) -> list[NodeInfo]:
        """主服务调用，返回所有存活节点（可选按能力过滤）"""

    async def cleanup_dead_nodes(self) -> int:
        """后台定时任务，清理心跳超时的节点，返回清理数量"""

    async def mark_draining(self, node_id) -> None:
        """标记节点为 draining 状态（优雅下线，不再分配新请求）"""
```

#### 3.2.3 节点生命周期状态机

```
                register()
    [未注册] ──────────────► [active]
                              │  ▲
                heartbeat()   │  │ heartbeat() (正常)
                              │  │
              mark_draining() │  │
                              ▼  │
                          [draining]  (优雅下线：不再分配新请求，
                              │        但允许存量请求完成)
              优雅停机完成      │
              或心跳超时 30s    │
                              ▼
                          [down] ──cleanup_dead_nodes()──► [未注册] (key 删除)
```

### 3.3 YFinanceRouter（客户端路由器）— 核心

主服务侧最关键组件，替代当前 `yfinance_service.py` 直接调用 `yf.Ticker()` 的部分。

#### 3.3.1 核心方法

```python
# backend/core/yfinance_router.py

class YFinanceRouter:
    """
    YFinance 分布式路由器。
    对上层暴露与原 YFinanceService 相同的接口，内部做透明 failover。
    """

    NODE_CACHE_TTL = 5.0       # 本地节点列表缓存秒数
    REQUEST_TIMEOUT = 15.0     # 单节点 HTTP 请求超时
    DEFAULT_MAX_FAILOVER = 3   # 默认最大 failover 次数

    def __init__(self):
        self._registry = ServiceRegistry()
        self._nodes_cache: list[NodeInfo] = []
        self._cache_refresh_lock = asyncio.Lock()
        self._last_refresh = 0.0
        self._rr_counter = 0    # 加权轮询计数器
        self._cb = circuit_breaker  # 复用 core/circuit_breaker.py
        self._http = httpx.AsyncClient(timeout=self.REQUEST_TIMEOUT)

    async def _refresh_nodes(self, force: bool = False) -> None:
        """从 Redis 拉取最新节点列表（5s 本地缓存，避免高频查 Redis）
        使用 double-check locking 防止并发刷新。"""

    def _select_nodes(self, capability: str | None = None) -> list[NodeInfo]:
        """
        加权轮询选节点：
        1. 过滤掉熔断 OPEN 状态的节点（查 circuit_breaker.get_state）
        2. 过滤掉 status != 'active' 的节点
        3. 按 weight 展开为列表，轮询取下一个作为首选
        4. 返回排序后的候选链（首选 + 备选）
        """

    async def _call_node(
        self, node: NodeInfo, endpoint: str, payload: dict
    ) -> dict:
        """对单个节点发起 HMAC 签名的 HTTP 请求，包裹在熔断器中"""

    async def call(
        self, endpoint: str, payload: dict,
        max_failover: int = DEFAULT_MAX_FAILOVER
    ) -> dict:
        """
        核心路由方法：
        1. 刷新节点列表
        2. 加权轮询选出候选链
        3. 依次尝试候选节点，失败则 failover
        4. 全部失败 → 降级 STALE 缓存
        """

    async def _fallback_stale_cache(
        self, endpoint: str, payload: dict, last_error: Exception | None = None
    ) -> dict:
        """
        降级策略：
        1. 用 endpoint+payload 生成 cache_key，查 Redis yf:stale:{key}
        2. 命中 → 返回过期数据 + 标记 stale=true
        3. 未命中 → 抛出 YFinanceAllNodesDownError
        """

    async def _save_stale_cache(
        self, endpoint: str, payload: dict, data: dict
    ) -> None:
        """成功响应写入 STALE 缓存（TTL 7d），供全节点失败时降级"""
```

#### 3.3.2 加权轮询算法

```python
def _select_nodes(self) -> list[NodeInfo]:
    alive = [
        n for n in self._nodes_cache
        if n.status == "active"
        and self._cb.get_state(f"yf_node:{n.node_id}") != CircuitState.OPEN
    ]
    if not alive:
        return []

    # 按 weight 展开为虚拟节点列表
    expanded = []
    for node in alive:
        expanded.extend([node] * node.weight)

    # 轮询取起始位置
    start = self._rr_counter % len(expanded)
    self._rr_counter += 1

    # 候选链：从 start 开始轮转，去重保持顺序
    seen = set()
    candidates = []
    for i in range(len(expanded)):
        node = expanded[(start + i) % len(expanded)]
        if node.node_id not in seen:
            seen.add(node.node_id)
            candidates.append(node)
    return candidates
```

## 四、关键流程详解

### 4.1 请求路由 + Failover 流程

```
业务层调用 router.call("history", {"ticker": "AAPL", "period": "1y"})
    │
    ├─ 1. _refresh_nodes() 刷新节点列表（5s 本地缓存）
    │
    ├─ 2. _select_nodes() 加权轮询选出候选链: [A(w=3), B(w=2), C(w=1)]
    │
    ├─ 3. 请求节点 A（首选）
    │     ├─ 检查 A 的熔断器: yf_node:A
    │     │   ├─ OPEN → 跳过 A，转 B
    │     │   └─ CLOSED / HALF_OPEN → 继续
    │     ├─ 发送 HMAC 签名的 HTTP 请求
    │     │   ├─ 成功 → _save_stale_cache() 存档 → 返回数据 ✅
    │     │   ├─ HTTP 429 (节点被 Yahoo 限流) → 熔断器 failures+1 → 转 B
    │     │   ├─ 连接超时 / 网络错误 → 熔断器 failures+1 → 转 B
    │     │   └─ HTTP 5xx (子服务内部错误) → 熔断器 failures+1 → 转 B
    │     └─ 熔断器失败计数 ≥3 → 该节点熔断 OPEN（60s 后半开探测）
    │
    ├─ 4. 请求节点 B（第一次 failover）
    │     └─ 同上逻辑
    │
    ├─ 5. 请求节点 C（第二次 failover）
    │     └─ 同上逻辑
    │
    └─ 6. 全部候选节点失败 → 降级 _fallback_stale_cache()
          ├─ 查 Redis yf:stale:{md5(endpoint+payload)}
          │   ├─ 命中 → 返回过期数据 + stale=true 标记
          │   └─ 未命中 → 抛出 YFinanceAllNodesDownError
          └─ 触发飞书告警（接 OBS-02 告警通道）
```

### 4.2 限流检测与节点冷却

```
子服务侧 (data_subservice/routes/yfinance.py):
    调用 yfinance → 检测到 YFRateLimitError → 返回 HTTP 429

主服务侧 (YFinanceRouter._do_http_request):
    收到 HTTP 429 → 抛出 RateLimitError
    → 熔断器捕获异常，failures + 1
    → 路由器捕获，failover 到下一节点
    → 连续 3 次 429 → 该节点熔断 OPEN (60s)

熔断恢复:
    60s 后 → HALF_OPEN → 下次请求作为探测
    ├─ 探测成功 → CLOSED (节点恢复)
    └─ 探测失败 → 重新 OPEN (再等 60s)
```

### 4.3 节点注册与心跳

```
子服务启动:
    │
    ├─ registry_client.register("california-01", "http://100.64.0.2:9100", weight=3, ...)
    │   ├─ Redis HSET yf:nodes:california-01 {node_id, url, weight, ...}
    │   ├─ Redis SADD yf:node_ids "california-01"
    │   └─ Redis ZADD yf:heartbeat <timestamp> "california-01"
    │
    ├─ 每 10s 心跳:
    │   ├─ Redis HSET yf:nodes:california-01 last_heartbeat <now> total_requests <n>
    │   └─ Redis ZADD yf:heartbeat <now> "california-01"
    │   └─ 心跳失败 → 指数退避重试 (1s, 2s, 4s, 8s...)，恢复后重新注册
    │
    └─ 停机 (SIGTERM):
        ├─ 标记 status=draining (优雅下线，主服务不再分配新请求)
        ├─ 等待存量请求完成 (最多 10s)
        └─ registry_client.deregister("california-01")
            ├─ Redis HDEL yf:nodes:california-01
            ├─ Redis SREM yf:node_ids "california-01"
            └─ Redis ZREM yf:heartbeat "california-01"

主服务侧后台清理 (每 15s):
    └─ cleanup_dead_nodes()
        ├─ ZRANGEBYSCORE yf:heartbeat -inf <now-30s>  获取超时节点
        └─ 逐个删除 (防止子服务进程被 kill -9 后残留)
```

## 五、降级与容错策略矩阵

| 场景 | 路由器行为 | 用户感知 | 告警 |
|:---|:---|:---|:---|
| 单节点 429 限流 | 熔断该节点 60s，failover 到其他节点 | 无感（透明切换） | 否 |
| 单节点宕机（连接失败） | 熔断该节点，心跳超时后注册表剔除 | 无感 | 否 |
| 单节点响应超时（>15s） | 请求超时，熔断器 failures+1，failover | 无感 | 否 |
| 所有节点都 429 | 返回 Redis STALE 缓存（标记 `stale=true`） | 数据可能过期，前端显示 STALE overlay | 是（飞书） |
| 所有节点宕机 + 有 STALE 缓存 | 返回过期数据 + `stale=true` | 同上 | 是（飞书） |
| 所有节点宕机 + 无缓存 | 抛出 `YFinanceAllNodesDownError` | 前端显示错误 + 重试按钮 | 是（飞书） |
| Redis 不可用 | 路由器使用最后一次本地缓存的节点列表 | 降级运行，可能有死节点（靠熔断器兜底） | 是（飞书） |
| 子服务启动晚于主服务 | 主服务发现 0 节点 → 降级 STALE 缓存 | 首次请求可能失败，重试即可 | 否 |

## 七、部署方案

### 7.1 本机多进程（开发 / 测试环境）

```yaml
# docker-compose.dev.yml
services:
  yf-worker-1:
    build: ./data_subservice
    environment:
      NODE_ID: local-01
      NODE_WEIGHT: 1
      NODE_PORT: 9101
      REDIS_URL: redis://redis:6379
      HMAC_SECRET: ${INTERNAL_API_SECRET}
    depends_on: [redis]

  yf-worker-2:
    build: ./data_subservice
    environment:
      NODE_ID: local-02
      NODE_WEIGHT: 1
      NODE_PORT: 9102
      REDIS_URL: redis://redis:6379
      HMAC_SECRET: ${INTERNAL_API_SECRET}
    depends_on: [redis]
```

> ⚠️ **本机多进程局限**：同一台机器的多个进程共享同一出口 IP，Yahoo 仍按 IP 限流。本机多进程主要用于**功能验证**，真正的限流缓解需要**不同 IP 的 VPS**。

### 7.2 多 VPS 生产环境

```
北京主节点                              加州 VPS
┌─────────────────────┐                ┌─────────────────────┐
│ Main Backend        │                │ yf-worker-ca-01     │
│ + YFinanceRouter    │◄──Tailscale──►│ (weight=3)          │
│ + ServiceRegistry   │   内网通信      │ 出口 IP: 美国直连    │
│ + Redis             │                ├─────────────────────┤
│                     │                │ yf-worker-ca-02     │
│ yf-worker-bj-01     │                │ (weight=2)          │
│ (weight=1, 兜底)    │                │ 出口 IP: 美国直连    │
│ 出口 IP: 国内代理   │                └─────────────────────┘
└─────────────────────┘
```

**权重设计 rationale**：
- 加州节点直连 Yahoo，延迟低、不易被限 → 高权重
- 北京节点经代理出海，延迟高、易被限 → 低权重（兜底用）
- 加州 2 节点 weight 分别为 3/2，保证流量不均（避免完全均衡导致同时触发限流）

### 7.3 Tailscale 组网

```bash
# 加州 VPS 安装 Tailscale
curl -fsSL https://tailscale.com/install.sh | sh
tailscale up --accept-routes

# 北京主节点同样加入 Tailnet
# 子服务注册时使用 Tailscale IP (100.64.x.x) 而非公网 IP
```

### 8.2 Grafana 面板（新增）

- **节点健康总览**：每个节点的状态灯（active/draining/down）、熔断状态、最近心跳

### 8.3 告警规则

| 规则 | 条件 | 级别 | 通知 |
|:---|:---|:---|:---|

| STALE 降级飙升 | `rate(yf_router_stale_cache_hits[5m]) > 10` | Warning | 飞书 |

## 九、关键风险与对策

| 风险 | 对策 | 备注 |
|:---|:---|:---|
| Yahoo 按出口 IP 限流，多节点效果线性叠加 | ✅ 核心收益。N 个 IP = N 倍配额 | 本机多进程无效（同 IP） |
| 子服务间数据不一致（缓存各自独立） | 主服务侧统一走 Redis 缓存层，子服务只做无状态代理 | 子服务重启无副作用 |
| Redis 单点故障导致注册表不可用 | 路由器有本地节点列表缓存（5s TTL），可降级运行 | 靠熔断器兜底死节点 |
| 节点列表抖动（频繁注册/注销） | 心跳超时 30s + 本地缓存 5s，避免雪崩 | 指数退避重注册 |
| HMAC 密钥管理 | 复用 `INTERNAL_API_SECRET`，环境变量注入，不落盘 | 子服务启动时校验 |
| Yahoo 封 IP 段（非单 IP） | 跨地域 VPS（美国 + 其他地区），避免同机房 IP 段 | 长期风险 |
| 子服务被 kill -9 未注销 | 主服务后台清理任务每 15s 扫描心跳超时节点 | 自动恢复 |

## 十、落地路线图

### 阶段划分（每阶段可独立上线验证）

| 阶段 | 任务 | 产出 | 验收标准 |
|:---|:---|:---|:---|
| **P0** | `ServiceRegistry` + `YFinanceRouter` 骨架 | 主服务能发现节点、failover、降级 STALE | mock 子服务联调通过，failover 透明 |
| **P1** | `data_subservice` 子服务工程 + yfinance 核心逻辑迁移 | 子服务可独立启动、注册、响应请求 | 子服务单测通过，本机 2 节点联调 |
| **P2** | HMAC 签名 + Docker Compose 多节点编排 + 功能开关 | 本机 2 节点联调通过，灰度切换可用 | `YF_ROUTER_ENABLED=true` 业务无感 |
| **P3** | 加州 VPS 部署 + Tailscale 组网 + Prometheus 监控 | 生产环境 N 节点运行 | Grafana 面板可见节点健康，告警生效 |

### 依赖关系

```
P0 (ServiceRegistry + Router) ──► P1 (子服务工程) ──► P2 (HMAC + 编排)
                                                            │
                                                            ▼
                                                      P3 (VPS 部署 + 监控)
```

> **P0 与 P1 可并行**：P0 用 mock 子服务验证路由逻辑，P1 独立开发真实子服务，两者通过 HTTP 契约对接。
