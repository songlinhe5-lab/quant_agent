### 0.1 单次 PR 自查（3 分钟）

**触发时机**：每次提交代码或合并 PR 前。

**步骤**：

```
1. 确认 CI 通过
   ├─ lint（ruff / eslint）无 error   ← 对应 A2
   ├─ 单测全绿，覆盖率未下降           ← 对应 A1
   └─ 依赖漏洞扫描无新增高危           ← 对应 A4

2. 对照本次改动检查：
   ├─ 新增接口 → 格式是否符合 docs/10 响应结构？    ← 对应 B3
   ├─ 新增模型 → 是否与 docs/11 领域定义对齐？       ← 对应 B3
   └─ 改动是否关联了 TODO.md 中某个任务？将其勾选。  ← 对应 B1

3. 没有特殊问题 → 正常合并，结束。
   有问题 → 修复后重走步骤 1。
```

> 这一步**不需要打分**，只是防止低级错误进主干。CI 应自动完成步骤 1，步骤 2/3 是 30 秒心智确认。

### 0.2 两周 Sprint Review（15 分钟）

**触发时机**：每两周固定一天（建议周五下午）。

**步骤**：

```bash
# Step 1：统计 TODO 完成情况（2 min）
grep -c "^- \[x\]" docs/TODO.md   # 已完成数
grep -c "^- \[ \]" docs/TODO.md   # 未完成数

# 分别统计各优先级（手动数 or 搜索关键字）
# P0 完成率 = 已完成 P0 数 / P0 总数
# P1 完成率 = 已完成 P1 数 / P1 总数
```

```
# Step 2：回顾本 Sprint（5 min）
  ├─ 本 Sprint 完成了哪些 TODO？
  ├─ 有没有任务被意外阻塞？原因是什么？
  └─ 有没有发现新的问题需要补充进 TODO.md？

# Step 3：排定下 Sprint 优先级（5 min）
  └─ 按 docs/TODO.md 依赖顺序图，确认下两周攻坚哪几个任务

# Step 4：更新 TODO.md（3 min）
  └─ 将已完成任务打 [x]，补充新发现的任务
```

**产出**：更新后的 `docs/TODO.md`，不需要填评分卡。

### 0.3 月度全量评审（2 小时）

**触发时机**：每月最后一个工作日。这是**最重要的 Review**，完整走三个维度。

**完整流程**：

#### 第一步：运行自动化评审脚本（20 min）

```bash
# 从项目根目录执行
bash scripts/quality-review.sh 2>&1 | tee /tmp/quality-report-$(date +%Y%m).txt
```

脚本会自动输出：A1 测试覆盖率、A2 静态检查结果、A4 安全漏洞、B1 TODO 完成率、C1 Bundle 大小。**把输出结果保存备用**。

#### 第二步：填写 A 维度分数（15 min）

打开本文档 **§二**，对照脚本输出逐项计算：

```
A1 后端覆盖率 = 脚本输出的百分比 × 5（上限 5 分）
A1 前端覆盖率 = 脚本输出的百分比 × 4（上限 4 分）
A2 = 根据 error 数量扣分
A3 = 脚本输出的 radon 平均复杂度对照阈值
A4 = 根据漏洞数量和 gitleaks 结果打分
```

#### 第三步：人工核查 B 维度（40 min）

B1 已由脚本给出数据，直接计算。  
**B2 功能模块完整性**需要逐模块打开系统实际验证：

```
打开系统，逐一点击以下模块：
□ 行情看板：实时价格有更新？K 线图可切换周期？          → 打 2/3/满分
□ 选股器：输入条件有结果返回？结果格式正确？             → 打分
□ Agent 对话：提问有响应？Tool 调用有结果？              → 打分
□ 持仓/订单：数据显示正确？模拟下单流程走通？            → 打分
□ 客户端：打开 App/浏览器，核心功能是否可用？            → 打分
□ 监控看板：Grafana 有数据？APM 面板有数据？             → 打分
```

**B3 API 契约**运行快速核查命令：

```bash
# 扫描是否有裸路径（不带版本前缀）
grep -rn 'router\.\(get\|post\|put\|delete\)("/' backend/routers/ \
  | grep -v '/api/v1\|/ws/v1\|/sse/v1\|/internal/v1'
```

#### 第四步：获取 C 维度数据（25 min）

```
打开 Grafana → 查看以下面板（取近 30 天均值）：
  C1-1：K 线推送延迟 P99（目标 ≤50ms）
  C1-2：API 响应 P95（目标 ≤500ms）
  C2-1：服务可用率（目标 ≥99.5%）
  C2-2：三方数据源成功率（目标 ≥95%）

C3 人工核查：
  □ 上个月有跑过备份恢复演练？（查 docs/12 执行记录）
  □ 能用某个真实 trace_id 追到完整日志链路？（抽查验证）
  □ 上个月 Grafana 有告警触发并成功通知？（查 Alerting History）
  □ 有超 90 天未更新的重要依赖吗？
```

#### 第五步：汇总填写评分卡，判定等级（10 min）

复制 **§五** 的评分卡模板，填入各子项分数，求和，对照 §5.2 判定系统等级。

#### 第六步：记录趋势与改进行动（10 min）

```
1. 将本月分数填入 §六 §6.3 评分趋势记录表
2. 找出本次得分最低的 3 个子项 → 在评分卡"最大改进空间"处记录
3. 将这 3 个改进项转化为具体 TODO 任务补入 docs/TODO.md
4. 在 CHANGELOG.md 新增一行：| YYYY-MM | 综合分: XX | 等级: X | 主要改进: ... |
```

### 0.4 季度架构 Review（半天）

**触发时机**：每季度第一个月初。比月度 Review 多了**架构层面的反思**。

**额外步骤**（在月度全量评审基础上追加）：

```
1. 重读 docs/MASTER_REVIEW.md 中的 ADR（架构决策记录）
   └─ 每个 ADR 现在回头看还合理吗？
   └─ 有没有需要新增 ADR 的重大决策？

2. 技术债清单审视
   └─ 哪些 P3 任务应该升级为 P2？
   └─ 哪些 P2 任务长期未做已成技术债？
   └─ 有没有已完成任务的实现方式现在看来需要重构？

3. 文档与代码一致性抽查（各子系统抽 1 个模块）
   └─ 找一个 docs/subsystems/ 文档描述的架构模式
   └─ 打开对应源码，验证实际实现是否符合文档描述
   └─ 不符合 → 更新文档或提 TODO 任务修正代码

4. 产出：更新 MASTER_REVIEW.md（如有新决策）+ TODO.md（新任务）
```

## 一、评估框架设计

### 1.1 三个正交评估维度

```
┌──────────────────────────────────────────────────────────────────────┐
│                       系统综合健康评分 (0~100)                          │
├─────────────────┬────────────────────┬─────────────────────────────────┤
│  维度 A         │  维度 B            │  维度 C                          │
│  代码质量       │  需求还原度         │  系统健康度                      │
│  （怎么做的）   │  （做了什么）       │  （跑得怎么样）                  │
│  权重 35%       │  权重 40%          │  权重 25%                        │
└─────────────────┴────────────────────┴─────────────────────────────────┘
```

| 维度 | 核心问题 | 数据来源 | 评估频率 |
|:---|:---|:---|:---|
| **A. 代码质量** | 工程做得好不好？ | CI 报告、代码扫描 | 每次 PR + 每月汇总 |
| **B. 需求还原度** | 文档说的都实现了吗？ | TODO 完成率、功能对照 | 每两周 Sprint Review |
| **C. 系统健康度** | 生产跑得稳不稳？ | Grafana、日志、拨测 | 每日自动 + 每月人工 |

## 二、维度 A：代码质量评分（35%）

### A1 测试覆盖率（12 分）

| 子项 | 满分 | 计算方式 | 红线 |
|:---|:---:|:---|:---|
| 后端单测覆盖率 | 5 | 覆盖率 × 5（≥70% 得满分） | <40% 本项 0 分 |
| 前端单测覆盖率 | 4 | 覆盖率 × 4（≥60% 得满分） | <30% 本项 0 分 |
| Flutter 测试覆盖率 | 3 | 覆盖率 × 3（≥50% 得满分） | <20% 本项 0 分 |

```bash
# 获取后端覆盖率
cd backend && pytest --cov=. --cov-report=json && cat coverage.json | jq .totals.percent_covered

# 获取前端覆盖率
cd frontend && pnpm test run --coverage && cat coverage/coverage-summary.json | jq '.total.lines.pct'
```

### A2 静态检查通过率（8 分）

| 子项 | 满分 | 计算方式 |
|:---|:---:|:---|
| 后端 ruff/mypy 无 error | 4 | 全通过 4 分，每个 error -0.2 分（最低 0） |
| 前端 ESLint/tsc 无 error | 4 | 全通过 4 分，每个 error -0.2 分（最低 0） |

```bash
# 后端
cd backend && ruff check . && mypy . --ignore-missing-imports
# 前端
cd frontend && pnpm lint && pnpm exec tsc --noEmit
```

### A3 代码复杂度（5 分）

| 子项 | 满分 | 阈值 |
|:---|:---:|:---|
| 后端函数圈复杂度（Cyclomatic） | 3 | 平均 ≤10 得 3 分，>15 得 0 分，线性插值 |
| 单文件行数规范（≤500行/文件） | 2 | 违规文件数占比 ≤5% 得 2 分 |

```bash
# 圈复杂度
cd backend && radon cc . -s -a | tail -1  # 看 Average complexity

# 超长文件检查
find backend -name "*.py" | xargs wc -l | awk '$1>500{print}' | wc -l
find frontend/src -name "*.tsx" -o -name "*.ts" | xargs wc -l | awk '$1>500{print}' | wc -l
```

### A4 安全漏洞（10 分）

| 子项 | 满分 | 计算方式 |
|:---|:---:|:---|
| Python 依赖无高危漏洞 | 4 | 无高危(CRITICAL/HIGH) 4 分，每个 -1 分 |
| Node 依赖无高危漏洞 | 3 | 无高危 3 分，每个 -1 分 |
| Secrets 未硬编码（gitleaks） | 3 | 0 发现 3 分，有发现直接 0 分（一票否决） |

```bash
pip-audit --format json | jq '[.[] | select(.vulns[].fix_versions | length > 0)]'
pnpm audit --json | jq '.vulnerabilities | to_entries[] | select(.value.severity=="high" or .value.severity=="critical")'
gitleaks detect --source=. --redact 2>&1 | tail -5
```

## 三、维度 B：需求还原度评分（40%）

### B1 TODO 任务完成率（15 分）

> 基于 `docs/TODO.md` 的 checkbox 完成情况，按优先级加权计算。

| 优先级 | 权重系数 | 满分 |
|:---|:---:|:---:|
| P0（安全红线） | 3× | 6 |
| P1（核心功能） | 2× | 6 |
| P2（体验优化） | 1× | 3 |

```
P0 得分 = (P0已完成数 / P0总数) × 6
P1 得分 = (P1已完成数 / P1总数) × 6
P2 得分 = (P2已完成数 / P2总数) × 3
B1 总分 = P0得分 + P1得分 + P2得分
```

```bash
# 统计 TODO.md 各优先级完成情况
grep -c "^- \[x\]" docs/TODO.md   # 已完成
grep -c "^- \[ \]" docs/TODO.md   # 未完成
```

### B2 功能模块完整性（15 分）

> 对照 `docs/01` 产品功能文档与 `docs/10` API 规范，逐功能模块人工核查。每月执行一次。

| 功能模块 | 满分 | 评分标准（三档） |
|:---|:---:|:---|
| 行情看板（实时报价/K线/涨跌榜） | 3 | 完整可用=3 / 核心可用=2 / 不可用=0 |
| 智能选股器 | 2 | 同上 |
| AI 研报对话（Hermes Agent） | 3 | 完整可用=3 / 核心可用=2 / 不可用=0 |
| 订单管理/持仓（模拟盘） | 2 | 同上 |
| 历史回测 | 2 | 同上 |
| 客户端（Flutter 三端） | 1 | 任一平台可用=1 / 均不可用=0 |
| 运维监控看板（APM/Grafana） | 2 | 同上 |

### B3 API 契约符合度（10 分）

> 以 `docs/10` 接口规范为基准，对实现进行核查（每次重要 API 变更后执行）。

| 子项 | 满分 | 评分标准 |
|:---|:---:|:---|
| 响应结构符合 `{code,msg,data,ts}` | 3 | 全部符合=3，每个违规接口 -0.5 |
| 所有端点带 `/api/v1/` 版本前缀 | 2 | 全符合=2，有裸路径=0 |
| 错误码使用 `docs/10` §1.4 定义表 | 2 | 全符合=2，有自定义错误码 -0.5/个 |
| WebSocket 消息格式符合规范 | 2 | 符合=2，不符合=0 |
| 内部 Tool API HMAC 已启用 | 1 | 启用=1，未启用=0 |

```bash
# 自动扫描裸路径（不带 /api/v1/ 的路由注册）
grep -rn '@router\.\(get\|post\|put\|delete\)' backend/routers/ | grep -v '/api/v1/' | grep -v '/ws/' | grep -v '/sse/' | grep -v '/internal/'
```

## 四、维度 C：系统健康度评分（25%）

### C1 性能 SLO 达标率（10 分）

> 对照 `docs/09` 性能规范，从 Grafana/Prometheus 读取实际数据。

| SLO 指标 | 满分 | 目标 | 取数方式 |
|:---|:---:|:---|:---|
| K线推送延迟 P99 | 3 | ≤50ms | `prometheus_query: histogram_quantile(0.99, rate(kline_push_duration_ms_bucket[5m]))` |
| API 响应 P95 | 2 | ≤500ms | Grafana 近 7 天均值 |
| WebSocket 连接成功率 | 2 | ≥99.9% | `ws_connection_success_total / ws_connection_total` |
| 前端首屏 JS Bundle | 2 | ≤300KB gzipped | `pnpm build && du -sh dist/assets/*.js` |
| Flutter 渲染帧率 | 1 | ≥55fps (P50) | AppMonitor APM 数据 |

```bash
# 快速获取 Bundle 大小
cd frontend && pnpm build 2>/dev/null && find dist/assets -name "*.js" -exec gzip -c {} \; | wc -c | awk '{printf "%.1f KB\n", $1/1024}'
```

### C2 可用性与稳定性（8 分）

| 子项 | 满分 | 计算方式（取最近 30 天） |
|:---|:---:|:---|
| 服务整体可用率 | 3 | 可用率 × 3（≥99.5% 满分） |
| Futu OpenD 连接稳定率 | 2 | 连接在线时长 / 交易时段时长 × 2 |
| 三方数据源成功率（SVC-02） | 2 | 各源成功率均值 × 2（≥95% 满分） |
| 零 P0 安全事件 | 1 | 0 事件=1 分，有事件=0 分（一票否决） |

### C3 运维成熟度（7 分）

> 每月人工核查，验证运维基础设施是否就位。

| 子项 | 满分 | 核查方式 |
|:---|:---:|:---|
| 备份可恢复性（上月执行过演练） | 2 | 见 `docs/12` 灾难恢复验收标准 |
| 日志可查性（能用 trace_id 追全链路） | 2 | 抽取一个真实 trace_id 全链路追踪 |
| 告警实际生效（上月有触发并通知） | 2 | 查 Grafana Alerting 历史记录 |
| 依赖更新（无超 90 天未更新的重要依赖） | 1 | `pip list --outdated` / `pnpm outdated` |

### 5.2 系统等级定义

| 综合得分 | 等级 | 含义 | 行动要求 |

### 6.2 每月 Review 执行脚本

```bash
#!/bin/bash
# scripts/quality-review.sh
# 使用方法: bash scripts/quality-review.sh

set -e
REPORT_DATE=$(date +%Y-%m-%d)
echo "=== Quant Agent 质量评审 ${REPORT_DATE} ==="

echo ""
echo "── A1 测试覆盖率 ──"
cd backend
python -m pytest --cov=. --cov-report=term-missing --quiet 2>/dev/null | tail -3
cd ../frontend
pnpm test run --coverage --reporter=verbose 2>/dev/null | tail -5
cd ..

echo ""
echo "── A2 静态检查 ──"
cd backend && ruff check . --statistics 2>&1 | tail -3; cd ..
cd frontend && pnpm exec tsc --noEmit 2>&1 | tail -3; cd ..

echo ""
echo "── A4 安全漏洞 ──"
pip-audit --format columns 2>/dev/null | tail -10
pnpm audit 2>/dev/null | tail -5
gitleaks detect --source=. --redact --no-banner 2>&1 | grep -E "leaks found|No leaks"

echo ""
echo "── B1 TODO 完成率 ──"
DONE=$(grep -c "^- \[x\]" docs/TODO.md 2>/dev/null || echo 0)
TOTAL=$(grep -c "^- \[" docs/TODO.md 2>/dev/null || echo 1)
echo "已完成: ${DONE} / 总计: ${TOTAL} = $(echo "scale=1; $DONE * 100 / $TOTAL" | bc)%"

echo ""
echo "── A3 代码复杂度（后端）──"
cd backend && radon cc . -s -a 2>/dev/null | tail -3; cd ..

echo ""
echo "── C1 Bundle 大小 ──"
cd frontend
if [ -d "dist/assets" ]; then
  find dist/assets -name "*.js" -exec gzip -c {} \; | wc -c | awk '{printf "JS gzip 总计: %.1f KB\n", $1/1024}'
else
  echo "请先运行 pnpm build"
fi
cd ..

echo ""
echo "=== 评审完成，请对照 docs/13 评分卡手动填写维度 B2/B3 和 C2/C3 ==="
```
