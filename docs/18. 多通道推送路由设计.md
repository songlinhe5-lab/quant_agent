# 18. 多通道推送路由设计（ALERT-03 · Notification Router）

> **文档定位**：告警中心多通道推送子系统的架构设计（WS 应用内 + 飞书 + Telegram，含 P0~P3 优先级路由、双层冷却期、失败重试与降级）。对应 `docs/TODO.md ALERT-03`。
> **版本**：V1.0 | **日期**：2026-07-13 | **状态**：待评审
> **关联**：`docs/01 §10.4`（优先级语义）、`docs/12 §4`（运维告警阈值）、`backend/workers/alert_engine.py`（ALERT-01 已交付）、`docs/17 §6.3`（paper_drift 告警复用本路由）
> **强约束**：推送路由是**全系统通知的唯一出口**——AlertEngine、RateLimitAlertMonitor、PaperSettlement 偏离告警、Kill Switch 等全部经 `AlertDispatcher` 投递，禁止各模块直连 Webhook/Telegram。

## 一、现状盘点与目标

### 1.1 现状：三套推送路径各自为政

| 路径 | 实现 | 问题 |
|:---|:---|:---|
| **AlertEngine**（ALERT-01） | `_dispatch_event()` 遍历 `rule.channels`，调 `register_push` 注册的回调 | 生产环境**未注册任何适配器**；AlertEngine **未在 main/worker 启动**；失败只打 log，无重试/状态 |
| **NotificationService**（OBS-02） | `send_alert()` → Redis `macro_alerts` PubSub + 钉钉/企微/飞书 Webhook 广播 | **无 Telegram**；无优先级；无冷却；无重试；与 macro 新闻 WS 混用同一 Redis 频道 |
| **Hermes NotificationTool** | Agent Tool 直连 Telegram + 飞书 + Server酱 | 与告警引擎完全独立；飞书签名逻辑与 NotificationService **重复**且不一致（Tool 无 FEISHU_SECRET 签名） |
| **RateLimitAlertMonitor**（RL-11） | 15min 冷却 `(source, alert_type)` + 调 NotificationService | 冷却模式可复用，但未与 AlertEngine 规则冷却统一建模 |

**已有可复用资产**：

- `AlertChannel` 枚举（`in_app` / `feishu` / `telegram`）— `alert_models.py:55-60`
- 规则级冷却 — `AlertRule.cooldown_seconds` + `last_triggered_at`，引擎内已实现
- 飞书 Webhook + HMAC 签名 — `notification_service._send_to_feishu()`（含 URL 校验、FEISHU_SECRET）
- Telegram 环境变量 — `core/config.py`：`TELEGRAM_BOT_TOKEN` / `TELEGRAM_CHAT_ID`
- Hermes Tool 的 Telegram HTTP 调用 — `notification_tool.py:36-46`（可抽取为适配器）
- RL-11 通道级冷却范式 — `(channel_key, alert_type) → last_sent_at`，15min 窗口

### 1.2 目标（Definition of Done · ALERT-03）

- **统一出口**：新建 `AlertDispatcher`，所有告警/系统通知经此路由；AlertEngine 的 `register_push` 改为注册 Dispatcher 适配器。
- **P0~P3 优先级路由**：由 `severity` + `source` + `rule_type` 解析优先级，决定通道集合、并发/串行策略、UI 行为提示、重试激进度。
- **三通道落地**：`in_app`（专用 Redis PubSub + `/alert/ws`）、`feishu`（复用 OBS-02 签名逻辑）、`telegram`（Bot API）。
- **双层冷却**：规则冷却（已有，防重复触发）+ **通道冷却**（防 Webhook 风暴与平台限流）。
- **失败重试**：外部通道异步重试队列（指数退避，最多 3 次）；终态失败写入 dead-letter + 结构化日志。
- **投递可观测**：每次推送记录 `DeliveryRecord`（通道/状态/attempt/latency/error），供 ALERT-04 前端「投递详情」与运维排查。

### 1.3 非目标（YAGNI）

- ❌ 不做邮件/SMS/APNs/FCM（ALERT-05 移动端推送另任务；本设计预留 `AlertChannel` 扩展位）
- ❌ 不做用户级多收件人/多 Chat ID（单用户系统，env 全局配置一个 Telegram chat）
- ❌ 不做消息模板国际化（中文金融术语硬编码即可）
- ❌ 不引入 Celery/RabbitMQ（asyncio 内存队列 + Redis dead-letter 够用）

## 二、总体架构

```
  事件源（均调用 AlertDispatcher.dispatch，禁止直连 Webhook）
  ┌──────────────────┬────────────────────┬─────────────────────────┐
  │ AlertEngine      │ RateLimitAlertMonitor│ 系统事件（Kill Switch /  │
  │ (用户规则触发)    │ (RL-11 限流)         │  PT paper_drift / Worker)│
  └────────┬─────────┴──────────┬───────────┴────────────┬────────────┘
           │                    │                        │
           └────────────────────┼────────────────────────┘
                                ▼
                    ┌───────────────────────┐
                    │   AlertDispatcher     │
                    │  1. PriorityResolver  │  severity+source → P0~P3
                    │  2. ChannelPlanner    │  优先级 × 规则.channels → 通道计划
                    │  3. CooldownGate      │  规则冷却(引擎) + 通道冷却(本层)
                    │  4. DeliveryExecutor  │  并发/串行 + 重试 + 降级
                    │  5. DeliveryRecorder │  写 DeliveryRecord
                    └───────────┬───────────┘
                                │
           ┌────────────────────┼────────────────────┐
           ▼                    ▼                    ▼
   InAppAdapter          FeishuAdapter         TelegramAdapter
   Redis PubSub          httpx + 签名          Bot API httpx
   quant:alerts:push     FEISHU_WEBHOOK_*      TELEGRAM_* env
           │                    │                    │
           ▼                    │                    │
   /api/v1/alert/ws           └──────────┬─────────┘
   (前端 Toast/弹窗/角标)                  ▼
                              RetryQueue (asyncio Task)
                              1s → 4s → 16s，max 3

```

**与 AlertEngine 的边界**：规则评估与规则级冷却留在 `AlertEngine`；**推送语义**（优先级、通道冷却、重试、降级）全部下沉到 `AlertDispatcher`。引擎触发后只调 `await dispatcher.dispatch(AlertEvent, rule)`。

## 三、优先级模型（P0~P3）

对齐 `docs/01 §10.4`，扩展为可执行的 routing table：

### 3.1 优先级解析（`PriorityResolver`）

```python
class NotificationPriority(str, Enum):
    P0 = "p0"   # 紧急：止损/熔断/IP封禁
    P1 = "p1"   # 高：策略信号/配额耗尽/CRITICAL 用户规则
    P2 = "p2"   # 中：委托成交/WARNING 用户规则/限流飙升
    P3 = "p3"   # 低：AI 完成/INFO/选股匹配/纸面偏离(非紧急)
```

**解析规则**（按序匹配，首条命中）：

| 条件 | 优先级 |
|:---|:---:|
| `source == "kill_switch"` 或 `source == "oms_circuit"` | **P0** |
| `severity == CRITICAL` 且 `source ∈ {rate_limit, system}` | **P0** |
| `severity == CRITICAL` 或 `rule_type == strategy_signal` | **P1** |
| `severity == WARNING` 或 `source == "trade_fill"` | **P2** |
| 其他（INFO、选股匹配、paper_drift 轻度偏离等） | **P3** |

`AlertEvent` 扩展字段（向后兼容，默认 `source="user_rule"`）：

```python
class AlertEvent(BaseModel):
    ...
    source: str = "user_rule"           # user_rule | rate_limit | system | paper_drift | trade_fill | kill_switch
    priority: NotificationPriority | None = None  # 为空时由 Resolver 计算
    ui_hint: dict = {}                  # 前端行为提示，见 3.2
```

### 3.2 优先级 → 路由策略矩阵

| 优先级 | 默认通道（在用户 `rule.channels` 基础上的**最小集合**） | 并发策略 | 外部通道重试 | 通道冷却 | 前端 `ui_hint` |
|:---:|:---|:---|:---|:---|:---|
| **P0** | `in_app` + `feishu` + `telegram`（**强制全开**，忽略用户只选 in_app 的配置——安全例外） | **并行**推送所有通道 | max **5** 次，退避 1/2/4/8/16s | **60s**（同 fingerprint） | `{mode:"fullscreen", flash:true, sound:true}` |
| **P1** | 用户配置的 channels；若为空则 `in_app` + `telegram` | 并行 | max **3** 次，1/4/16s | **300s** | `{mode:"toast", duration:8000}` |
| **P2** | 用户 channels；默认 `in_app` + `feishu` | 并行 | max **2** 次，1/4s | **300s** | `{mode:"statusbar"}` |
| **P3** | 用户 channels；默认仅 `in_app` | **串行**（in_app 成功则跳过外部，省 Webhook 配额） | max **1** 次（仅 in_app 失败才试外部） | **900s** | `{mode:"badge"}` |

**P0 强制全开的安全例外**：止损/熔断类告警即使用户关闭了 Telegram，仍推送外部通道——个人系统可接受，需在规则创建 UI（ALERT-04）明示。

### 3.3 通道优先级（同优先级内的发送顺序）

当**串行模式**（P3）或**降级**触发时，通道尝试顺序：

```
in_app  →  feishu  →  telegram
```

理由：`in_app` 零成本零限流；飞书是国内运维主通道（OBS-02 已配置）；Telegram 跨境备用。并行模式（P0/P1/P2）无顺序差异。

**降级规则**：P1/P2 并行推送时，若 `telegram` 连续 3 次全局失败（适配器 health=down），当次事件跳过 Telegram，仅 in_app+feishu；health 每 5min 探活恢复。

## 四、双层冷却期

### 4.1 第一层：规则冷却（已有，保留在 AlertEngine）

- 键：`AlertRule.last_triggered_at` + `cooldown_seconds`（默认 300s，API 最小 60s）
- 语义：**同一规则**在冷却期内不重复评估触发——防止价格来回穿越阈值产生事件风暴
- 不变：继续在 `evaluate_quote()` 内检查，Dispatcher 不重复

### 4.2 第二层：通道冷却（新增，AlertDispatcher）

- 键：`quant:alerts:cooldown:{channel}:{fingerprint}`，TTL = 通道冷却秒数
- **fingerprint** 计算：`sha256(source + rule_id + ticker + severity)[:16]`（不含 trigger_value——同规则同标的同级别告警在窗口内只推一次外部消息）
- 语义：即使**不同规则**在同一标的上同时触发 WARNING，飞书/Telegram 在 300s 内也只收一条合并提示（可选，见 4.3）
- **in_app 不受通道冷却限制**（应用内可以高频，靠前端 debounce）；仅 `feishu` / `telegram` 受冷却

| 优先级 | 通道冷却 TTL |
|:---:|:---:|
| P0 | 60s |
| P1/P2 | 300s |
| P3 | 900s |

### 4.3 可选合并推送（P2/P3 外部通道，V1.1 观察项）

冷却窗口内多条被抑制的告警，在窗口结束时发一条摘要：`「过去 5 分钟：AAPL 突破×2、MSFT 跌破×1」`。V1 **不做合并**，只做抑制+计数（`DeliveryRecord` 标记 `suppressed=true`），降低实现复杂度。

## 六、失败重试与 Dead Letter

### 6.1 RetryQueue（进程内 asyncio）

```
DeliveryExecutor 对外部通道失败时：
  1. 写 DeliveryRecord(status=failed, attempt=N, error=...)
  2. 若 attempt < max_retries(priority)：
       asyncio.create_task(_retry_later(task, delay=backoff[attempt]))
  3. 否则：
       LPUSH quant:alerts:dlq {task_json}  （保留 7 天，LTRIM 10000）
       logger.error + structlog 结构化字段
       P0/P1：可选降级——若 telegram 全失败，feishu 已 success 则标记 partial_success
```

**退避表**：

| 优先级 | 退避序列（秒） | max_attempts |
|:---:|:---|:---:|
| P0 | 1, 2, 4, 8, 16 | 5 |
| P1 | 1, 4, 16 | 3 |
| P2 | 1, 4 | 2 |
| P3 | （不重试外部） | 1 |

**可重试错误**：HTTP 5xx、timeout、连接错误、Telegram 429、飞书 `code=11232`（频率限制）。
**不可重试**：HTTP 4xx（除 429）、invalid token、chat not found——立即 dead-letter，不浪费重试。

### 6.2 投递记录（DeliveryRecord）

Redis Hash（热数据，7 天 TTL）+ 可选 PG 表（ALERT-03b 扩展）：

```python
class DeliveryRecord(BaseModel):
    delivery_id: str
    event_id: str
    channel: AlertChannel
    priority: NotificationPriority
    status: Literal["pending", "success", "failed", "suppressed", "partial"]
    attempt: int
    latency_ms: float | None
    error: str | None
    created_at: float
```

键：`quant:alerts:delivery:{event_id}` → JSON array of records。

## 七、AlertDispatcher 核心接口

```python
# backend/services/alert_dispatcher.py

class AlertDispatcher:
    async def dispatch(
        self,
        event: AlertEvent,
        rule: AlertRule | None = None,          # 用户规则触发时传入
        channels: list[AlertChannel] | None = None,  # 系统事件显式指定
    ) -> DispatchResult:
        """
        1. priority = event.priority or PriorityResolver.resolve(event, rule)
        2. channels = ChannelPlanner.plan(priority, rule.channels if rule else channels)
        3. for ch in channels:
               if ch != IN_APP and CooldownGate.is_blocked(ch, fingerprint): mark suppressed
        4. DeliveryExecutor.execute(planned_channels, event, priority)
        5. DeliveryRecorder.persist(...)
        return DispatchResult(event_id, per_channel_status)
        """

    async def health(self) -> dict:
        """各适配器 enabled 状态 + 最近 1h 成功率（供 /alert/engine/status 扩展）"""
```

**AlertEngine 改造点**（最小 diff）：

```python
# alert_engine.py _dispatch_event 替换为：
await alert_dispatcher.dispatch(event, rule=self._rules.get(event.rule_id))
# 删除 register_push 公开 API（测试改为 mock dispatcher）
```

**Worker 启动接线**（当前 AlertEngine 未启动，ALERT-03 一并完成）：

```python
# worker.py 或 main.py lifespan:
engine = AlertEngine(redis)
dispatcher = AlertDispatcher(redis)
await engine.start()
await dispatcher.start()  # 启动 RetryQueue 消费 task
```
