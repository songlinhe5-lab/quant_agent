# Vibe Coding 工程规范 (V4.3)

> **定位**：跨模型 AI 辅助编码（Vibe Coding）的操作手册——回答「怎么做」：控幻觉、控 Token、稳架构、核心少动、改动需确认。  
> **核心哲学**：够用就行（YAGNI）、简单优先（KISS）、测试同行（Test-Alongside）、日志可追（Observable-by-Default）、**改动先确认（Confirm-Before-Edit）**。  
> **最后更新**：2026-07-13 | **版本**：V4.3

## 零、SSOT 层级与多模型加载规则（V4.3 新增）

> **问题**：双宪法 + 行数冲突 + 过载上下文 → 不同大模型随机选一套规则，产出漂移。  
> **原则**：冲突时以上层为准；下层只加载任务相关切片，禁止整库灌入。

### 0.1 单一事实来源（SSOT）层级

| 层级 | 文档 | 职责 | 冲突时 |
|:---|:---|:---|:---|
| **L0** | `.cursor/rules/vibe-coding.mdc` | 铁律（alwaysApply）：栈禁令、零 GC、安全锁 | **最高** |
| **L1** | `AGENTS.md` 附录 A | 工程规范摘要 + 附录 B 上下文排除 | 次之 |
| **L2** | **本文档 `docs/02`** | SOP：SMART / MCP / Verify / PCE / 冻结区 | 操作流程以此为准 |
| **L3** | `docs/15~19` · `docs/subsystems/*` · `docs/TODO.md` | 子系统架构与任务 SSOT | 按任务 **单章** 加载 |
| **L4** | `AI_INSTRUCTIONS.md` | 详细参考（非必加载） | 与 L0/L1 冲突时忽略 L4 |

**禁止**：会话开头同时全文加载 L0+L1+L2+L4。默认：**L0 已由 Cursor 注入**；人工/Agent 再按 §7.1 最小上下文包补齐 L2 相关节 + L3 单章。

### 0.2 多模型统一行为契约

无论 Cursor / Claude / GPT / Copilot / 本地模型，生成代码前必须遵守：

1. **先定位再改写** — Grep/Glob 证明符号存在，禁止臆造路径/API/env  
2. **先计划再动手** — 见 §7.6 Plan-Confirm-Execute；核心区改动必须等人确认  
3. **最小 diff** — 单次任务涉及文件 ≤5；超限先拆任务或二次确认  
4. **测试同行** — 业务改动同 PR 带测；写完必须跑通（§4.0）  
5. **不顺手重构** — 无 TODO 任务 ID 的 L0/L1 冻结区禁止改（§2.3）

### 0.3 与 `.aiexclude` / 附录 B 对齐

自动排除目录（`node_modules/`、`.venv/`、`*.log`、`data/`、lock 文件、大文档附件等）**不得主动 Read/cat**。需要 lock 文件中某一版本号时，用户必须显式点名路径。详规见 `AGENTS.md` 附录 B 与仓库根 `.aiexclude`。

## 一、反过度设计原则（Anti-Over-Engineering）

> 量化系统的最大陷阱不是功能不够，而是**把精力花在了不需要的抽象层和防御性代码上**，导致核心策略逻辑永远不稳定。

### 1.1 YAGNI — 只建当下需要的

```
❌ 错误思维：
  "将来可能需要支持多券商，所以现在就抽象一个 BrokerInterface..."
  "以后可能需要多租户，所以现在就设计租户隔离层..."

✅ 正确思维：
  现在只接 Futu，就直接写 FutuService，不抽象 Interface。
  当第二个券商真的来了，再重构。重构的成本远低于过早抽象的维护成本。
```

**判断标准**：如果某个抽象在 3 个月内不会有第二个实现，就不需要这个抽象。

> **V4.3**：§2.2 的 OCP/DIP 注册表示例 **仅适用于已有第二实现或架构 doc 已规定 Registry 的场景**（如 `DataSourceRegistry` · docs/14）。日常新功能默认 YAGNI，禁止为「将来可能」引入 Interface。

### 1.2 KISS — 简单是最高优先级

| 场景 | ❌ 过度设计 | ✅ 够用方案 |
|:---|:---|:---|
| 数据验证 | 为每个字段写专属 Validator 类 | 直接用 Pydantic Field 约束 |
| 错误处理 | 多层自定义 Exception 继承树 | 3-4 个语义清晰的领域异常 |
| 配置管理 | 多环境配置继承体系 | 单个 `.env` + Pydantic Settings |
| 前端状态 | 复杂的 Redux Saga / Observable | 简单 Zustand slice + useRef |
| 缓存策略 | 多级缓存自动失效框架 | Redis TTL + 手动刷新 |

### 1.3 不过度防御性编程

```python
# ❌ 过度防御：检查永远不会发生的情况，污染可读性
def get_price(symbol: str) -> float:
    if symbol is None:          # Pydantic 已保证不为 None
        raise ValueError(...)
    if len(symbol) == 0:        # Pydantic 已保证非空
        raise ValueError(...)
    if not isinstance(symbol, str):  # 类型注解已保证
        raise TypeError(...)
    ...

# ✅ 信任上游契约，只处理真实的业务异常
def get_price(symbol: str) -> float:
    try:
        return futu_client.get_quote(symbol).price
    except FutuConnectionError:
        raise MarketDataUnavailable(f"无法获取 {symbol} 行情")
```

## 二、Clean Architecture 与 SOLID 设计原则

> **核心目标**：通过分层隔离，确保业务逻辑不依赖具体的技术实现（框架、数据库、外部 API），使策略引擎与 OMS 保持纯粹可测试。

### 2.1 Clean Architecture 分层规范

系统严格按以下四层物理隔离，**依赖方向只能从外向内**（外层依赖内层，内层绝不依赖外层）：

```
┌─────────────────────────────────────────────────────┐
│  表现层 (Presentation / UI)                          │
│  React 组件、FastAPI Router、WebSocket Handler       │
│  职责：接收请求、参数校验、返回响应                    │
│  禁止：业务逻辑计算、直接访问数据库/外部 API           │
├─────────────────────────────────────────────────────┤
│  逻辑层 (Domain / Strategy Logic)                    │
│  ScreenerService、BacktestEngine、OMS StateMachine   │
│  职责：核心业务规则、策略信号生成、订单状态机          │
│  禁止：import FastAPI / React、直接发 HTTP 请求       │
├─────────────────────────────────────────────────────┤
│  数据接口层 (Data Access / Gateway)                  │
│  FutuService、YFinanceService、FinnhubService        │
│  职责：封装外部数据源、数据库读写、Redis 操作          │
│  禁止：包含业务决策逻辑                               │
├─────────────────────────────────────────────────────┤
│  基础设施层 (Infrastructure)                         │
│  PostgreSQL、Redis、Docker、Prometheus               │
│  职责：提供运行时基础设施                              │
└─────────────────────────────────────────────────────┘
```

**物理隔离红线**：

| 规则 | 说明 | 违规示例 |
|:---|:---|:---|
| Router 不含业务逻辑 | Router 只做参数解析 + 调用 Service + 格式化响应 | `@router.get("/quote")` 里写 PE 计算公式 |
| Service 不依赖框架 | Service 层不 import FastAPI / Request / Response | `from fastapi import Request` 出现在 `services/` |
| 第三方服务收口 | 所有外部 API 调用集中在 Gateway 层 | 前端或 Hermes Agent 直接 `fetch("https://api.finnhub.io")` |
| 依赖方向单向 | 内层不 import 外层 | `services/screener_service.py` 里 `from backend.routers.market import ...` |

### 2.2 SOLID 原则实践指南

#### S — 单一职责原则（Single Responsibility）

每个类/模块只对**一个干系人**负责，只有一个变更理由。

```python
# ❌ 违反 SRP：一个类同时管行情获取和订单执行
class TradingService:
    def get_quote(self, symbol): ...      # 行情职责
    def place_order(self, symbol, qty): ...  # 订单职责
    def calculate_pnl(self): ...           # 核算职责

# ✅ 符合 SRP：按职责拆分
class QuoteService:      # 只管行情
class OrderService:      # 只管订单
class AccountingService: # 只管核算
```

**判断标准**：如果描述这个类时需要用"和"连接两个职责，就需要拆分。

#### O — 开闭原则（Open/Closed）

对扩展开放，对修改封闭。新增功能通过新增代码实现，不修改已有代码。

```python
# ❌ 违反 OCP：每加一个数据源就要改 switch
def get_data(source: str):
    if source == "futu": return futu_api.get()
    elif source == "yfinance": return yf.get()
    elif source == "finnhub": return finnhub.get()  # 每次都改这里

# ✅ 符合 OCP：通过注册表扩展
DATA_SOURCES = {}

def register_source(name: str, fetcher: Callable):
    DATA_SOURCES[name] = fetcher

def get_data(source: str):
    return DATA_SOURCES[source]()
```

#### L — 里氏替换原则（Liskov Substitution）

子类必须能完全替换父类而不破坏程序行为。

```python
# ❌ 违反 LSP：子类偷偷改变了前置条件
class BaseScreener:
    def screen(self, market: str) -> list: ...

class HKScreener(BaseScreener):
    def screen(self, market: str) -> list:
        if market != "HK":
            raise ValueError("只支持港股")  # 子类收窄了入参范围
        ...

# ✅ 符合 LSP：子类不收窄前置条件
class HKScreener(BaseScreener):
    def screen(self, market: str) -> list:
        if market != "HK":
            return []  # 不支持的返回空列表，而非抛异常
        ...
```

#### I — 接口隔离原则（Interface Segregation）

客户端不应被迫依赖它不使用的方法。接口要小而专注。

```python
# ❌ 违反 ISP：胖接口强迫实现者实现不需要的方法
class DataGateway(ABC):
    @abstractmethod
    def get_quote(self): ...
    @abstractmethod
    def get_history(self): ...
    @abstractmethod
    def get_fund_flow(self): ...
    @abstractmethod
    def place_order(self): ...  # 只读数据源被迫实现下单

# ✅ 符合 ISP：按能力拆分接口
class QuoteGateway(ABC):
    @abstractmethod
    def get_quote(self): ...

class OrderGateway(ABC):
    @abstractmethod
    def place_order(self): ...

# FutuService 同时实现两个接口；YFinanceService 只实现 QuoteGateway
```

#### D — 依赖倒置原则（Dependency Inversion）

高层模块不依赖低层模块，二者都依赖抽象。

```python
# ❌ 违反 DIP：Service 直接依赖具体的数据源实现
class ScreenerService:
    def __init__(self):
        self.data_source = FutuService()  # 硬编码依赖

# ✅ 符合 DIP：通过依赖注入传入抽象
class ScreenerService:
    def __init__(self, quote_gateway: QuoteGateway):
        self.data_source = quote_gateway  # 依赖抽象接口
```

> **V4.3 提醒**：DIP/OCP 不得压过 §1 YAGNI。无第二实现时直接依赖具体类；Registry/Gateway 抽象以 `docs/14` / `docs/15` 已落地契约为准，禁止 AI 自行发明新 Interface 层。

### 2.3 模块冻结分级（Frozen Zone · V4.3 新增）

> **目标**：核心模块减少改动；改动必须绑定 `docs/TODO.md` 任务 ID，并经 §7.6 Confirm。

| 级别 | 规则 | 典型路径 |
|:---|:---|:---|
| 🔴 **L0 冻结** | 无对应 TODO 任务 **禁止改**；Plan 中必须列出；用户 Confirm 后才可 Edit | `backend/core/` · `backend/services/data_source_registry.py` · `backend/workers/collector_registry.py` · `frontend/src/hooks/use-ws-manager.ts` · `frontend/src/lib/api-client.ts` · `backend/engine/`（BT-01 落地后） |
| 🟠 **L1 谨慎** | 仅 bugfix / 接线；禁止无关重构、禁止重命名对外 export | `oms_service` · `alert_engine` · `bot_runtime` · `kline_warehouse` · 已发布 Pydantic Schema |
| 🟢 **L2 开放** | 按 SMART 任务正常改；仍遵守行数与 PR ≤400 | `frontend/src/features/*` · 新 router/service · 新测试 |

**绑定规则**：

- L0/L1 改动的 Plan 必须写明任务 ID（如 `[BT-01a]`、`[ALERT-03a]`）；无 ID → 先在 TODO 建任务或用户书面授权  
- 「顺手重构」「顺便拆文件」「统一命名」若触及 L0/L1 → **视为超范围**，须二次 Confirm  
- 新增文件落在 L0 目录 → 等同改动 L0

## 三、代码尺寸限制与拆分规范

> **核心原则**：代码是给人读的。超长文件和超长行是维护成本的罪魁祸首。

### 3.1 行宽限制

| 语言 | 最大行宽 | 强制工具 |
|:---|:---:|:---|
| Python | **120 字符** | ruff (line-length=120) |
| TypeScript / JavaScript | **120 字符** | ESLint (max-len) |
| CSS / SCSS | **120 字符** | stylelint |
| Markdown 文档 | **无硬限制**，但正文段落建议 ≤ 120 字符 | — |

**超长行处理优先级**：
1. 提取为变量/常量
2. 链式调用换行
3. 函数提取

```python
# ❌ 超过 120 字符
result = await screener_service.screen_stocks(market="HK", filters=[{"field": "PE_TTM", "min": 10, "max": 20}, {"field": "ROE", "min": 0.15}])

# ✅ 提取变量
pe_filter = {"field": "PE_TTM", "min": 10, "max": 20}
roe_filter = {"field": "ROE", "min": 0.15}
result = await screener_service.screen_stocks(
    market="HK",
    filters=[pe_filter, roe_filter],
)
```

### 3.2 文件行数限制（V4.3 统一表）

> **冲突消项**：历史文档曾出现 150 / 300 / 80-150-250 三套说法。以下为 **唯一行数 SSOT**（L0 `.cursor/rules` 的「单文件 ≤300」与本表 Feature 级对齐；原子组件更严）。

| 文件类型 | 最大行数 | 超限处理 |
|:---|:---:|:---|
| Python Service / Handler | **400 行** | 按职责拆分为多个 Handler |
| Python Router | **200 行** | 按业务域拆分为多个 Router |
| Python Model / Schema | **300 行** | 按领域拆分为多个 Schema 文件 |
| Python Test | **500 行** | 按测试类拆分 |
| React 原子组件 (.tsx) | **150 行** | 提取子组件 |
| React Feature 容器 / 页级 | **300 行** | 拆子组件或 hooks；禁止 >1000 行 |
| React Hook (.ts) | **200 行** | 提取为多个 Hook |
| TypeScript Utils / Store Slice | **200 行** | 按功能域拆分 |
| 配置文件 | **100 行** | 按环境拆分 |

### 3.3 拆分操作规范

当文件超过限制时，**必须在本 PR 中完成拆分**，不允许"下次再拆"。

**拆分决策树**：

```
文件超限？
  ├─ 是否有多个不相关的职责？
  │    └─ YES → 按职责拆分为独立文件（SRP）
  │
  ├─ 是否有重复的代码块？
  │    └─ YES → 提取为公共函数/组件
  │
  ├─ 是否有一个超长函数（> 50 行）？
  │    └─ YES → 提取子函数，每个子函数 < 30 行
  │
  └─ 以上都不是？
       └─ 按逻辑段落（如 CRUD 的每个操作）拆分为独立文件
```

**拆分命名规范**：

```
# 原始文件：backend/services/futu_service.py (800 行)
# 拆分后：
backend/services/futu/
├── __init__.py              # 导出 FutuService 兼容旧接口
├── service.py               # FutuService 主类（< 200 行）
├── quote_handler.py         # 行情处理逻辑
├── trade_handler.py         # 交易处理逻辑
├── screener_handler.py      # 选股处理逻辑
└── connection_manager.py    # 连接管理逻辑
```

## 四、测试同行工作流（Test-Alongside Workflow）

> **核心规则**：修改任何业务逻辑，必须在同一次提交中更新对应的测试。不允许"先改代码，测试以后再补"。

### 4.0 强制验证铁律（Zero-Tolerance）

> **写完测试不运行 = 没写测试。** 测试代码只有在运行通过后才算完成。

**每个测试文件编写完成后的强制操作流程**：

```
┌──────────────────────────────────────────────────────────┐
│  测试验证闭环 (Test Verification Loop)                    │
│                                                          │
│  1. 编写测试代码                                          │
│  2. 运行测试 → 记录结果                                   │
│  3. 测试失败？                                            │
│     ├─ YES → 修复测试或被测代码 → 回到步骤 2              │
│     └─ NO  → 进入步骤 4                                  │
│  4. 运行覆盖率检查                                        │
│     ├─ 覆盖率达标？                                       │
│     │   ├─ YES → 测试完成，可提交                         │
│     │   └─ NO  → 补充测试用例 → 回到步骤 2               │
│  5. 运行全量测试套件（确保无回归）                         │
│     ├─ 全部通过 → 可提交                                  │
│     └─ 有失败   → 修复回归 → 回到步骤 2                  │
└──────────────────────────────────────────────────────────┘
```

**禁止行为**：
- ❌ 写完测试不运行就提交
- ❌ 测试运行失败但不修复
- ❌ 覆盖率不达标但提交（除非有明确豁免理由并在 PR 中说明）
- ❌ 只运行单个测试文件，不运行全量套件验证回归

**验证命令速查**：

```bash
# 后端单文件测试验证
cd backend && python -m pytest tests/test_xxx.py -v --tb=short

# 后端全量测试 + 覆盖率
cd backend && python -m pytest --cov=backend --cov-report=term-missing --cov-fail-under=70

# 前端单文件测试验证
pnpm vitest run src/path/to/test.test.ts

# 前端全量测试 + 覆盖率
pnpm vitest run --coverage
```

### 4.1 标准开发循环

```
1. 明确需求  →  写测试（先写失败的用例）
2. 写最少的代码  →  让测试通过
3. 重构优化  →  确保测试仍然通过
4. 提交（代码 + 测试同时进 PR）
```

这不是严格的 TDD，而是**测试同行**——你可以先写实现，但在提交前必须补上测试。

### 4.2 测试分层与覆盖率目标

| 层级 | 测试类型 | 工具 | 目标覆盖率 | 优先覆盖内容 |
|:---|:---|:---|:---:|:---|
| **后端 Service** | 单元测试 | pytest + AsyncMock | **≥ 80%** | 所有业务逻辑分支 |
| **后端 Router** | 接口测试 | FastAPI TestClient | **≥ 70%** | 参数校验、错误码 |
| **后端 Worker** | 集成测试 | pytest-asyncio | **≥ 60%** | 重连逻辑、降级路径 |
| **Hermes Tool** | 单元测试 | pytest + AsyncMock | **≥ 90%** | 所有工具调用路径 |
| **前端 Hook** | 单元测试 | Vitest + MSW | **≥ 70%** | 数据转换、重连逻辑 |
| **前端 Store** | 单元测试 | Vitest | **≥ 80%** | 状态转换函数 |
| **前端 Utils** | 单元测试 | Vitest | **≥ 90%** | 所有纯函数 |

**不需要测试的内容**：
- UI 组件的样式（不测 class 名是否包含某个 Tailwind 类）
- 第三方库的行为（不测 AG Grid 能否渲染）
- 简单的 getter/setter
- 配置文件中的常量值

### 4.3 后端测试规范

```python
# tests/services/test_screener_service.py
# 命名规范：test_<功能>_<场景>_<预期结果>

class TestScreenerService:
    """每个 Service 一个 TestClass，隔离 fixtures"""

    @pytest.fixture
    def service(self, mock_redis, mock_futu_client):
        """只 Mock 外部依赖，不 Mock 被测对象本身"""
        return ScreenerService(redis=mock_redis, futu=mock_futu_client)
```

### 6.1 核心原则：永远不用 print()

### 6.2 日志级别语义

## 七、AI 辅助编码工作流（Vibe Coding SOP）

### 7.0 SMART 目标设定原则

> 在向 AI 下达任何开发任务前，必须用 **SMART** 框架定义任务目标。模糊的指令只会产出模糊的代码。

每个任务指令必须满足以下五个维度：

| 维度 | 含义 | 检查问题 | ❌ 反面示例 | ✅ 正面示例 |
|:---|:---|:---|:---|:---|
| **S**pecific | 具体的 | AI 能否仅凭指令定位到确切的文件和函数？ | "优化一下选股性能" | "将 `screener_handler.py` 的 `screen_stocks` 方法中 `for f in filters` 循环改为字典推导式" |
| **M**easurable | 可衡量的 | 完成标准是否可以用测试或指标验证？ | "让代码更好" | "补充测试用例使 `screener_handler.py` 覆盖率从 45% 提升至 ≥ 80%" |
| **A**chievable | 可实现的 | 以当前代码库状态，AI 是否有能力完成？ | "实现一个完整的量化交易平台" | "实现 `POST /api/v1/screener/save-template` 接口，含参数校验和重复名称冲突处理" |
| **R**elevant | 相关联的 | 这个任务是否服务于当前迭代目标？ | 在做行情模块时要求"顺便重构日志系统" | "为选股模板保存功能增加持久化层，服务于本迭代选股模板管理需求" |
| **T**ime-bound | 有时限的 | 任务是否有明确的完成边界（文件数/PR 大小）？ | "把所有测试补完" | "为本 PR 涉及的 3 个文件补充测试，单 PR 不超过 400 行" |

**SMART 指令模板**：

```
[目标 S] 在 <文件路径> 的 <函数/类> 中实现 <具体功能>
[衡量 M] 完成标准：<可验证的测试用例或指标>
[可行 A] 约束条件：<依赖的现有模块、禁止的改动范围>
[关联 R] 业务背景：<本任务服务于哪个迭代目标>
[时限 T] 边界限制：<涉及文件数 / PR 行数上限>
```

**实战示例**：

```
[S] 在 backend/routers/screener.py 新增 POST /api/v1/screener/save-template 接口，
    接收 ScreenerTemplateCreate Schema（name, dsl, markets），调用 ScreenerService.save_template 持久化
[M] 完成标准：tests/routers/test_screener_router.py 中新增 2 个测试用例
    （正常保存 200 + 重复名称 409），覆盖率 ≥ 80%
[A] 约束：复用现有的 get_db 依赖和 UnifiedResponse 封装，不引入新依赖
[R] 服务于本迭代的"选股模板管理"需求（TODO: SCREENER-03）
[T] 涉及 3 个文件（router + service + test），PR ≤ 200 行
```

### 7.1 最小上下文包（MCP · V4.3）

> **取代**旧版「必须同时加载 AI_INSTRUCTIONS + 全文 docs/02」——那是 Token 浪费源。  
> 工作流固定：**Grep/Glob 定位 → Read 单文件 → 再 Write**；禁止先读整个 `backend/` / `frontend/`。

| 任务类型 | 必加载（建议 ≤5 文件） | 禁止加载 |
|:---|:---|:---|
| 修 Bug | 目标文件 + 对应 test + 出错栈相关 1 个调用方 | 全目录、`*.log` 全文、`uv.lock` / `pnpm-lock.yaml` |
| 新 Router | 同类 router 1 个 + 对应 service/schema + `docs/10` 相关片段 | §5 技术栈全文、无关 docs |
| 新 Feature UI | `features/<domain>/` 参考组件 1 个 + 相关 store slice + 现有 test 风格 | `node_modules/`、全量 `docs/` |
| 架构接线（BT/ALERT/DQ…） | 对应 `docs/15~19` **单章** + TODO 任务条目 + 目标代码路径 | 同时灌入全部架构 doc |
| 纯文档 | 目标 md + 交叉引用的 1 个相关 doc | 源码树 |

**默认会话加载**：

```
1. L0 已由 Cursor rules 注入（无需重复粘贴全文）
2. 本文档按需节：§0 / §2.3 / §7（本 SOP）— 勿整本 900 行塞入
3. L3：本任务架构章或 TODO 条目（单点）
4. 目标源文件 + 镜像测试文件（精确定位后 Read）
```

不确定路径时：**先 Grep 文件名/符号，再 Read**；禁止「浏览整个模块再猜」。

### 7.4 禁止事项速查（10 条红线）

| # | ❌ 禁止 | ✅ 替代 |
|:---|:---|:---|
| 1 | `print()` 调试 | `logger.debug()` |
| 2 | 裸 `except:` | `except SpecificError as e:` |
| 3 | `useState` 存 Tick | `useRef` + Float64Array |
| 4 | 前端直连外部 API | 通过后端 Gateway 代理 |
| 5 | Router 里写业务逻辑 | 逻辑下沉至 Service 层 |
| 6 | async 函数里同步阻塞 | `asyncio.to_thread()` |
| 7 | 硬编码配置值 | `os.getenv()` + Pydantic Settings |
| 8 | 提交无测试的业务代码 | 测试同行，同 PR 提交 |
| 9 | 跳过 ReAct 的 Verify 步骤 | Plan → Tool → **Verify** → Output |
| 10 | 生成 AI 数字不引用 Tool | 数据来源 100% 来自 Tool 返回 |

### 7.5 Verify-Before-Write 反幻觉协议（V4.3 新增）

> 与 Hermes ReAct 的 Verify 对齐。写代码前必须完成，禁止 silent guess。

```
1. Plan   — 列出将改动的文件（≤5）+ 明确「不动清单」+ 任务 ID
2. Locate — Grep/Glob 证明函数、路由、类型、env 名已存在
3. Patch  — 最小 diff；禁止顺手重构 L0/L1
4. Prove  — 给出 pytest/vitest 命令 + 预期结果；跑通才算完成
```

**Uncertainty Gate**：路径、API、字段、配置不确定 → **提问用户**，禁止用预训练知识填假路径/假端点/假 env。  
**禁止臆造**：

- 不存在的 `GET/POST /api/v1/...`（必须对现有 `routers/` 或 `docs/10`）  
- 不存在的 `os.getenv("...")` 键名（必须 Grep 现有用法）  
- 「估计」的金融数字或行情（数字类必须来自 Tool/代码/测试 fixture）

### 7.6 Plan-Confirm-Execute（PCE · V4.3 新增）

> **目标**：改动需经过确认；核心少动。

```
1. Plan（只读）
   - 改动文件列表（≤5）与预估行数
   - 不动清单（含触及的 L0/L1）
   - 依赖任务 ID（TODO）
   - 回归测试命令
2. Confirm
   - 用户回复「确认」/「缩小范围」/「取消」
   - 触及 🔴 L0 → 必须显式 Confirm，禁止默认继续
3. Execute
   - 仅修改 Plan 内文件；超范围 → 二次 Confirm
4. 默认禁止（除非用户明确要求）
   - git commit / push
   - 改 docker-compose、CI workflow、生产 env 示例中的密钥占位以外的敏感值
```

**例外（可跳过 Confirm 直接 Execute）**：纯文档错字、单测内断言文案、用户已在同一条消息写明「直接改某某文件」且不触及 L0。

### 7.7 多模型差异速查（V4.3 新增）

| 能力画像 | 推荐策略 |
|:---|:---|
| 有 Grep/Read/Shell（Cursor Agent 等） | 完整走 MCP + §7.5 + §7.6 |
| 强推理、大上下文 | Plan 可含 3~5 文件；仍须 Confirm 后批量改 |
| 上下文 < 32k / 本地小模型 | 单文件任务；只加载 MCP 3 文件包；禁止跨模块重构 |
| 仅补全（Copilot inline） | 禁止跨文件 refactor；单函数建议 ≤30 行；大改走 Agent + PCE |
| 无工具、只出文本 | 只输出 unified diff / 补丁，由人手工应用；标注「未 Verify」 |

语言约定：代码标识符与 API field 保持仓库现有英文；PR/任务描述可用中文；契约字段以 `docs/10` / 现有 Schema 为准。

## 八、代码审查规范（Code Review）

### 8.0 Git 分支策略与提交规范

> **核心原则**：`main` 分支永远保持可部署状态，所有开发在 `develop` 分支进行，通过 PR 合并实现质量门禁。

**双分支模型**：

```
main       ← 受保护的生产分支，仅接受来自 develop 的 PR
  └─ develop  ← 日常开发分支，所有功能在此合并
       ├─ feature/xxx  ← 可选的功能分支，完成后 PR 回 develop
       └─ fix/xxx      ← 可选的修复分支
```

**分支保护规则**（`main` 分支）：
- ❌ 禁止直接 push（必须通过 PR）
- ❌ 禁止 force push
- ❌ 禁止删除
- ✅ CI 必须通过才能合并（lint + test + security）
- ✅ 管理员也不绕过

**CI/CD 触发规则**：

| 事件 | CI 检查 | 部署 |
|:---|:---:|:---:|
| push `develop` | ✅ lint + test + security | ❌ 不部署 |
| PR → `main` | ✅ 质量门禁 | ❌ 不部署 |
| push `main` (merge) | ✅ | ✅ 四节点部署（US-MASTER + YF×2） |
| `workflow_dispatch` | ✅ | ✅ 可选目标节点 |

**提交规范**：

```bash
# 所有提交都提交到 develop 分支
git checkout develop
git add . && git commit -m "feat(screener): 新增选股模板保存功能"
git push origin develop

# 准备发布: 从 develop 发起 PR 到 main
# GitHub 网页上创建 PR → CI 通过 → 合并 → 自动部署
```

**Commit Message 格式**：

```
<type>(<scope>): <description>

type: feat | fix | refactor | perf | test | docs | chore
scope: screener | futu | market | frontend | cluster | ci ...
description: 简短描述，祈使句，首字母小写，不加句号

示例:
  feat(screener): 新增选股模板保存与复用功能
  fix(futu): 修复停牌标的行情返回 None 的 KeyError
  test(cluster): 补充 ClusterManager payload 格式测试
  docs(deploy): 更新本地开发指南与分支策略
```

### 8.1 PR 大小控制

- **单个 PR 不超过 400 行改动**（不含测试文件、不含生成代码）
- 超过 400 行的需求必须拆分为多个 PR
- 每个 PR 必须有清晰的描述：解决什么问题、涉及哪些模块、测试如何验证

### 8.2 PR 标题规范

```yaml
    - pnpm lint           # ESLint 检查

    - pnpm test           # Vitest 单元测试
    - pnpm build          # 构建产物验证（无编译错误）
```

## 九、文档维护规范

### 9.1 架构文档更新触发条件

当以下变更发生时，**必须同步更新对应文档**：

| 变更类型 | 需更新的文档 |
|:---|:---|
| 新增/删除 API 端点 | `docs/10` 接口契约（勿再写已 Deprecated 的 `docs/backend.md`） |
| 新增/修改 Hermes Tool | `docs/subsystems/agent/architecture.md` |
| 调整目录结构 | `AGENTS.md` 目录地图（L1）；L4 `AI_INSTRUCTIONS.md` 可选同步 |
| 新增第三方依赖 / 换栈 | **先 ADR** → 更新 L0/L1 → 本文档 §5 索引一行 |
| 修改部署配置 | `docs/06` · `AGENTS.md` §9 |
| 发现性能问题并优化 | `docs/09` 基准数据 |
| 产品模块 / 用户旅程 | `docs/01`；任务状态以 `docs/TODO.md` 为 SSOT |
| 子系统深度设计 | `docs/15~19` 或 `docs/subsystems/*`（单章维护） |

### 9.2 子系统架构文档位置

```
docs/subsystems/
├── frontend/architecture.md     前端架构与组件关系图
├── backend/architecture.md      后端节点架构与数据流图
├── agent/architecture.md        Hermes Agent 推理架构
├── client/architecture.md       Flutter 客户端
└── deployment/architecture.md   部署拓扑与网络架构

docs/15~19/                      回测同构 / 策略实验室 / 纸面 / 推送 / 数据快照
```
