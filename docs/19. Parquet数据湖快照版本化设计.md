# 19. Parquet 数据湖快照版本化设计（DQ-03 · Data Lake Snapshots）

> **文档定位**：K 线 Parquet 数据湖快照版本化、回测引用机制、旧快照保留与冷归档的架构设计（设计先行，未动手编码）。对应 `docs/TODO.md DQ-03`，并作为 **BT-02 回测可复现性** 的数据地基。
> **版本**：V1.1 | **日期**：2026-07-13 | **状态**：已落地（DQ-03a~e）
> **关联**：`docs/15 §RunManifest.data_snapshot_id`、`docs/TODO.md BT-02`、`backend/services/kline_warehouse.py`、`backend/services/kline_cache.py`（L3 R2 预留）、`docs/06 §2.4` R2 存储分类、DQ-01/02 已完成模块
> **定位一句话**：回测可复现性的另一半是「代码 hash」——另一半是「数据 hash」。没有不可变的数据快照，同一份策略代码在不同日期跑回测，结果必然漂移（复权修正、新 bar 追加、退市标的补全）。

## 一、现状盘点与问题

### 1.1 当前数据湖形态

```
data/kline_warehouse/              ← 唯一「真相」层，可变（in-place 增量 merge）
├── K_DAY/
│   ├── US_AAPL.parquet
│   └── HK_00700.parquet
└── K_60M/
    └── ...

遗留路径（待废弃）：
data/parquet_db/{ticker}_{interval}.parquet   ← routers/backtest.py 仍在用
```

| 能力 | 现状 | 对回测可复现性的影响 |
|:---|:---|:---|
| K 线存储 | `KlineWarehouse.update_ticker()` 增量 merge，`keep='last'` 覆盖同日期 bar | **同一 ticker 文件随时间变化**——今天跑和昨天跑结果不同 |
| 读取入口 | `get_history(ticker, ktype, num)` 永远读 live 层 | 回测无「锁定某一版数据」的能力 |
| 每日同步 | `daemon_sync_task()` 03:00 + Redis 锁，同步后**不生成快照** | 同步与版本化脱节 |
| L3 冷存储 | `kline_cache.py` 注释预留 R2，**零实现** | 无长期归档 |
| DQ-01 幸存者 | `SurvivorshipBiasTracker` 内存/CSV，**未与 K 线版本绑定** | 回测 universe 状态不可复现 |
| DQ-02 PIT 财务 | `PointInTimeStore` 内存单例，**未持久化** | 含财务因子的回测不可复现 |
| BT-02 | `RunManifest.data_snapshot_id` 字段已定义，**恒为 None** | 插座已留，无数据源 |
| 回测 API | `RunSandboxPayload` 无 snapshot 参数；`_fetch_backtest_data` 读 live 层 | 用户无法指定数据版本 |

### 1.2 目标（Definition of Done · DQ-03）

- **不可变快照**：每个交易日同步完成后，从 live 层生成一份**只读**快照包，含 manifest（文件清单 + 校验和 + 元数据）。
- **稳定 snapshot_id**：`snap_YYYYMMDD`（UTC+8 交易日口径），全局唯一，写入 PG `data_snapshots` 表。
- **回测引用**：回测请求可指定 `data_snapshot_id`；默认使用 `latest_published`（最近已发布快照，**非** live 层）；`RunManifest` / BT-02 报告持久化绑定 snapshot_id + manifest_hash。
- **保留策略**：近 90 天每日快照全保留；90 天~12 个月仅保留每月锚点快照；>12 个月压缩归档 R2、删除本地副本。
- **DQ 捆绑**：快照 manifest 记录同步时刻的 universe 摘要 hash（DQ-01）与 PIT store 导出 hash（DQ-02，V1 可选 sidecar）。

### 1.3 非目标（YAGNI）

- ❌ 不做 Tick 级快照（数据湖只有 K 线粒度）
- ❌ 不做分钟级快照（V1 仅 `K_DAY` + `K_60M` 入包；其他周期随 live 层演进）
- ❌ 不替换 live 层（实时图表、纸面结算、采集仍读 live；快照专供回测/审计）
- ❌ 不做跨节点分布式快照一致性（单主节点加州 VPS，与北京辅助节点无关）
- ❌ 不在 V1 实现快照 diff/增量存储（全量 copy + hardlink 优化即可；个人系统磁盘 <500GB 可承受）

## 二、总体架构

```
  03:00  KlineWarehouse.daemon_sync_task()
           │ 增量更新 live 层 (data/kline_warehouse/)
           ▼
  03:30  SnapshotPublisher.create_daily_snapshot(as_of_date)
           │ 1. 校验 live 层 SVC-04 质量门禁（脏数据率阈值）
           │ 2. hardlink/copy → data/snapshots/snap_YYYYMMDD/
           │ 3. 导出 universe sidecar + 可选 PIT sidecar
           │ 4. 生成 manifest.json + 计算 manifest_hash
           │ 5. PG data_snapshots 写入 (status=published)
           │ 6. Redis quant:datalake:latest_snapshot_id 更新
           ▼
  04:00  SnapshotRetentionJob（每周日 + 每月 1 日额外跑）
           │ Tier-1 保留 / Tier-2 月度锚点 / Tier-3 R2 归档+删除
           ▼
  回测读取链：
  BacktestDriver / _fetch_backtest_data
           │ data_snapshot_id 指定
           ▼
  SnapshotReader.get_history(snapshot_id, ticker, ktype, num)
           │ 读 snapshots/snap_YYYYMMDD/kline/...（只读）
           ▼
  RunManifest { data_snapshot_id, manifest_hash, code_hash, ... }
           ▼
  BT-02 backtest_reports 持久化
```

**核心原则**：

1. **Live 可变、Snapshot 不可变**：同步只写 live；快照是时间点冻结副本，生成后禁止修改。
2. **回测默认读快照、不读 live**：避免「无意用了最新数据」；显式 `snapshot_id=live` 仅供调试（开发环境开关）。
3. **manifest_hash 是数据指纹**：BT-02 可复现性对比 manifest_hash 即可判定数据输入是否一致，无需逐文件 diff。

### 3.2 snapshot_id 命名

| 类型 | 格式 | 示例 | 说明 |
|:---|:---|:---|:---|
| 日快照 | `snap_YYYYMMDD` | `snap_20260713` | 每个交易日同步后生成；`YYYYMMDD` = 亚洲/上海时区日历日（与现有 daemon 03:00 HKT 对齐） |

## 四、PostgreSQL 数据模型

```sql
data_snapshots
├── snapshot_id     VARCHAR(32) PK     -- snap_YYYYMMDD
├── as_of_date      DATE NOT NULL
├── status          VARCHAR(12)        -- building | published | failed | archived
├── manifest_hash   CHAR(64) NOT NULL
├── manifest_json   JSONB NOT NULL     -- 完整 manifest（查询用，文件为 SSOT）
├── ticker_count    INT
├── total_bytes     BIGINT
├── is_monthly_anchor BOOLEAN DEFAULT FALSE
├── storage_tier    VARCHAR(8)         -- local | r2 | deleted

├── published_at    TIMESTAMPTZ NULL
└── archived_at     TIMESTAMPTZ NULL

INDEX idx_data_snapshots_as_of (as_of_date DESC)
```

## 五、快照生成流程（SnapshotPublisher）

### 5.1 触发时机

挂 `kline_warehouse.daemon_sync_task()` **完成之后**（同进程顺序调用，不另起 03:30 定时——避免 sync 延迟导致空快照）：

```
daemon_sync_task 每日分支:
  if lock acquired and sync completed:
      await snapshot_publisher.create_daily_snapshot(as_of_date=yesterday_calendar)
```

**as_of_date 语义**：快照冻结的是「同步完成时刻」live 层内容；`as_of_date` 标记为**同步目标交易日**（通常 = 昨日，因 03:00 跑的是前一日收盘后数据）。manifest 的 `time_max` per file 是真实校验依据。

### 5.2 生成步骤

```
1. 抢锁 quant:lock:snapshot_create:{YYYYMMDD}（NX, ex=2h）
2. status=building 写入 PG
3. 枚举 live 层 K_DAY + K_60M 全部 parquet 文件
4. 对每个文件：
     - hardlink 到 snapshots/snap_YYYYMMDD/kline/{ktype}/{file}（同盘 zero-copy）
     - 失败则 asyncio.to_thread(shutil.copy2)
     - 计算 sha256 + rows + time range
5. 导出 sidecars/universe.json（SurvivorshipBiasTracker.export_snapshot()——DQ-03a 新增导出 API）
6. SVC-04 质量门禁：aggregate dirty_rate <= 2% 才允许 publish；否则 status=failed + 告警（AlertDispatcher P2）
7. 写 manifest.json + 计算 manifest_hash
8. PG status=published, published_at=now
9. Redis SET quant:datalake:latest_snapshot_id {snapshot_id}
10. 结构化日志 + Prometheus 计数器 datalake_snapshot_created_total
```

### 5.3 失败与补生成

- 同步成功但快照失败 → 当日无 published 快照；回测 `latest_published` 回退到上一个 published（**不**读 live）
- 补生成 API：`POST /api/v1/datalake/snapshots/rebuild?as_of_date=YYYYMMDD`（管理员，手动触发，复用同流程）
- 重复生成同一 `snapshot_id`：若已 published 则幂等返回；若 failed 则允许覆盖重建

## 七、旧快照保留策略

### 7.1 三级保留 Tier

| Tier | 年龄 | 本地保留 | R2 归档 | 删除 |
|:---|:---|:---|:---|:---|
| **T1 热** | 0–90 天 | **全部日快照** | 否 | 否 |
| **T2 温** | 91–365 天 | **仅月锚点**（每月 1 个） | 否（可选异步预上传） | 非锚点日快照删除 |
| **T3 冷** | >365 天 | 否 | **tar.zst 整包** | 本地删除；R2 保留 5 年 |

### 7.2 月锚点选取规则

每月 1 日 04:00 保留任务：

```
对 month M：
  candidates = 所有 snap_YYYYMMDD where YYYYMM=M and status=published
  anchor = max(candidates by as_of_date)   -- 该月最后一个 published 日快照
  标记 anchor.is_monthly_anchor = true
  删除同月其他日快照（本地文件 + PG storage_tier=deleted，保留 PG 元数据行）
```

### 7.3 R2 归档格式

对齐 `docs/06` R2 分类，新增前缀：

```
quant-datalake/
  └── snapshots/
        └── snap_202503.tar.zst    # zstd 压缩整包（manifest + kline + sidecars）
```

环境变量：`R2_DATALAKE_BUCKET` / `R2_ACCESS_KEY` / `R2_SECRET_KEY`（复用 OPS 备份同一套凭证）。

归档流程：

```
1. tar.zst 打包 snapshots/snap_YYYYMMDD/
2. 上传 R2，校验 etag
3. PG storage_tier=r2, r2_key=..., archived_at=now
4. 删除本地 snapshots/snap_YYYYMMDD/ 目录
5. SnapshotReader 遇 storage_tier=r2 → 按需下载到 /tmp/cache 或拒绝读取（V1：**仅支持 local tier 读取**，>365 天快照提示「已归档，请 restore」）
```

V1 读取只支持 `storage_tier=local`；R2 快照需 `POST /datalake/snapshots/{id}/restore` 拉回本地（管理员操作，YAGNI 自动 hydrate）。

### 7.4 磁盘预算估算

| 假设 | 数值 |
|:---|:---|
| 监控标的 | ~850 tickers × 2 ktypes |
| 单文件均值 | ~200 KB |
| 单日快照（hardlink 同 inode 时**增量磁盘≈manifest+sidecar ~5MB**；首次 copy 全量 ~340MB） |
| 90 日全保留 | ~30 GB（按 hardlink 优化后 ~1–5 GB 量级） |
| 9 个月度锚点 | ~3 GB |

个人系统 VPS 磁盘可承受；hardlink 是 V1 必选优化。

## 十、风险与开放问题

| # | 风险 | 处置 |
|:---|:---|:---|
| 1 | hardlink 跨文件系统失败（Docker volume 挂载） | 自动降级 copy2；manifest 记录 copy 模式 |
| 2 | 同步未完成就生成快照 | 顺序调用：仅 sync 成功分支触发 publisher |
| 3 | 复权修正导致同 snapshot_id 文件被误改 | 快照目录 chmod 只读；live 与 snapshot 物理隔离 |
| 4 | latest_published 过旧（连续 failed） | 超过 3 天无 published → P1 告警 + 回测 API 返回 503 DATA_SNAPSHOT_STALE |
| 5 | R2 restore 未实现导致 >1 年快照不可用 | V1 接受；UI 标注 archived；BT-02 报告仍保留 manifest 供审计 |
| 6 | universe sidecar 与 K 线不同步 | 同一 publish 事务内顺序导出；manifest 分别 hash |
