# 17. 纸面组合系统架构（PT-01/02 · Paper Portfolio Tracking）

> **文档定位**：纸面组合追踪（虚拟撮合引擎 + 净值结算 + 绩效归档 + 回测对比）的架构设计文档（设计先行，未动手编码）。对标 QuantConnect Paper Trading。
> **版本**：V1.0 | **日期**：2026-07-13 | **状态**：待评审
> **关联**：`docs/15 回测实盘同构引擎设计`（SimBroker/LiveDriver/paper 模式为本系统执行底座）、`docs/16 策略实验室完整架构`（版本 hash 绑定）、`docs/TODO.md PT-01/02`、`docs/11 数据模型`
> **定位一句话**：纸面组合是策略上实盘前的"长期考察期档案"——过拟合的最后一道防线，核心交付物是**可信的日终净值序列**与**纸面 vs 回测的偏离度监控**。

## 一、现状盘点与目标（2026-07-13 代码摸底）

### 1.1 现状：系统内不存在任何"虚拟账本"

| 能力 | 现状 | 结论 |
|:---|:---|:---|
| 虚拟撮合 | 唯一完整实现在回测引擎（`event_engine.py` 的 cash/position/限价撮合/滑点），与 OMS/Bot 不通；Algo 引擎 SANDBOX 下模拟成交但无组合级记账 | ❌ 无常驻虚拟撮合 |
| 模拟单语义 | `trading_mode=SANDBOX` 时 `trade.py` **仍然真实调用** `futu_service.place_order()`（Futu OpenD 模拟盘环境 `TrdEnv.SIMULATE` 硬编码），`is_simulated` 只是落库标签 | ⚠️ "模拟"实际托管给 Futu 模拟盘，不是系统内账本 |
| 净值持久化 | `nav_snapshots` 表 + `_nav_snapshot_daemon`（5min）已有，但绑定 **Futu 券商账户（HK/US）** 维度，无 portfolio_id/bot_id | ❌ 不可直接复用，可借鉴模式 |
| 持仓真相源 | `quant:oms:positions:{market}` 由 `sync_positions_from_futu` 同步，来源是券商账户 | ❌ 纸面持仓必须自维护，严禁混用 |
| 绩效计算 | Sharpe/MaxDD 内联在 `event_engine.py` / `runners.py` / `risk_engine.py` 三处，各写各的 | ⚠️ 无共享绩效库，PT 顺手抽取 |
| 定时任务 | 无 APScheduler；范式已定型：`while True` + 时分判断 + `quant:lock:*` NX 锁（如 kline_warehouse 每日 3:00） | ✅ 沿用范式 |
| 交易日历 | `MarketSession.is_trading_hours`（工作日+时段），**无节假日表**；`detect_suspension` 可从 K 线推断停牌 | ⚠️ 日结不能依赖日历，需数据驱动兜底 |
| 前端曲线叠加 | `backtest-report.tsx` 的 `SandboxChart` 已支持 equity + benchmark 双线（Lightweight-Charts） | ✅ PT-02 对齐 `equity_curve` 契约即可复用 |

### 1.2 与 docs/15 的关系（执行底座 vs 归档系统）

`docs/15 §4.3` 已确定：**paper 模式 = LiveDriver + SimBroker**（真实行情驱动、模拟撮合记账，零新增引擎）。本文档不重复设计撮合执行——PT 系统的本体是执行底座之上的**账本、结算、归档、对比**四层：

```
docs/15 负责（BT-01b/d/e 交付）        docs/17 负责（PT-01/02 交付）
┌─────────────────────────┐          ┌──────────────────────────────┐
│ LiveDriver (真实行情流)   │  fills   │ PaperLedger 账本（PG SSOT）    │
│ + SimBroker (虚拟撮合)    │ ───────► │ EOD 结算 daemon（净值/持仓快照）│
│ + ExecutionGateway       │          │ 绩效归档（日频指标序列）         │
│   (paper 分支路由)        │          │ 回测对比（跟踪误差 + 偏离告警）  │
└─────────────────────────┘          └──────────────────────────────┘
```

### 1.3 目标（Definition of Done）

- **PT-01**：策略可部署为 paper 模式常驻组合（独立虚拟资金），成交流水与持仓落 PG；每个交易日收盘后自动 mark-to-market 结算出当日净值；重启不丢账（账本可从流水重放重建）。
- **PT-02**：任一纸面组合可与其绑定的回测基准曲线叠加对比；输出跟踪误差与偏离归因摘要（择时偏离 vs 成交偏离）；偏离超阈值经告警中心（ALERT-01 引擎，已上线）推送。
- **纪律性目标**：为"策略上实盘"提供硬门槛数据——纸面运行天数、纸面 Sharpe、纸面 vs 回测偏离度，三项在部署实盘确认框中强制展示（不做强制拦截，个人系统保留 override）。

### 1.4 非目标（防过度设计）

- ❌ 不做多币种/汇率折算（组合按标的所属市场单币种记账，HK=HKD、US=USD，跨市场组合 V1 不支持）
- ❌ 不做保证金/融资/做空账本（现有策略均为现货多头语义；`side=SELL` 仅允许平仓）
- ❌ 不做公司行动全套处理（拆股/分红按 §5.4 简化口径）
- ❌ 不做盘中逐 tick 净值持久化（盘中快照只进 Redis 环形，PG 只落日终）
- ❌ 不替代 Futu 模拟盘链路（`trade.py` 手动下单仍走 Futu SIMULATE，两套并存：手动模拟=Futu 托管，策略纸面=自研账本；收敛留给 BT-01f 之后评估）

## 二、总体架构

```
                     策略（docs/15 Strategy 契约，mode="paper"）
                                     │ OrderIntent
                                     ▼
                  ExecutionGateway ──paper分支──► SimBroker（虚拟撮合）
                                                     │ Fill 事件
                     ┌───────────────────────────────┤
                     ▼                               ▼
        Redis（热状态，可重建）              PaperLedgerService（PG SSOT）
        quant:paper:{pid}:positions         paper_portfolios   组合主档
        quant:paper:{pid}:nav_intraday      paper_fills        成交流水（只增）
        （List 环形，288点@5min）            paper_positions    持仓现状（可重放重建）
                     │                      paper_nav_daily    日终净值（不可变）
                     │                               ▲
                     ▼                               │ EOD 写入
        ┌────────────────────────────────────────────┴───────────┐
        │  PaperSettlementDaemon（每市场收盘后触发，Redis NX 锁）    │
        │  收盘价: kline_warehouse K_DAY.close（停牌→前收兜底+标记） │
        │  产出: 当日 NAV / 持仓快照 / 日收益 → paper_nav_daily     │
        └────────────────────────────────────────────────────────┘
                     │ 日终 NAV 序列
                     ▼
        PerformanceService（新增共享绩效库 backend/services/performance.py）
        sharpe / max_drawdown / 年化 / 跟踪误差 / 偏离归因
                     │
                     ▼
        对比层（PT-02）：paper 曲线 ⊕ 回测基准曲线（经 code_hash 绑定）
        → GET /paper/{pid}/compare → 前端叠加图 + 偏离告警（AlertEngine 复用）
```

**三条核心原则**：

1. **PG 流水是唯一事实**：`paper_fills` 只增不改；`paper_positions` 与 Redis 热状态都是流水的投影，任何不一致以"从流水重放"为准（重建函数是一等公民，也是重启恢复路径）。
2. **纸面与实盘物理隔离**：独立表、独立 Redis 键空间（`quant:paper:*`），与 `quant:oms:positions:{market}`（券商真相）零交集；纸面组合永远不出现在 Risk 实盘面板的资产合计里。
3. **日终净值是产品级承诺**：盘中快照尽力而为（行情断流可缺），日终结算必须完备（缺收盘价有兜底链，结算失败有补结算机制）——PT-02 的所有对比与告警都建立在日终序列上。

## 三、领域模型（PostgreSQL，Alembic 迁移）

```
paper_portfolios                     # 组合主档
├── id             UUID PK
├── name           VARCHAR(64)
├── strategy_name  VARCHAR(64)          # 关联策略
├── strategy_version_id UUID NULL       # docs/16 strategy_versions.id（版本绑定）
├── code_hash      CHAR(64)             # 冗余存储，版本表未落地前也可工作
├── params         JSONB                # 部署参数
├── market         VARCHAR(4)           # HK | US（单市场，见非目标）
├── initial_capital NUMERIC(18,2)
├── benchmark_backtest_ref VARCHAR(64) NULL  # 绑定的回测基准（BT-02 落地前存 run 摘要 JSON 的引用键，落地后为 backtest_reports.run_id）
├── bot_id         VARCHAR(64) NULL     # 关联的常驻 Bot（bot_runtime）
├── status         VARCHAR(12)          # running | paused | closed
├── created_at / closed_at

paper_fills                          # 成交流水（只增，账本 SSOT）
├── id             UUID PK
├── portfolio_id   UUID FK (indexed)
├── fill_seq       BIGINT               # 组合内单调序号（重放排序依据，不依赖时间戳精度）
├── dt             TIMESTAMPTZ          # 成交时间（UTC）
├── symbol / side / qty / price
├── commission     NUMERIC(12,4)        # 佣金（与 SimBroker 摩擦模型同源参数）
├── slippage       NUMERIC(12,4)
├── intent_tag     VARCHAR(64) NULL     # docs/15 OrderIntent.tag 贯穿
└── UNIQUE(portfolio_id, fill_seq)

paper_positions                      # 持仓现状（投影，可重建）
├── portfolio_id + symbol  复合 PK
├── qty            INT
├── avg_cost       NUMERIC(18,4)
├── last_fill_seq  BIGINT               # 投影水位（重放增量恢复用）
└── updated_at

paper_nav_daily                      # 日终净值（不可变，PT-02 数据基座）
├── portfolio_id + trade_date  复合 PK
├── nav            NUMERIC(18,2)        # 现金 + Σ(持仓×收盘价)
├── cash           NUMERIC(18,2)
├── market_value   NUMERIC(18,2)
├── daily_return   NUMERIC(10,6)
├── stale_symbols  JSONB NULL           # 当日用兜底价的标的清单（停牌/缺数据）
└── settled_at     TIMESTAMPTZ
```

**设计要点**：

- `fill_seq` 而非时间戳作为重放序——避免同秒多笔成交的排序不确定性；由 LedgerService 在写入时分配（PG sequence per portfolio 或乐观锁自增）。
- 不建 `paper_orders` 表：挂单是 SimBroker 的内存/Redis 热状态（重启由 Driver 恢复），账本只关心成交事实。挂单历史审计价值低（YAGNI）。
- `benchmark_backtest_ref` 的双轨：BT-02（回测报告持久化）未落地前，创建组合时把基准回测的 `equity_curve` 快照存入 Redis `quant:paper:{pid}:benchmark`（JSON，永久 TTL）+ R2 备份；BT-02 落地后切换为外键引用。**PT-02 不阻塞于 BT-02**。

## 四、虚拟撮合（复用 SimBroker，仅定义 paper 特有行为）

撮合内核复用 `docs/15 drivers/sim_broker.py`（限价/滑点/佣金逻辑与回测同一套代码——这本身就是"纸面可信度"的来源：纸面与回测的摩擦模型零差异）。paper 模式下的差异行为：

| 维度 | 回测模式 | paper 模式 |
|:---|:---|:---|
| 驱动价格 | 历史 bar 的 OHLC | LiveDriver 的实时 QuoteSnapshot（`quant:quotes:stream` 订阅，降级轮询） |
| 市价单成交价 | bar close ± 滑点 | 最新 quote.price ± 滑点（quote.stale=True 时**拒单**而非乱成交，记 rejected 日志） |
| 限价单撮合 | bar high/low 穿越判断 | quote 价格穿越判断（逐快照检查，粒度受行情频率限制，文档明示） |
| 成交落地 | 内存 trades 列表 | **同步写 `paper_fills`**（先落库后回调 `on_order_update`，保证账本先于策略感知） |
| 交易时段 | 无需检查 | `MarketSession.is_trading_hours` 之外拒单（防收盘后 stale 价成交） |

**资金校验**：SimBroker 在 paper 模式下启用现金约束（买入超现金拒单）与持仓约束（卖出超持仓拒单）——回测模式此逻辑已有，参数同源。

**幂等与恢复**：Bot 重启 → LiveDriver 恢复 → SimBroker 从 `paper_positions` + `quant:paper:{pid}:open_orders`（Redis）恢复状态；`intent_tag` + fill_seq 去重防止重放期间重复记账。

## 五、净值结算（PaperSettlementDaemon）

### 5.1 触发机制（沿用既有 daemon 范式）

```
常驻 asyncio task（挂 worker.py，与 API 进程解耦）：
  while True:
    每 5 分钟检查各市场状态：
      ── 盘中（is_trading_hours）→ 盘中快照：
           批量取 quant:quotes:latest（缺失回退 get_market_snapshots）
           → NAV 写 Redis List quant:paper:{pid}:nav_intraday（ltrim 288）
           （仅展示用途，不落 PG，行情断流可容忍缺帧）
      ── 收盘后触发窗口（HK: 16:30–20:00 HKT；US: 16:30–20:00 ET）→ EOD 结算：
           锁 quant:lock:paper_settle:{market}:{yyyymmdd}（NX, ex=4h）
           对该市场所有 running 组合执行日终结算（见 5.2）
    与 kline_warehouse 3:00 同步错峰：结算窗口内轮询等待当日 K_DAY bar 就绪，
    最迟兜底至次日 kline 同步完成后补结算（见 5.3）
```

### 5.2 EOD 结算流程（单组合）

```
1. 判定今天是否交易日：kline_warehouse 中该市场基准标的（HK: 00700 / US: SPY）
   是否产出当日 K_DAY bar —— 数据驱动判定，绕开"无节假日表"缺口
   → 无 bar = 非交易日/数据未到，跳过（由 5.3 补结算兜底）
2. 逐持仓取收盘价：kline_warehouse.get_history() 当日 close
   ├─ 有 bar → 采用
   ├─ 无 bar（停牌/退市嫌疑）→ detect_suspension 确认 → 沿用前收盘价，
   │   symbol 记入 stale_symbols
   └─ 连续 N=5 日 stale → 组合标记 warning，前端黄色提示
3. NAV = cash + Σ(qty × close)；daily_return 相对前一交易日 nav
4. 原子写 paper_nav_daily（复合 PK 天然幂等，重复结算覆盖同值）
5. 结算后钩子：触发绩效指标增量更新（§六）+ 偏离检查（§6.3）
```

### 5.3 补结算与对账（可信度保障）

- **补结算**：daemon 每次醒来检查各组合 `paper_nav_daily` 最新日期与最近交易日的缺口，缺口 ≤7 天自动逐日补结算（收盘价历史可查，补算无损）；>7 天记 ERROR 告警人工介入。
- **账本对账**：每周日跑一次重放校验——从 `paper_fills` 全量重放得到的持仓/现金 vs `paper_positions` 投影，不一致即 CRITICAL 告警（说明记账路径有 bug）。这是账本系统的自检底线。

### 5.4 价格口径（明确声明，防歧义）

- 收盘价采用 kline_warehouse 现行口径（**前复权**）。分红/拆股发生时前复权历史会整体平移，导致"持仓成本 vs 复权价"出现口径差——V1 处置：结算用当日原始 close 语义（当日 bar 的 close 即当日真实价，前复权只影响历史段），NAV 当日值不失真；**历史 NAV 序列不回溯重算**（净值是当时事实）。拆股导致的 qty 变化 V1 不自动处理，组合详情页提示人工调整（个人系统低频事件，YAGNI）。

## 六、绩效归档与回测对比（PT-02）

### 6.1 共享绩效库（顺手还债）

新增 `backend/services/performance.py`：**纯函数、矢量化（NumPy/Pandas）、无 I/O**：

```
输入统一为日收益/净值 Series：
  sharpe(returns, freq=252) / max_drawdown(nav) / annualized_return(nav)
  / volatility / win_rate(trades) / tracking_error(r_a, r_b)
  / active_return(r_a, r_b)
```

消费方：PT 绩效归档（本期接入）、回测引擎 `event_engine`/`runners`（BT-01b 平移撮合时顺手切换调用，消除三处内联重复）、`risk_engine`（后续迭代自愿迁移）。测试目标 ≥90%（纯函数，doc 02 Utils 级标准）。

### 6.2 对比数据对齐（PT-02 的技术核心）

```
纸面曲线:  paper_nav_daily（交易日频，真实日期轴）
回测基准:  benchmark_backtest_ref 的 equity_curve（回测的历史日期轴）
对齐策略:  双方归一化为"自组合启动日起的累计收益率"（nav/nav₀），
          按交易日序号对齐（第 N 个交易日 vs 第 N 个交易日），而非日历日期
          —— 回测是历史区间、纸面是当下区间，日期轴天然不重合，
             序号对齐才有"策略行为一致性"的可比语义
输出指标:  跟踪误差 TE = std(r_paper − r_backtest) × √252
          累计偏离 = cum_paper − cum_backtest
          偏离归因摘要（轻量版）：
            信号一致率 = 双方同日（序号）持仓方向一致的天数占比
            成交偏离 = 同信号日的成交价差均值（滑点/延迟实现差）
```

**归因不做全量交易逐笔 diff**（回测历史区间与纸面区间的行情本就不同，逐笔对不上是常态）——V1 的偏离归因回答一个问题即可："偏离主要来自策略在新行情下的行为漂移（信号不一致），还是执行摩擦（价差）"。

### 6.3 偏离告警（复用 ALERT-01 引擎）

EOD 结算钩子中计算滚动 20 交易日 TE 与累计偏离，超组合配置阈值（默认 TE>15% 年化 或 累计偏离>10pp）→ 写入 AlertEngine 的规则触发通道（规则类型新增 `paper_drift`，复用现有去重冷却/多通道推送）。**不自建通知链路。**

### 6.4 实盘前检查点（纪律产品化）

`deploy-to-oms`（实盘部署）确认框展示该策略绑定纸面组合的三项硬数据：纸面运行交易日数 / 纸面 Sharpe / 当前 TE。无纸面组合或运行 <20 交易日 → 橙色警示文案（不拦截）。改造点挂在 docs/16 STRAT-03a 之后的 deploy 流程上，前端一处文案组件。

## 九、风险与开放问题

| # | 风险/开放问题 | 处置 |
|:---|:---|:---|
| 1 | PT-01b 强依赖 BT-01d/e，若 BT-01 延期则执行层空转 | 01a/01c/02a 均不依赖执行层可先行（账本+结算+绩效可用历史 fixture 测试）；最坏情况下 01b 前可提供"手动记账 API"过渡（不建议，评审定夺） |
| 2 | 限价单撮合粒度受行情快照频率限制（非逐笔），纸面成交优于/劣于真实的系统性偏差 | 文档明示 + TE 归因中"成交偏离"项可观测该偏差量级；V1 接受 |
| 3 | 交易日数据驱动判定依赖基准标的 bar 及时产出，kline 同步延迟会推迟结算 | 补结算机制兜底（≤7 天自愈）；结算窗口拉长至收盘后 3.5h |
| 4 | 前复权口径下拆股/分红的持仓数量不自动调整 | V1 人工调整 + 组合详情页提示；发生频率低，记录 V2 观察项 |
| 5 | 盘中 NAV 快照与日终结算价可能跳变（快照用实时价、结算用官方收盘价） | 前端盘中段虚线渲染 + 日终点实心，视觉上区分两种数据质量 |
| 6 | 序号对齐的对比语义在"纸面中途暂停"时错位 | 暂停日不产生序号（结算跳过 paused 组合），恢复后序号连续；文档明示对比序列为"实际运行交易日" |
