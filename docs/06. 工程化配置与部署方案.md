## 一、全局部署拓扑设计

### 1.1 节点规划原则

个人量化系统的部署面临三个核心约束：
1. **akshare 等国内数据源必须从境内直连**（境外 IP 无法访问或被限速）
2. **yfinance / finnhub / futuopenai 等海外数据源需境外 IP**（境内 IP 被封或严重限速）
3. **Web 前端需要对全球用户（包括中国大陆）快速访问**，通过 Cloudflare Pages CDN 分发

基于以上约束，采用 **统一 Compose + Profiles** 的主从采集集群架构：

```
                    Cloudflare 边缘网络（全球免费）
                    ┌─────────────────────────────┐
                    │  Pages (前端 CDN)            │
                    │  R2 (对象存储)               │
                    └─────────────┬───────────────┘
                                  │ HTTPS
     ┌────────────────────────────┴────────────────────────────┐
     │                                                         │
 Master 节点 (COMPOSE_PROFILES=master)        Slave 节点 (COMPOSE_PROFILES=slave)
 ┌──────────────────────────────────┐    ┌──────────────────────────────────┐
 │  redis / postgres / quant-agent  │    │  slave-collector (轻量 FastAPI)  │
 │  quant-worker                    │    │  无本地 Redis/PG                │
 │  + monitoring (可选)             │    │  通过 MASTER_NODES 连接 Master   │
 │                                  │    │  心跳多写所有 Master Redis       │
 │  ClusterManager 管理从节点服务池  │←───│  按需采集写入调用方 Master Redis  │
 │  带 failover 的 HTTP 代理调用    │───→│  collector_registry 声明能力    │
 └──────────────────────────────────┘    └──────────────────────────────────┘
         ↑ 北京 VPS (4C4G)                       ↑ 加州 VPS (2C2G)
           COMPOSE_PROFILES=master                 COMPOSE_PROFILES=slave
```

**核心设计**：
- **单一 Compose 文件**：`docker-compose.yml` 通过 `profiles` 控制角色，代码拉取到 VPS 后 `docker compose build && docker compose up -d`
- **多 Master 支持**：Slave 通过 `MASTER_NODES` JSON 数组注册到多个 Master 的 Redis，心跳多写
- **请求级回调**：Master 调用 Slave 采集时，携带本 Master 的 Redis 连接信息，Slave 将数据写入调用方 Redis

### 1.2 节点角色与 Compose Profiles

| Profile | 服务 | 节点位置 | 说明 |
|:---|:---|:---|:---|
| `master` | redis, postgres, quant-agent, quant-worker | 北京 VPS | API + 基础设施 + 采集器 |
| `slave` | slave-collector | 加州/海外 VPS | 轻量采集 API，连接 Master Redis |
| `monitoring` | prometheus, grafana | 北京 VPS (可选) | 监控面板 |

**环境变量控制**（`.env`）：
```bash
# Master 节点
COMPOSE_PROFILES=master              # 或 master,monitoring 启用监控
MASTER_NODES=[]                      # Master 自身为空数组

# Slave 节点
COMPOSE_PROFILES=slave
MASTER_NODES=[{"id":"beijing-1","host":"100.x.x.x","port":6379,"password":"...","db":0}]
NODE_PORT=8001                       # Slave 对外端口
```

### 1.3 集群管理与服务发现架构

```
┌──────────────────────────────────────────┐
│          Master 节点 (北京 VPS)           │
│                                          │
│  quant-agent (FastAPI 主服务)            │
│    └─ ClusterManager                     │
│       ├─ 扫描 Redis 发现 slave 节点      │
│       ├─ 按采集器类型维护服务池           │
│       │   collector → [slave_node, ...]  │
│       └─ 带 failover 的 HTTP 代理调用    │
│          (失败自动降级，恢复自动升级)      │
│                                          │
│  Redis: quant:node:{node_id}            │
│         (slave 心跳写入, TTL 15s)        │
└──────────────────┬───────────────────────┘
                   │ HTTP (携带 callback_redis)
┌──────────────────┴───────────────────────┐
│          Slave 节点 (加州 VPS)            │
│                                          │
│  slave-collector (slave_app.py)          │
│    ├─ /health           健康检查          │
│    ├─ /collect/{action} 按需采集         │
│    ├─ 心跳循环 → 写入所有 Master Redis   │
│    └─ collector_registry 声明采集能力     │
└──────────────────────────────────────────┘
```

**集群管理核心 — ClusterManager**：

```python

    """集群管理器 - 服务发现 + 服务池 + failover 代理 (多 Master 支持)"""

    async def refresh_pool(self):
        """扫描 Redis 中所有 quant:node:* 键，刷新服务池"""
        # 按 collector 类型分组 slave 节点

    async def call_slave(self, collector: str, action: str, params: dict,
                         callback_redis: dict) -> dict:
        """带 failover 的 HTTP 代理调用"""
        # 1. 从服务池选取可用节点
        # 2. HTTP POST /collect/{action}，携带 callback_redis
        # 3. 失败时自动切换下一节点
        # 4. 连续失败 3 次标记 unhealthy，30s 后尝试恢复
```

**从节点注册 — Slave 心跳协议**：

```python
# backend/slave_app.py — 多 Master 心跳

async def _multi_heartbeat_loop():
    """同时向所有 Master Redis 写入心跳"""
    for client in multi_redis.all_clients:
        await client.set(f"quant:node:{NODE_ID}", node_json, ex=15)
    # 若无 Master 配置，写本地 Redis (独立运行模式)
    if not multi_redis.all_clients:
        await redis_client.set(node_key, node_json, ex=15)
```

**采集器注册表 — CollectorRegistry**：


### 5. Git 分支策略与 CI/CD 触发规则

**双分支模型**：

```
main       ← 受保护的生产分支，仅接受来自 develop 的 PR
  └─ develop  ← 日常开发分支，所有功能在此合并
       ├─ feature/xxx  ← 可选的功能分支，完成后 PR 回 develop
       └─ fix/xxx      ← 可选的修复分支
```

**分支保护规则**（GitHub Settings → Branches → main）：
- ✅ **Require a pull request before merging** — 禁止直接 push 到 main
- ✅ **Require status checks to pass** — CI 必须通过才能合并
  - `backend-ci / lint`
  - `backend-ci / test`
  - `backend-ci / security`
- ✅ **Do not allow force pushes** — 禁止强制推送
- ✅ **Do not allow deletions** — 禁止删除分支
- ✅ **Do not allow bypassing** — 管理员也不绕过

**CI/CD 触发规则**：

| 事件 | lint | test | security | 🚀 部署 |
|------|:---:|:---:|:---:|:---:|
| push `develop` | ✅ | ✅ | ✅ | ❌ |
| PR → `main` | ✅ | ✅ | ✅ | ❌ (质量门禁) |
| push `main` (merge) | ✅ | ✅ | ✅ | ✅ 三节点矩阵部署 |
| `workflow_dispatch` | ✅ | ✅ | ✅ | ✅ 可选目标节点 |

**日常开发流程**：

```bash
# 1. 拉取最新代码
git checkout develop
git pull origin develop

# 2. 同步依赖（如有变更）
make install

# 3. 启动开发环境
make dev

# 4. 编码 & 实时测试
#    - 修改代码后 API 自动重载
#    - 前端修改实时生效 (HMR)

# 5. 运行测试验证
make test
make test-cluster  # 如涉及集群逻辑

# 6. 提交前检查
make lint
make format

# 7. 提交并推送
git add . && git commit -m "feat(xxx): xxx"
git push origin develop
# → 自动触发 CI (lint + test)，不部署
```

**准备发布**：


## 二、Cloudflare 免费资源完整利用方案

### 2.1 Cloudflare Pages（前端 SPA 托管）

**零费用替代 Nginx 静态服务**，全球 CDN 节点，国内访问速度显著优于 VPS 直连：

```
# .github/workflows/deploy-frontend.yml

on:
  push:
    branches: [main]
    paths: ['frontend/**']

jobs:
  deploy-pages:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'pnpm' }
      - run: pnpm install && pnpm build
        working-directory: frontend
        env:
          VITE_API_BASE_URL: ${{ secrets.API_BASE_URL }}   # Tailscale 内网 IP 或公网域名
          VITE_WS_BASE_URL:  ${{ secrets.WS_BASE_URL }}
      - uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CF_API_TOKEN }}
          accountId: ${{ secrets.CF_ACCOUNT_ID }}
          command: pages deploy frontend/dist --project-name=quant-edge
```

**Cloudflare Pages 配置**：
```
项目名：quant-edge
构建命令：pnpm build（GitHub Actions 负责，不用 CF 内置构建）
部署目录：frontend/dist
自定义域名：app.yourdomain.com（绑定后 SSL 自动签发）
```

### 2.4 Cloudflare R2（对象存储）

**完全替代 AWS S3**，免费额度：10GB 存储 + 100万次 Class A 操作 / 月，存储研报 PDF、回测结果等：

```python
# backend/services/r2_storage.py — 使用 boto3（S3 兼容 API）

import boto3
from botocore.config import Config

r2_client = boto3.client(
    's3',
    endpoint_url=f'https://{CF_ACCOUNT_ID}.r2.cloudflarestorage.com',
    aws_access_key_id=CF_R2_ACCESS_KEY,
    aws_secret_access_key=CF_R2_SECRET_KEY,
    config=Config(signature_version='s3v4'),
    region_name='auto',
)

async def upload_backtest_report(report_id: str, content: bytes) -> str:
    key = f"reports/{report_id}.html"
    r2_client.put_object(
        Bucket='quant-reports',
        Key=key,
        Body=content,
        ContentType='text/html',
    )
    return f"https://reports.yourdomain.com/{key}"  # R2 自定义域名
```

### 2.5 Cloudflare Workers（边缘缓存 / 轻量 API）

**适合：API 缓存代理、宏观日历静态数据缓存**（减少 VPS 请求压力）：

```javascript
// cloudflare-workers/macro-cache-worker.js
// 部署：wrangler deploy — 全球边缘节点，响应 < 10ms

export default {
  async fetch(request, env) {
    const url = new URL(request.url)

    // 宏观日历：每小时缓存
    if (url.pathname === '/macro/calendar') {
      const cached = await env.MACRO_KV.get('calendar', 'json')
      if (cached) {
        return Response.json(cached, {
          headers: { 'X-Cache': 'HIT', 'Cache-Control': 'max-age=3600' }
        })
      }
      // 穿透到 VPS
      const upstream = await fetch(`https://api.yourdomain.com${url.pathname}`, {
        headers: { 'Authorization': request.headers.get('Authorization') }
      })
      const data = await upstream.json()
      await env.MACRO_KV.put('calendar', JSON.stringify(data), { expirationTtl: 3600 })
      return Response.json(data, { headers: { 'X-Cache': 'MISS' } })
    }

    return fetch(request)
  }
}
```

### 3.1 GitHub Container Registry（Docker 镜像仓库）

**替代 Docker Hub**，私有镜像免费，无速率限制：

```yaml
# .github/workflows/build-push.yml

on:
  push:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}   # ghcr.io/yourusername/quant_agent

jobs:
  build-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
```

```bash
COLLECTOR_FINNHUB=true
COLLECTOR_FUTU=true
FINNHUB_API_KEY=xxx
```

### 4.2 多 Master 心跳与回调机制

```
Slave 启动 → 读取 MASTER_NODES JSON 数组
  ↓
为每个 Master 创建独立 Redis 客户端 (MultiRedis)
  ↓
心跳循环 (每 5s):
  → 遍历所有 Master Redis
  → SET quant:node:{node_id} {node_json} EX 15
  → node_json 包含: node_id, host, port, collectors[], status
  ↓
Master 的 ClusterManager 扫描 Redis:
  → SCAN quant:node:* 发现所有活跃 slave
  → 按 collector 类型维护服务池
  ↓
Master 调用 Slave 采集:
  → HTTP POST /collect/{action}
  → body 中携带 callback_redis: {host, port, password, db}
  → Slave 采集完成后将数据写入调用方 Master 的 Redis
```

### 4.3 条件化 Local Redis

Slave 节点**不运行本地 Redis**。当 `MASTER_NODES` 未配置时（独立开发模式），Slave 会降级使用本地 Redis：

```python
# slave_app.py lifespan
_use_local_redis = master_count == 0
if _use_local_redis:
    redis_batch_writer.start()  # 独立模式启用本地 batch writer
```

### 4.4 采集集群降级设计（防 Slave 断连）

```
Slave 断连期间:
  ↓ ClusterManager 心跳检测超时 (> 15s TTL 过期)
  ↓ 服务池移除该节点
  ↓ Master 采集请求降级:
    - 有本地采集器 → 本地直接执行
    - 无本地采集器 → 返回 Redis 最后一次有效数据 (带 STALE 标记)
  ↓ Slave 恢复 → 心跳重新写入 Redis → ClusterManager 自动发现恢复
```

## 六、向量数据库（pgvector）架构

### 6.1 表结构设计

```
-- PostgreSQL + pgvector 向量存储表

CREATE EXTENSION IF NOT EXISTS vector;

-- 知识库文档块（RAG 检索）
CREATE TABLE knowledge_chunks (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_url  TEXT NOT NULL,
    title       TEXT,
    content     TEXT NOT NULL,
    embedding   vector(1536),           -- OpenAI text-embedding-3-small 维度
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW(),
    expires_at  TIMESTAMPTZ,            -- NULL = 永不过期；有值 = 定期清理过期研报
    metadata    JSONB DEFAULT '{}'
);

-- 向量相似度索引（IVFFlat，平衡速度与精度）
CREATE INDEX idx_knowledge_embedding
ON knowledge_chunks USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);                     -- lists = sqrt(行数)，按需调整

-- 全文检索索引（混合检索：向量 + 关键词）
CREATE INDEX idx_knowledge_content_fts
ON knowledge_chunks USING gin(to_tsvector('english', content));
```

### 6.2 向量查询（混合检索）

```python
# backend/services/knowledge_service.py

async def hybrid_search(
    query: str,
    query_embedding: list[float],
    days_back: int | None = None,
    top_k: int = 10,
) -> list[KnowledgeChunk]:
    """向量相似度 + 关键词的混合检索"""

    expire_filter = ""
    if days_back:
        expire_filter = f"AND created_at > NOW() - INTERVAL '{days_back} days'"

    sql = """
    SELECT id, content, source_url, title,
           1 - (embedding <=> $1::vector) AS similarity,   -- 余弦相似度
           ts_rank(to_tsvector('english', content), plainto_tsquery($2)) AS text_rank
    FROM knowledge_chunks
    WHERE expires_at IS NULL OR expires_at > NOW()
    {expire_filter}
    ORDER BY
      (1 - (embedding <=> $1::vector)) * 0.7 +            -- 向量权重 70%
      ts_rank(to_tsvector('english', content), plainto_tsquery($2)) * 0.3  -- 关键词 30%
    DESC
    LIMIT $3
    """.format(expire_filter=expire_filter)

    return await db.fetch(sql, query_embedding, query, top_k)
```

## 七、第三方数据获取稳定性设计

### 7.1 Futu OpenD 隔离（已覆盖 §四）

### 7.2 外部 API（YFinance / FRED / NewsAPI）稳定抓取

> **架构变更**：YFinance / Finnhub / FutuOpenAI 等海外数据源已迁移至加州子服务节点（节点 B），北京主服务通过服务注册表动态路由。

```python
# data_subservice/fetcher.py（运行在加州节点）

class ExternalDataFetcher:
    """外部 API 熔断 + 重试 + Redis 缓存三层防护"""

    def __init__(self, redis: Redis):
        self.redis = redis
        self._circuit_breakers: dict[str, CircuitBreaker] = {}

    async def fetch_with_cache(
        self,
        cache_key: str,
        fetch_fn: Callable,
        ttl: int = 300,
        circuit_name: str = "default",
    ) -> Any:
        # 1. 先查 Redis 缓存
        cached = await self.redis.get(cache_key)
        if cached:
            return json.loads(cached)

        # 2. 熔断器检查（连续失败 3 次则熔断 60s）
        cb = self._circuit_breakers.setdefault(
            circuit_name, CircuitBreaker(fail_max=3, reset_timeout=60)
        )

        try:
            with cb:
                data = await asyncio.to_thread(fetch_fn)
                await self.redis.setex(cache_key, ttl, json.dumps(data))
                return data
        except CircuitBreakerError:
            logger.warning('circuit_open', source=circuit_name)
            stale = await self.redis.get(f"stale:{cache_key}")
            return {"data": json.loads(stale), "stale": True} if stale else None
        except Exception as e:
            logger.error('fetch_failed', source=circuit_name, error=str(e))
            await self.redis.rename(cache_key, f"stale:{cache_key}")
            raise
```
