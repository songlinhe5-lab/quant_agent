## 一、全局部署拓扑设计

### 1.1 架构概述（三节点高可用方案）

采用三节点高可用架构，解决 YFinance 限流问题和 AKShare 国内数据源访问问题：

```
                    Cloudflare 边缘网络
                    ┌───────────────────────────────────────────────┐
                    │  Pages (前端 CDN)                              │
                    └──────────────────┬────────────────────────────┘
                                       │ HTTPS
                    ┌──────────────────┴────────────────────────────┐
                    │                                              │
                    │  ┌─────────────────────────────────────────┐  │
                    │  │  海外主节点 (Master Node)                │  │
                    │  │  ┌─────────────────────────────────┐    │  │
                    │  │  │ redis / postgres                │    │  │
                    │  │  │ quant-agent (FastAPI)           │    │  │
                    │  │  │ quant-worker                    │    │  │
                    │  │  │ + monitoring (可选)             │    │  │
                    │  │  │ Futu OpenD (宿主机本地)         │    │  │
                    │  │  └─────────────────────────────────┘    │  │
                    │  └─────────────────┬───────────────────────┘  │
                    │                    │ Tailscale 内网              │
                    │  ┌─────────────────┴───────────────────────┐  │
                    │  │  海外备用节点 (YFinance Backup)         │  │
                    │  │  ┌─────────────────────────────────┐    │  │
                    │  │  │ quant-agent (仅 YFinance)       │    │  │
                    │  │  └─────────────────────────────────┘    │  │
                    │  └─────────────────────────────────────────┘  │
                    │                                              │
                    │  ┌─────────────────────────────────────────┐  │
                    │  │  国内 AKShare 节点 (CN AKShare Node)   │  │
                    │  │  ┌─────────────────────────────────┐    │  │
                    │  │  │ quant-agent (仅 AKShare)         │    │  │
                    │  │  └─────────────────────────────────┘    │  │
                    │  └─────────────────────────────────────────┘  │
                    └───────────────────────────────────────────────┘
```

**核心设计**：
- **三节点部署**：海外主节点承载核心业务，海外备用节点应对 YFinance 限流，国内节点提供 AKShare 数据
- **Tailscale 组网**：节点间通过 Tailscale 内网通信，安全且无需公网暴露
- **数据源路由**：主节点通过 `data_source_router` 自动路由到最优数据源
- **独立 Compose 文件**：每个节点有独立的 compose 配置文件

### 1.2 节点职责矩阵

| 节点 | 定位 | 核心服务 | 数据源 | Compose 文件 |
|:---|:---|:---|:---|:---|
| **海外主节点** | 核心业务 | Redis, PostgreSQL, quant-agent, quant-worker | Futu OpenD(本地), YFinance(本地+远程), AKShare(远程) | `docker-compose.master.yml` |
| **海外备用节点** | YFinance 备用 | 仅 quant-agent | YFinance(本地) | `docker-compose.yf-backup.yml` |
| **国内数据节点** | 国内数据源 | 仅 quant-data-worker | AKShare(本地), YFinance(本地) | `docker-compose.slave.yml` |

### 1.3 单一 VPS 简化模式（旧方案，保留兼容）

对于测试环境或小规模部署，仍支持单一 VPS 模式：

```bash
# 使用旧版单一 VPS 配置
docker-compose up -d
```

详见 §八、单一 VPS 简化部署方案。

### 1.4 服务矩阵（主节点）

| 服务 | 说明 | 端口 |
|:---|:---|:---|
| redis | 行情缓存 | 6379 |
| postgres | 数据持久化 + 向量库 | 5432 |
| quant-agent | FastAPI 主服务 API | 8000 |
| quant-worker | Worker 进程（采集器 daemon + 后台任务） | - |

### 5. Git 分支策略与 CI/CD 触发规则

**双分支模型**：

```
main       ← 受保护的生产分支，仅接受来自 develop 的 PR
  └─ develop  ← 日常开发分支，所有功能在此合并
       ├─ feature/xxx  ← 可选的功能分支，完成后 PR 回 develop
       └─ fix/xxx      ← 可选的修复分支
```

**分支保护规则**（GitHub Settings → Branches → main）：
- ✅ **Require a pull request before merging** — 禁止直接 push 到 main
- ✅ **Require status checks to pass** — CI 必须通过才能合并
  - `backend-ci / lint`
  - `backend-ci / test`
  - `backend-ci / security`
- ✅ **Do not allow force pushes** — 禁止强制推送
- ✅ **Do not allow deletions** — 禁止删除分支
- ✅ **Do not allow bypassing** — 管理员也不绕过

**CI/CD 触发规则**：

| 事件 | lint | test | security | 🚀 部署 |
|------|:---:|:---:|:---:|:---:|
| push `develop` | ✅ | ✅ | ✅ | ❌ |
| PR → `main` | ✅ | ✅ | ✅ | ❌ (质量门禁) |
| push `main` (merge) | ✅ | ✅ | ✅ | ✅ 三节点矩阵部署 |
| `workflow_dispatch` | ✅ | ✅ | ✅ | ✅ 可选目标节点 |

**日常开发流程**：

```bash
# 1. 拉取最新代码
git checkout develop
git pull origin develop

# 2. 同步依赖（如有变更）
make install

# 3. 启动开发环境
make dev

# 4. 编码 & 实时测试
#    - 修改代码后 API 自动重载
#    - 前端修改实时生效 (HMR)

# 5. 运行测试验证
make test
make test-cluster  # 如涉及集群逻辑

# 6. 提交前检查
make lint
make format

# 7. 提交并推送
git add . && git commit -m "feat(xxx): xxx"
git push origin develop
# → 自动触发 CI (lint + test)，不部署
```

**准备发布**：

```bash

# 通过后合并到 main → 自动触发三节点部署
```

## 二、Cloudflare 免费资源完整利用方案

### 2.1 Cloudflare Pages（前端 SPA 托管）

**零费用替代 Nginx 静态服务**，全球 CDN 节点，国内访问速度显著优于 VPS 直连：

```
# .github/workflows/deploy-frontend.yml

on:
  push:
    branches: [main]
    paths: ['frontend/**']

jobs:
  deploy-pages:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'pnpm' }
      - run: pnpm install && pnpm build
        working-directory: frontend
        env:
          VITE_API_BASE_URL: ${{ secrets.API_BASE_URL }}   # Tailscale 内网 IP 或公网域名
          VITE_WS_BASE_URL:  ${{ secrets.WS_BASE_URL }}
      - uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CF_API_TOKEN }}
          accountId: ${{ secrets.CF_ACCOUNT_ID }}
          command: pages deploy frontend/dist --project-name=quant-edge
```

**Cloudflare Pages 配置**：
```
项目名：quant-edge
构建命令：pnpm build（GitHub Actions 负责，不用 CF 内置构建）
部署目录：frontend/dist
自定义域名：app.yourdomain.com（绑定后 SSL 自动签发）
```

### 2.2 Tailscale（零信任内网穿透）

**核心价值**：北京与加州 VPS 组建 Tailscale Tailnet 私有网络，跨节点通信走加密 P2P 直连，无需开放任何公网端口。管理页面（Grafana / Prometheus）通过 Tailscale 内网 IP 访问，无需暴露到公网。

```bash

# Ubuntu/Debian

# 启动并登录（首次会输出登录链接）

# 验证连接
tailscale status
# 输出示例：
# 100.x.x.x  beijing-node    linux  active
# 100.x.x.y  california-node linux  active
```

**跨节点通信配置**（北京 ↔ 加州）：

```python
# .env.node-b（加州节点）
MAIN_SERVICE_URL=http://100.x.x.x:8000  # 北京节点的 Tailscale IP
HMAC_SECRET=your_hmac_secret
```

**SSH 安全接入**：

### 2.4 Cloudflare R2（对象存储）

**完全替代 AWS S3**，免费额度：10GB 存储 + 100万次 Class A 操作 / 月，存储研报 PDF、回测结果等：

```python
# backend/services/r2_storage.py — 使用 boto3（S3 兼容 API）

import boto3
from botocore.config import Config

r2_client = boto3.client(
    's3',
    endpoint_url=f'https://{CF_ACCOUNT_ID}.r2.cloudflarestorage.com',
    aws_access_key_id=CF_R2_ACCESS_KEY,
    aws_secret_access_key=CF_R2_SECRET_KEY,
    config=Config(signature_version='s3v4'),
    region_name='auto',
)

async def upload_backtest_report(report_id: str, content: bytes) -> str:
    key = f"reports/{report_id}.html"
    r2_client.put_object(
        Bucket='quant-reports',
        Key=key,
        Body=content,
        ContentType='text/html',
    )
    return f"https://reports.yourdomain.com/{key}"  # R2 自定义域名
```

**R2 存储分类**：
```
quant-reports/
  ├── backtest/{strategy_id}/{timestamp}.html    回测撕裂报告
  ├── research/{company}/{date}.pdf              研报 PDF
  └── exports/{user_id}/{timestamp}.csv          持仓导出

quant-backups/
  ├── redis/{date}/dump.rdb                      Redis 每日备份
  └── pgvector/{date}/vector_export.pgdump       向量库备份
```

### 2.5 Cloudflare Workers（边缘缓存 / 轻量 API）

**适合：API 缓存代理、宏观日历静态数据缓存**（减少 VPS 请求压力）：

```javascript
// cloudflare-workers/macro-cache-worker.js
// 部署：wrangler deploy — 全球边缘节点，响应 < 10ms

export default {
  async fetch(request, env) {
    const url = new URL(request.url)

    // 宏观日历：每小时缓存
    if (url.pathname === '/macro/calendar') {
      const cached = await env.MACRO_KV.get('calendar', 'json')
      if (cached) {
        return Response.json(cached, {
          headers: { 'X-Cache': 'HIT', 'Cache-Control': 'max-age=3600' }
        })
      }
      // 穿透到 VPS
      const upstream = await fetch(`https://api.yourdomain.com${url.pathname}`, {
        headers: { 'Authorization': request.headers.get('Authorization') }
      })
      const data = await upstream.json()
      await env.MACRO_KV.put('calendar', JSON.stringify(data), { expirationTtl: 3600 })
      return Response.json(data, { headers: { 'X-Cache': 'MISS' } })
    }

    return fetch(request)
  }
}
```

## 三、GitHub 免费资源完整利用方案

### 3.1 GitHub Container Registry（Docker 镜像仓库）

**替代 Docker Hub**，私有镜像免费，无速率限制：

```yaml
# .github/workflows/build-push.yml

on:
  push:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}   # ghcr.io/yourusername/quant_agent

jobs:
  build-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - uses: docker/metadata-action@v5
        id: meta
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=sha-
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}

      - uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha        # GitHub Actions 缓存层 — 加快构建
          cache-to: type=gha,mode=max
```

**VPS 拉取镜像部署**：
```bash
# 节点 A 的部署脚本
echo $GITHUB_TOKEN | docker login ghcr.io -u yourusername --password-stdin
docker pull ghcr.io/yourusername/quant_agent:latest
docker-compose up -d --no-build   # 直接用拉取的镜像，无需本地构建
```

### 3.2 GitHub Actions CI/CD 完整流水线

```
代码推送 → 质量检查 → 构建镜像 → 自动部署 → 通知
```

```yaml
# .github/workflows/ci-cd.yml — 完整流水线

name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  # ── 质量门禁 ──────────────────────────────────────────
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
      - run: uv sync --all-extras
      - run: uv run ruff check && uv run ruff format --check
      - run: uv run pytest tests/ -x --tb=short -q
        env:
          REDIS_HOST: localhost        # CI 环境用本地 Redis
          DATABASE_URL: sqlite:///./test.db

  # ── 前端构建 + Cloudflare Pages 部署 ─────────────────
  deploy-frontend:
    needs: quality-gate
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - run: pnpm install && pnpm build
        working-directory: frontend
        env:
          VITE_API_BASE_URL: ${{ secrets.API_BASE_URL }}
      - uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CF_API_TOKEN }}
          command: pages deploy frontend/dist --project-name=quant-edge

  # ── 构建并推送 Docker 镜像 ─────────────────────────────
  build-backend:
    needs: quality-gate
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions: { contents: read, packages: write }
    outputs:
      image_tag: ${{ steps.meta.outputs.version }}
    steps:
      - uses: actions/checkout@v4
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/metadata-action@v5
        id: meta
        with:
          images: ghcr.io/${{ github.repository }}
          tags: type=sha,prefix=sha-
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ── SSH 部署至节点 A（香港 VPS）─────────────────────────
  deploy-hk:
    needs: build-backend
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
```

## 四之一、跨节点数据源路由方案

### 4.5 YFinance 多源切换（限流适配）

**问题背景**：Yahoo Finance 对单一 IP 有严格的请求频率限制，频繁请求会触发 429 限流。

**解决方案**：引入多节点 YFinance 数据源，主节点被限流时自动切换到备用节点。

```python
# backend/services/data_source_router.py — 数据源路由服务

class DataSourceRouter:
    def __init__(self):
        self._nodes = {
            "yf_primary": DataSourceNode(
                name="yf_primary",
                url="http://localhost:8000",
                weight=10,
                capabilities=["yfinance", "quote", "history", "tech"],
            ),
            "yf_backup": DataSourceNode(
                name="yf_backup",
                url="http://备用VPS:8000",
                weight=5,
                capabilities=["yfinance", "quote", "history", "tech"],
            ),
        }

    async def fetch_yfinance(self, ticker, fetch_type, **kwargs):
        for node in self._get_healthy_nodes("yfinance"):
            try:
                result = await self._send_request(node, "yfinance", {
                    "ticker": ticker,
                    "fetch_type": fetch_type,
                    "kwargs": kwargs,
                })
                if "429" in str(result.get("message", "")):
                    await self._update_node_status(node.name, False, "rate_limit")
                    continue
                return result
            except:
                await self._update_node_status(node.name, False)
        return await self._fetch_local(ticker, fetch_type, **kwargs)
```

**环境变量配置**：

```bash
# .env — YFinance 多源切换配置

# 是否启用数据源路由（默认关闭，单一 VPS 模式）
DATA_SOURCE_ROUTER_ENABLED=false

# YFinance 主节点（当前 VPS）
YF_PRIMARY_NODE_URL=http://localhost:8000

# YFinance 备用节点（可选，当主节点被限流时切换）
YF_BACKUP_NODE_URL=http://100.x.x.x:8000

# 节点间通信签名密钥（防止伪造请求）
DATA_SOURCE_HMAC_SECRET=your-secret-key
```

**熔断与切换机制**：

```
YFinance 请求流程:
  ↓
  检测 DATA_SOURCE_ROUTER_ENABLED
  ↓
  ┌─ true ─→ 遍历健康节点（按权重排序）
  │           ├─ 尝试主节点请求
  │           │   ├─ 成功 → 返回数据
  │           │   └─ 失败/限流 → 标记节点状态，尝试下一个
  │           ├─ 尝试备用节点请求
  │           └─ 全部失败 → 降级到本地
  │
  └─ false ─→ 直接调用本地 yfinance_service
```

```python

async def fetch_akshare(self, action, **kwargs):
    remote_node = self._nodes.get("akshare_remote")
    
    if not remote_node or remote_node.status != "healthy":
        return await self._call_local_akshare(action, **kwargs)
    
    try:
        result = await self._send_request(remote_node, "akshare", {
            "action": action,
            "kwargs": kwargs,

    return await self._call_local_akshare(action, **kwargs)
```

## 六、向量数据库（pgvector）架构

### 6.1 表结构设计

```
-- PostgreSQL + pgvector 向量存储表

CREATE EXTENSION IF NOT EXISTS vector;

-- 知识库文档块（RAG 检索）
CREATE TABLE knowledge_chunks (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_url  TEXT NOT NULL,
    title       TEXT,
    content     TEXT NOT NULL,
    embedding   vector(1536),           -- OpenAI text-embedding-3-small 维度
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW(),
    expires_at  TIMESTAMPTZ,            -- NULL = 永不过期；有值 = 定期清理过期研报
    metadata    JSONB DEFAULT '{}'
);

-- 向量相似度索引（IVFFlat，平衡速度与精度）
CREATE INDEX idx_knowledge_embedding
ON knowledge_chunks USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);                     -- lists = sqrt(行数)，按需调整

-- 全文检索索引（混合检索：向量 + 关键词）
CREATE INDEX idx_knowledge_content_fts
ON knowledge_chunks USING gin(to_tsvector('english', content));
```

### 6.2 向量查询（混合检索）

```python
# backend/services/knowledge_service.py

async def hybrid_search(
    query: str,
    query_embedding: list[float],
    days_back: int | None = None,
    top_k: int = 10,
) -> list[KnowledgeChunk]:
    """向量相似度 + 关键词的混合检索"""

    expire_filter = ""
    if days_back:
        expire_filter = f"AND created_at > NOW() - INTERVAL '{days_back} days'"

    sql = """
    SELECT id, content, source_url, title,
           1 - (embedding <=> $1::vector) AS similarity,   -- 余弦相似度
           ts_rank(to_tsvector('english', content), plainto_tsquery($2)) AS text_rank
    FROM knowledge_chunks
    WHERE expires_at IS NULL OR expires_at > NOW()
    {expire_filter}
    ORDER BY
      (1 - (embedding <=> $1::vector)) * 0.7 +            -- 向量权重 70%
      ts_rank(to_tsvector('english', content), plainto_tsquery($2)) * 0.3  -- 关键词 30%
    DESC
    LIMIT $3
    """.format(expire_filter=expire_filter)

    return await db.fetch(sql, query_embedding, query, top_k)
```

### 7.2 外部 API（YFinance / FRED / Finnhub）稳定抓取

所有外部数据源直接在主服务中运行，通过熔断器 + 重试 + Redis 缓存三层防护：

```python
# backend/services/external_fetcher.py

class ExternalDataFetcher:
    """外部 API 熔断 + 重试 + Redis 缓存三层防护"""

    def __init__(self, redis: Redis):
        self.redis = redis
        self._circuit_breakers: dict[str, CircuitBreaker] = {}

    async def fetch_with_cache(
        self,
        cache_key: str,
        fetch_fn: Callable,
        ttl: int = 300,
        circuit_name: str = "default",
    ) -> Any:
        cached = await self.redis.get(cache_key)
        if cached:
            return json.loads(cached)

        cb = self._circuit_breakers.setdefault(
            circuit_name, CircuitBreaker(fail_max=3, reset_timeout=60)
        )

        try:
            with cb:
                data = await asyncio.to_thread(fetch_fn)
                await self.redis.setex(cache_key, ttl, json.dumps(data))
                return data
        except CircuitBreakerError:
            logger.warning('circuit_open', source=circuit_name)
            stale = await self.redis.get(f"stale:{cache_key}")
            return {"data": json.loads(stale), "stale": True} if stale else None
        except Exception as e:
            logger.error('fetch_failed', source=circuit_name, error=str(e))
            await self.redis.rename(cache_key, f"stale:{cache_key}")
            raise
```

GitHub Actions 配置三个独立的部署环境：

```yaml
# .github/workflows/backend.yml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/songlinhe5-lab/quant-agent:${{ github.sha }}

  deploy-master:
    needs: build
    runs-on: ubuntu-latest
    environment: master
    steps:
      - uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.MASTER_HOST }}
          username: ${{ secrets.MASTER_USER }}
          key: ${{ secrets.MASTER_SSH_KEY }}
          script: |
            cd /opt/quant_agent
            git pull origin main
            docker-compose -f docker-compose.master.yml pull
            docker-compose -f docker-compose.master.yml up -d
            sleep 10
            curl -sf http://localhost:8000/api/v1/health

  deploy-yf-backup:
    needs: build
    runs-on: ubuntu-latest
    environment: yf-backup
    steps:
      - uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.YF_BACKUP_HOST }}
          username: ${{ secrets.YF_BACKUP_USER }}
          key: ${{ secrets.YF_BACKUP_SSH_KEY }}
          script: |
            cd /opt/quant_agent
            git pull origin main
            docker-compose -f docker-compose.yf-backup.yml pull
            docker-compose -f docker-compose.yf-backup.yml up -d
            sleep 10
            curl -sf http://localhost:8000/api/v1/health

  deploy-akshare:
    needs: build
    runs-on: ubuntu-latest
    environment: akshare-cn
    steps:
      - uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          script: |
            cd /opt/quant-agent
            git pull origin main
            docker compose -f docker-compose.slave.yml pull
            docker compose -f docker-compose.slave.yml up -d
            sleep 10
            docker ps --filter name=quant_data_worker --filter status=running | grep -q quant_data_worker
```

## 九、安全加固清单

### 9.1 端口管理（最小暴露原则）

```
Tailscale 模式（推荐）：
  VPS 防火墙：所有入站端口关闭，仅允许 Tailscale 内网 IP 访问
  SSH 通过 tailscale ssh 替代公网 SSH，无需开放 22 端口
  跨节点通信走 Tailscale P2P 直连，无需公网暴露
  前端 API 通过 Cloudflare Pages 对外提供服务

备用 Nginx 模式（无 Tailscale）：
  开放：80, 443（Nginx）
  开放：22（SSH，改为 2222 或其他非标端口）
  禁止：所有其他端口
```

### 9.2 SSH 加固

```bash
# /etc/ssh/sshd_config
Port 2222                    # 非标准端口，避免机器人扫描
PermitRootLogin no
PasswordAuthentication no    # 只允许 SSH Key 登录
MaxAuthTries 3
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
```
