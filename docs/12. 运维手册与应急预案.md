## 二、日常运维操作

### 2.1 服务状态检查

```bash
# 香港 VPS（Node A — 执行核心）
ssh user@hk.yourdomain.com

# 查看所有服务状态
docker compose -f /opt/quant/docker-compose.prod.yml ps

# 查看 Futu OpenD 守护进程
systemctl status futu-opend

# 检查 API 健康状态
curl -s https://api.quant.yourdomain.com/api/v1/health | jq .

# 查看 Redis 状态
docker exec quant-redis redis-cli ping
docker exec quant-redis redis-cli info stats | grep -E "connected_clients|used_memory_human"
```

### 2.2 查看日志

```bash
# 实时日志（后端）
docker compose logs -f --tail=200 backend

# 查看特定时间段日志（JSON 格式，用 jq 过滤）
docker compose logs backend --since 1h | jq 'select(.level == "error")'

# Futu OpenD 日志
journalctl -u futu-opend -f --since "1 hour ago"

# 查看最近错误日志
docker compose logs backend --since 30m | grep '"level":"error"'
```

### 2.3 常用 Redis 操作

```bash
docker exec -it quant-redis redis-cli

# 查看所有行情快照 Key
KEYS quote:snapshot:*

# 查看特定标的快照
HGETALL quote:snapshot:US.AAPL

# 查看内存使用
MEMORY USAGE quote:snapshot:US.AAPL

# 手动清理过期 K 线缓存（强制刷新）
DEL kline:hot:US.AAPL:D1

# 查看熔断器状态
GET circuit:futu_api
GET circuit:yfinance_api
```

### 2.4 数据库操作

```bash
# 连接 PostgreSQL
docker exec -it quant-postgres psql -U quant_user -d quant_db

# 查看最近订单
SELECT order_id, symbol, side, status, created_at 
FROM orders ORDER BY created_at DESC LIMIT 20;

# 查看 RAG 知识库大小
SELECT source_type, COUNT(*) FROM knowledge_chunks GROUP BY source_type;

# 查看审计日志
SELECT action, detail, ip, created_at 
FROM audit_logs ORDER BY created_at DESC LIMIT 50;
```

### 2.5 手动触发部署

```bash
# 在 VPS 上拉取最新镜像并重启（不停机滚动更新）
cd /opt/quant
docker compose pull backend
docker compose up -d --no-deps backend

# 检查部署是否成功
docker compose ps
curl -s http://localhost:8000/api/v1/health | jq .data.services
```

## 三、应急预案（Runbook）

### 🔴 P0-001：Futu OpenD 连接断开

**症状**: 
- `/api/v1/health` 返回 `futu_opend.status: "error"`
- 前端行情停止更新，WebSocket 推送 `{"type": "error", "code": 3001}`
- Grafana 告警：`futu_connected = 0`

**诊断步骤**:
```bash
# 1. 检查 OpenD 进程是否存活
systemctl status futu-opend

# 2. 查看 OpenD 日志
journalctl -u futu-opend -n 100

# 3. 检查 Python watchdog 重连日志
docker compose logs backend --since 10m | grep "futu"
```

**恢复操作**:
```bash
# 方案 A：自动恢复（Python watchdog 应在 30s 内自动重连，等待观察）
# 查看重连状态
docker compose logs backend -f | grep "reconnect"

# 方案 B：手动重启 OpenD
systemctl restart futu-opend
sleep 10
# 验证重连
curl -s http://localhost:8000/api/v1/health | jq .data.services.futu_opend
```

**预防措施**:
- systemd `Restart=always` 保证 OpenD 进程自动拉起
- Python asyncio watchdog 每 30s 心跳检测，断连后指数退避重连（5s → 10s → 20s → 60s）

---

### 🔴 P0-002：Redis OOM / 不可用

**症状**:
- 后端大量 `ConnectionError: Redis connection failed` 日志
- API 响应 500 错误，`code: 3002`
- 行情推送全停

**诊断步骤**:
```bash
# 1. 检查 Redis 容器状态
docker compose ps redis
docker stats quant-redis --no-stream

# 2. 查看内存使用
docker exec quant-redis redis-cli info memory | grep -E "used_memory_human|maxmemory_human|mem_fragmentation_ratio"

# 3. 查看大 Key（内存占用排行）
docker exec quant-redis redis-cli --bigkeys
```

**恢复操作**:
```bash
# 方案 A：内存超限，手动清理 K 线温数据缓存
docker exec quant-redis redis-cli --scan --pattern "kline:hot:*" | xargs docker exec quant-redis redis-cli DEL

# 方案 B：重启 Redis（数据从 AOF 恢复）
docker compose restart redis
# 等待 AOF 加载完毕（通常 < 30s）
docker exec quant-redis redis-cli ping

# 方案 C：Redis 崩溃，从最近 RDB 备份恢复
# 备份文件在 Cloudflare R2: r2://quant-backup/redis/dump-YYYYMMDD.rdb
# 下载并替换 /opt/quant/redis-data/dump.rdb，然后重启 Redis
```

**预防措施**:
- `maxmemory 2gb` + `maxmemory-policy allkeys-lru`（内存满时自动淘汰最旧 Key）
- 每日 03:00 自动 RDB 备份到 Cloudflare R2
- K线热缓存设置 TTL = 24h，防止无限增长

---

### 🔴 P0-003：VPS 不可访问（网络中断/宕机）

**症状**:
- SSH 无法连接
- `https://api.quant.yourdomain.com` 返回 520/521/522

**诊断步骤**:
```bash
# 1. 检查 Cloudflare Tunnel 状态（在 CF Dashboard 查看）
# https://dash.cloudflare.com → Zero Trust → Tunnels

# 2. 从另一台机器 Ping
ping hk.yourdomain.com

# 3. 检查 VPS 控制台（登录 VPS 提供商控制台查看实例状态）
```

**恢复操作**:
```bash
# 方案 A：网络抖动，等待 Cloudflare 自动重连（通常 2-5 分钟）

# 方案 B：VPS 实例停止，在控制台手动启动实例
# 实例启动后，Docker Compose 和 systemd 服务应自动启动（restart: always）

# 方案 C：VPS 彻底挂掉，切换到备用节点（手动 DNS 切换或 Cloudflare Load Balancing）
# 1. 临时将 Cloudflare Tunnel 指向国际 VPS
# 2. 国际 VPS 启动后端服务（降级模式，不带 Futu 实盘能力）
```

**预防措施**:
- Cloudflare Tunnel 替代固定公网 IP，VPS 重启后自动重连 CF
- 所有服务配置 `restart: always`，VPS 重启后自动恢复
- 关键配置文件备份在 Cloudflare R2 + GitHub 私有仓库

### 🟠 P1-001：数据库连接池耗尽

**症状**:
- 大量 `asyncpg.exceptions.TooManyConnectionsError`
- API 响应 503，但 Redis 和 Futu 状态正常

**诊断**:
```bash
# 查看当前连接数
docker exec quant-postgres psql -U quant_user -c "SELECT count(*) FROM pg_stat_activity;"

# 查看持连接最久的查询
docker exec quant-postgres psql -U quant_user -c "
SELECT pid, now() - query_start AS duration, query, state 
FROM pg_stat_activity 
WHERE state != 'idle' 
ORDER BY duration DESC LIMIT 10;"
```

**恢复操作**:
```bash
# 终止空闲连接
docker exec quant-postgres psql -U quant_user -c "
SELECT pg_terminate_backend(pid) FROM pg_stat_activity 
WHERE state = 'idle' AND query_start < now() - interval '5 minutes';"

# 如果持续泄漏，重启后端服务（连接池会重建）
docker compose restart backend
```

**症状**:

# 1. 查看 CI 失败原因（GitHub → Actions → 失败的 run）

```bash

cd backend && python -m pytest tests/ --cov=. --cov-fail-under=70
cd frontend && pnpm lint && pnpm test run

# 修复后手动触发 CI
gh workflow run deploy.yml --ref main
```

## 六、灾难恢复（Disaster Recovery）

### 完整系统重建流程

```bash
# 1. 初始化新 VPS（Ubuntu 22.04 LTS）
apt update && apt install -y docker.io docker-compose-v2 git

# 2. 配置 SSH 密钥
mkdir -p ~/.ssh && chmod 700 ~/.ssh
# 将公钥添加到 authorized_keys

# 3. 克隆私有配置仓库（存放加密后的 .env 文件）
git clone git@github.com:your/quant-config.git /opt/quant-config

# 4. 下载最新 Docker 镜像
docker pull ghcr.io/your/quant-backend:latest
docker pull ghcr.io/your/quant-frontend:latest

# 5. 恢复 PostgreSQL 数据
# 从 Cloudflare R2 下载最新备份
aws s3 cp s3://quant-backup/postgres/quant-$(date +%Y%m%d).sql.gz . \
  --endpoint-url https://xxx.r2.cloudflarestorage.com
docker compose up -d postgres
gunzip -c quant-$(date +%Y%m%d).sql.gz | docker exec -i quant-postgres psql -U quant_user quant_db

# 6. 恢复 Redis 数据
aws s3 cp s3://quant-backup/redis/dump-$(date +%Y%m%d).rdb /opt/quant/redis-data/dump.rdb \
  --endpoint-url https://xxx.r2.cloudflarestorage.com
docker compose up -d redis

# 7. 启动所有服务
cp /opt/quant-config/.env.prod /opt/quant/.env
docker compose -f docker-compose.prod.yml up -d

# 8. 安装 Futu OpenD
# 从 https://www.futunn.com/download/OpenD 下载对应版本
systemctl start futu-opend

# 9. 验证系统健康
curl http://localhost:8000/api/v1/health
```

**RTO（恢复时间目标）**：< 2 小时  
**RPO（恢复点目标）**：< 24 小时（依赖每日备份频率）

## 七、VPS 安全加固清单（SEC-16）

> 2026-07-12 第三轮 Review 补充。所有生产 VPS 必须执行以下安全加固。

### 7.1 SSH 加固（所有 VPS）

```bash
# /etc/ssh/sshd_config 关键配置
PasswordAuthentication no          # 禁用密码登录，仅允许密钥认证
PermitRootLogin no                 # 禁止 root 直接登录
Port 2222                          # 改用非标端口（默认 22 易被扫描）
MaxAuthTries 3                     # 最大认证尝试次数
ClientAliveInterval 300            # 5 分钟无活动自动断开
AllowUsers quant                   # 仅允许指定用户登录
```

### 7.2 防火墙规则（UFW）

```bash
# 加州主节点
ufw allow 2222/tcp                 # SSH (非标端口)
ufw allow 80/tcp                   # HTTP (Cloudflare Tunnel 验证)
ufw allow 443/tcp                  # HTTPS (Cloudflare Tunnel)
ufw deny 6379                      # Redis 禁止公网 (仅 Tailscale 内网)
ufw deny 5432                      # PostgreSQL 禁止公网
ufw enable

# 北京辅助节点
ufw allow 2222/tcp
ufw deny 6379
ufw deny 5432
ufw enable
```

### 7.3 端口暴露检查清单

| 端口 | 服务 | 生产环境 | 本地开发 |
|:---|:---|:---|:---|
| 6379 | Redis | ❌ 禁止公网 (Tailscale 内网) | ✅ 本地暴露 |
| 5432 | PostgreSQL | ❌ 禁止公网 | ✅ 本地暴露 |
| 8000 | API | ✅ 经 Cloudflare Tunnel | ✅ 本地暴露 |
| 9090 | Prometheus | ❌ 仅内网 | ✅ 本地暴露 |
| 3001 | Grafana | ❌ 经 Cloudflare Access | ✅ 本地暴露 |

## 八、Futu OpenD 部署前提与约束（ARCH-01）

> 2026-07-13 架构审计补漏。Futu OpenD 是港股/美股实盘交易的核心依赖，部署前必须满足以下硬性前提。

### 8.1 宿主机硬件要求

| 约束项 | 要求 | 原因 |
|:---|:---|:---|
| **CPU 架构** | **必须 x86_64 (amd64)** | Futu OpenD 官方仅提供 x86 二进制，**不支持 ARM (aarch64)**。在 ARM Linux 上无法启动，且无编译源码可供自行构建。 |
| **操作系统** | Linux (Ubuntu 20.04+/Debian 11+) 或 macOS | 生产环境推荐 Ubuntu 22.04 LTS；本地开发可用 macOS |
| **内存** | ≥ 2 GB 可用 | OpenD 内部维护行情快照 + 订阅表，内存不足会导致 OOM Kill |
| **磁盘** | ≥ 500 MB（含日志轮转空间） | OpenD 日志文件增长较快，需配置 logrotate |
| **网络** | 能访问 `*.futu.com` 域名（443/11111 端口） | OpenD 需连接 Futu 网关服务器 |

**验证命令**：
```bash
# 确认 CPU 架构（必须输出 x86_64）
uname -m

# 确认 FUTU OpenD 二进制兼容性
file FutuOpenD
# 期望输出: ELF 64-bit LSB executable, x86-64
```

> ⚠️ **常见踩坑**：AWS Graviton / 阿里云倚天 等 ARM 实例 **无法运行** OpenD。选购 VPS 时必须确认是 x86 实例（如 AWS t3/m5、阿里云 ecs.g7）。

### 8.2 跨地域部署限制

| 场景 | 地域要求 | 原因 |
|:---|:---|:---|
| **港股实盘交易** | **必须部署在香港节点** | OpenD 到 Futu 港股交易网关的 RTT 必须 < 10ms，否则下单延迟导致滑点甚至拒单 |
| **美股实盘交易** | 推荐美国节点（加州/纽约） | 美股网关延迟容忍度稍高，但跨太平洋（>200ms）仍有拒单风险 |
| **纯数据采集（非交易）** | 任意节点均可 | 历史 K 线拉取、行情订阅等非交易操作对延迟不敏感 |
| **A 股数据采集 (AKShare)** | 推荐北京/国内节点 | AKShare 数据源在国内，境外节点访问慢且不稳定 |

**当前部署方案**：
- **加州主节点 (38.60.126.42)**：API + Worker + Redis + PostgreSQL，运行非交易采集器（YFinance/AKShare 代理）
- **北京辅助节点**：仅运行 AKShare 采集器（南向/北向资金 + 宏观日历）
- **港股实盘节点（待部署）**：需独立香港 VPS，运行 OpenD + OMS 交易链路

> ⚠️ **关键决策**：当系统进入港股/美股实盘阶段时，**必须**将 OpenD + OMS 模块部署到对应市场的低延迟节点。当前加州主节点仅支持美股数据采集，不支持港股实盘交易。

### 8.3 OpenD 版本管理

```bash
# 查看当前 OpenD 版本
FutuOpenD --version

# 版本升级流程
# 1. 备份当前二进制
cp FutuOpenD FutuOpenD.bak
# 2. 下载新版本（从 Futu 官网或 GitHub Release）
# 3. 停止服务 → 替换二进制 → 启动 → 验证
systemctl stop futu-opend
mv FutuOpenD_new FutuOpenD
chmod +x FutuOpenD
systemctl start futu-opend
# 4. 验证连接
curl -s http://localhost:8000/api/v1/health | jq .data.services.futu_opend
```

## 九、DuckDB 数据湖分区策略（ARCH-02）

> 2026-07-13 架构审计补漏。定义 Parquet 文件的分区规则，防止单文件过大影响查询性能。

### 9.1 当前存储结构

```
data/kline_warehouse/
├── K_DAY/                    # 日 K 线
│   ├── US_AAPL.parquet       # 每标的一个文件（全量历史）
│   ├── HK_00700.parquet
│   └── ...
├── K_60M/                    # 60 分钟 K 线
│   └── ...
└── K_{ktype}/                # 其他周期
```

**问题**：单标的全量历史 Parquet 文件随时间增长（如 40 年日 K ≈ 10000 行），虽然 Parquet 列式格式本身高效，但在以下场景存在性能隐患：
- 回测引擎只需近 5 年数据时，仍需读取全量文件再 `tail(num)`
- 多标的大批量扫描时无法利用分区裁剪
- 快照版本化（DQ-03）时无法按日期切片

### 9.2 分区规则定义

**分区键**：`{market}_{ticker}` → `{ktype}` → `{year}` 三级分区

```
data/kline_warehouse/
├── K_DAY/
│   ├── US_AAPL/
│   │   ├── 2020.parquet      # 2020 年及以前的数据
│   │   ├── 2021.parquet
│   │   ├── ...
│   │   └── 2026.parquet      # 当年（持续追加）
│   ├── HK_00700/
│   │   └── ...
│   └── ...
├── K_60M/
│   ├── US_AAPL/
│   │   ├── 2025.parquet
│   │   └── 2026.parquet
│   └── ...
└── ...
```

**分区阈值**：

| K 线周期 | 单分区最大行数（建议） | 切分粒度 | 理由 |
|:---|:---|:---|:---|
| K_DAY | ≤ 252 行/年 | 按年 | 日 K 一年约 252 个交易日，数据量小 |
| K_60M | ≤ 252 × 4 行/年 | 按年 | 60 分钟 K 一天约 4 根（港股/美股） |
| K_MIN | ≤ 252 × 390 行/月 | 按月 | 分钟 K 数据量大，按月切分 |
| K_1M | ≤ 252 × 390 行/月 | 按月 | 同上 |

### 9.3 迁移策略（向后兼容）

```
阶段 1（当前）：保持 flat 结构，新写入仍为 {ticker}.parquet
阶段 2（回测引擎就绪后）：KlineWarehouse 新增 partitioned 模式开关
  - KLINE_WAREHOUSE_PARTITIONED=true 时启用分区写入
  - 读取时优先查分区目录，fallback 到 flat 文件
阶段 3（数据量达标后）：一次性迁移脚本将 flat 文件拆分为分区格式
  - scripts/migrate_kline_partition.py
  - 备份原文件 → 按年切分 → 校验行数一致 → 删除原文件
```

### 9.4 查询优化原则

1. **回测引擎默认只读当年 + 前 1 年分区**，避免全量扫描
2. **DuckDB 查询使用分区裁剪**：`WHERE year >= 2024` 自动跳过旧分区
3. **快照版本化（DQ-03）按分区粒度生成 manifest**，而非全文件
4. **冷数据归档**：超过 5 年的分区文件压缩移至对象存储（R2/S3）

## 十、Futu OpenD 断连恢复 SOP（ARCH-03）

> 2026-07-13 架构审计补漏。定义 OpenD 断连时的完整恢复流程，确保在途订单不丢失、状态可对齐。

### 10.1 断连影响矩阵

| 影响域 | 断连时行为 | 恢复后行为 |
|:---|:---|:---|
| **行情推送** | WebSocket 停止推送新 Tick；前端显示 STALE 标签 | 自动恢复订阅 → 拉取断连期间缺失 K 线 → 推送补全数据 |
| **在途订单** | 已发送但未成交的订单状态未知（可能部分成交/全部成交/被拒） | 必须执行状态对账（§10.3） |
| **新订单** | OMS 拒绝新订单（Kill Switch 自动触发） | 对账完成后解除 Kill Switch |
| **持仓同步** | 30s 定时守护进程拉取失败，Redis 缓存过期 | 对账完成后立即刷新持仓缓存 |
| **数据采集** | YFinance 降级为唯一数据源（非 Futu 依赖） | Futu 恢复后自动切回主数据源 |

### 10.2 自动恢复流程（已实现）

```
┌─────────────────────────────────────────────────────────┐
│  FutuWatchdog (backend/services/futu/watchdog.py)       │
│                                                         │
│  1. 健康检查（每 30s）                                    │
│     └─ get_global_snapshot() 失败 → 标记断连             │
│                                                         │
│  2. 断连检测 → 触发重连循环                               │
│     ├─ 指数退避：5s → 10s → 20s → 40s → 60s（上限）     │
│     ├─ 连续失败 ≥ 10 次 → 触发熔断，停止重试              │
│     └─ Prometheus 指标：futu_reconnect_total             │
│                                                         │
│  3. 重连成功                                             │
│     ├─ 恢复订阅（_restore_subscriptions）                 │
│     ├─ 重置 consecutive_failures                         │
│     └─ 广播连接恢复事件 → 前端 STALE 标签消失             │
└─────────────────────────────────────────────────────────┘
```

### 10.3 在途订单对账流程（SOP）

**断连 detected 时**：

```
Step 1: 暂停接单（自动）
  - AlertEngine 检测到 futu_connected=0 → 触发 Kill Switch
  - OMS 状态切换为 FROZEN，拒绝所有新订单
  - 前端显示橙色横幅："交易通道中断，暂停接单"

Step 2: 记录断线时刻的在途订单
  - 从 PostgreSQL orders 表查询 status IN ('submitted', 'partial_filled') 的订单
  - 写入 Redis Set: quant:oms:disconnected_orders:{timestamp}
  - 这些订单的状态在断连期间不可信

Step 3: 等待自动重连（Watchdog 处理）
  - 通常 30s~5min 内自动恢复
  - 若超过 10min 未恢复 → 飞书告警 → 人工介入

Step 4: 重连后状态对账
  - 遍历 Step 2 记录的在途订单
  - 对每个 order_id 调用 Futu API 查询最新状态：
    a. Futu 返回 FILLED → 更新本地 status='filled'，同步成交记录
    b. Futu 返回 PARTIAL_FILLED → 更新成交数量，剩余部分标记为 cancelled
    c. Futu 返回 CANCELLED/EXPIRED → 更新本地 status
    d. Futu 返回 NOT_FOUND → 标记为 unknown，人工确认
  - 对账完成后清空 Redis Set，解除 Kill Switch

Step 5: 持仓刷新
  - 立即触发一次 Futu 持仓全量拉取（不等 30s 定时器）
  - 与本地 paper_positions 对比，差异写入告警日志

Step 6: 恢复报告
  - 生成对账报告：在途 N 笔 / 已成交 M 笔 / 已撤单 K 笔 / 未知 J 笔
  - 推送至飞书 + 前端通知中心
```

### 10.4 人工介入检查清单

当自动恢复失败（连续失败 ≥ 10 次 / 超过 10min），运维人员执行：

```bash
# 1. 检查 OpenD 进程
systemctl status futu-opend
journalctl -u futu-opend -n 50 --no-pager

# 2. 检查网络连通性
nc -zv <futu_gateway_host> 11111

# 3. 手动重启 OpenD
systemctl restart futu-opend
sleep 10
curl -s http://localhost:8000/api/v1/health | jq .data.services.futu_opend

# 4. 如果重启失败，检查 Futu 账户状态
# - 登录 Futu 客户端确认账户未被冻结
# - 检查 Futu 服务状态页是否有公告

# 5. 手动触发在途订单对账
# POST /api/v1/oms/reconcile（待实现）

# 6. 如果确认 Futu 服务长时间不可用
# - 通知用户：交易通道中断，建议通过 Futu 官方 App 手动管理持仓
# - 系统降级为只读模式（行情由 YFinance 兜底）
```
